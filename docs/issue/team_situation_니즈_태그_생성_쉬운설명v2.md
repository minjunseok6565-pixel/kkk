# team_situation 니즈 태그 생성 로직 쉬운 설명 v2

이 문서는 `data/team_situation.py`의 **개편된 니즈 태그(need tag) 생성 방식**을,
기존 문서와 비슷한 톤으로 쉽게 설명합니다.

---

## 0) 한 줄 요약

이제 `team_situation`의 니즈 태그는
**"역할 점수(role fit) + 실제 전술 사용량(usage) + 현재 로스터 커버리지(coverage)"**를
한 번에 비교해서 만듭니다.

즉, 예전처럼 스타일/로스터갭/효율 보정을 따로 합치는 방식이 아니라,
**역할 단위로 부족분을 직접 계산하는 단일 파이프라인**으로 바뀌었습니다.

---

## 1) 니즈 데이터 형태: TeamNeed (형식은 동일)

니즈 1개는 아래 필드를 가집니다.

- `tag`: 니즈 종류 (예: `OFF_ENGINE_PRIMARY`, `DEF_BACKLINE_ANCHOR`)
- `weight`: 중요도 (0~1)
- `reason`: 사람이 읽는 설명 문장
- `evidence`: 계산 근거(점수/커버리지/사용량/포지션 부족 여부 등)

핵심은 그대로 **설명 가능한 구조**를 유지한다는 점입니다.

---

## 2) 큰 변화: 니즈 소스가 "역할 엔진"으로 단일화

개편 전에는 대략 아래를 따로 만들고 합쳤다면,

- role-fit 기반
- 스타일 기반
- 로스터 갭 기반
- 효율 백분위 보정

개편 후에는 실제 최종 니즈가 사실상 ` _compute_role_fit_and_needs()` 결과로 정해집니다.

- `_style_to_needs()` → legacy 처리(빈 리스트)
- `_roster_gap_needs()` → legacy 처리(빈 리스트)
- `_boost_needs_by_efficiency_percentiles()` → legacy 처리(현재는 그대로 통과)

즉 **역할 기반 계산이 메인이고, 나머지 경로는 제거/비활성화**된 상태입니다.

---

## 3) 입력: 어떤 데이터를 보고 역할 니즈를 계산하나?

### (1) 로테이션 10인
- 로스터 상위 10명(rotation)을 대상으로 계산합니다.

### (2) 포지션 분포
- G/W/B 인원 비율을 계산하고,
- 상대 비중이 낮고 절대 숫자도 적으면(shortage) 해당 포지션 부족으로 판단합니다.

### (3) 스타일 시그널
- `pnr_rate`, `drive_rate`, `dho_rate`, `post_rate`, `transition_rate`, `setplay_rate`, `iso_rate`
- 이 값을 이용해 "우리 전술이 실제로 어떤 역할을 얼마나 요구하는지"(usage)를 만듭니다.

### (4) 수비 스킴 비중
- 팀의 수비 스킴 비중(`defense_scheme_share`)이 있으면 반영,
- 없으면 스킴별 가중치 테이블을 사용해 기본 usage를 구성합니다.

---

## 4) 핵심 계산 1: 역할 점수(score)와 커버리지(coverage)

각 선수에 대해 역할 점수를 계산하고,
팀 단위로 역할별 종합 점수/커버리지를 만듭니다.

- 공격 역할: 선수별 상위 3개 역할 점수만 사용(순위별 가중치 적용)
- 수비 역할: 수비 역할 점수 테이블 기반으로 동일하게 집계
- 커버리지: 해당 역할을 "충분히 수행 가능한 선수"가 로테이션에 얼마나 있는지 비율화

직관적으로 보면,
- `score` = "우리 팀 평균 수행 능력"
- `coverage` = "그 역할을 맡길 수 있는 인원층"
입니다.

---

## 5) 핵심 계산 2: 전술 사용량 usage

### 공격 usage
스타일 지표를 역할에 분배합니다. 예를 들어:
- `pnr_rate`는 `Engine_Primary`, `Roll_Man` 등에 분배
- `drive_rate`는 `Rim_Pressure`, `Shot_Creator` 등에 분배
- `iso_rate`는 `Shot_Creator` 비중을 크게 반영

분배 후 전체 합을 정규화해서 역할별 공격 usage를 만듭니다.

### 수비 usage
수비 스킴별 역할 가중치를 합산해 역할별 수비 usage를 만들고,
마찬가지로 정규화합니다.

즉 usage는 "이론상 필요한 역할"이 아니라,
**우리 팀 전술이 실제로 많이 요구하는 역할**에 가깝습니다.

---

## 6) 최종 weight 공식: "약점 + 미스매치"를 같이 본다

역할별 니즈 강도(`weight`)는 아래 감각으로 만들어집니다.

1. **기본 부족분(base)**
   - 점수가 낮으면 가중치 증가 (`low_score`)
   - 커버리지가 낮으면 가중치 증가 (`low_cov`)
   - 둘 중 큰 값을 기본 부족분으로 사용

2. **usage-coverage 미스매치 보정(adj)**
   - `usage > coverage`이면 가중치 추가(실전 요구 대비 인력 부족)
   - `usage < coverage`이면 가중치 일부 감산(과잉 커버)

3. **클램프 + 컷오프**
   - 최종값을 0~1로 제한
   - 너무 작은 값(`w < 0.18`)은 노이즈로 보고 버림

한 줄로 말하면,
**"못하는 역할" + "많이 쓰는데 사람이 없는 역할"을 우선순위로 올리는 방식**입니다.

---

## 7) 태그 이름 규칙: OFF_/DEF_ + (필요시) 포지션 프리픽스

### 기본 태그
역할 이름을 정규 태그로 변환합니다.
- 공격: `OFF_*` (예: `OFF_SPOTUP_SPACER`)
- 수비: `DEF_*` (예: `DEF_PNR_POA_DEFENDER`)

### 포지션 shortage가 확인된 경우
해당 역할이 특정 포지션 버킷(G/W/B)과 연결돼 있고,
그 포지션이 부족(shortage)이라면 태그 앞에 접두를 붙입니다.

예)
- `OFF_ENGINE_PRIMARY` → `G_OFF_ENGINE_PRIMARY`
- `DEF_BACKLINE_ANCHOR` → `B_DEF_BACKLINE_ANCHOR`

즉 "무엇이 필요한지"뿐 아니라
**"어느 포지션에서 그 니즈가 더 급한지"**를 태그 자체에 담습니다.

---

## 8) 중복 처리와 개수 제한

여러 경로에서 같은 태그가 생길 수 있으므로,
최종 정리 단계에서:

- 같은 태그는 최대 weight 기준으로 병합
- evidence는 합쳐서 보존
- 정렬 후 상위 10개만 유지

이 규칙은 v1과 비슷하지만,
이제 입력 자체가 역할 엔진 중심이라 결과 태그의 해석이 더 일관적입니다.

---

## 9) v1 대비 핵심 차이 요약

1. **멀티 소스 합성 → 단일 역할 엔진 중심**
2. **스타일/로스터갭/효율 보정 태그는 legacy 처리(실질 비활성)**
3. **태그 어휘가 일반 니즈(`SPACING`)에서 역할 태그(`OFF_...`, `DEF_...`)로 세분화**
4. **usage-coverage 차이(전술 수요 vs 인력 공급)를 수식에 직접 반영**
5. **포지션 shortage 시 `G_/W_/B_` 접두로 긴급성 위치를 표시**

---

## 10) 실무적으로 읽는 법

개편 이후 니즈를 볼 때는 아래 순서가 가장 빠릅니다.

1. `tag`의 OFF/DEF와 역할명으로 "어떤 농구 기능"이 부족한지 확인
2. `weight`로 급한 순서 파악
3. `evidence`에서
   - `score`(능력)
   - `coverage`(인원층)
   - `usage`(전술 수요)
   - `usage_minus_coverage`(미스매치)
   - `pos_shortage`(포지션 부족)
   를 같이 확인

이렇게 보면 "왜 이 태그가 떴는지"를 거의 역추적 없이 바로 이해할 수 있습니다.
