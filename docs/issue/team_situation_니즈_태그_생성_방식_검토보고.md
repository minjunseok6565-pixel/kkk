# team_situation 니즈 태그 생성 방식 검토 보고

작성 목적:
- 개편된 니즈 태그 생성 방식이 이전 방식 대비 현실성과 설득력이 있는지 점검
- 프로젝트 관점에서 병목/리스크가 있는지 확인
- 개선 우선순위를 제안

---

## 1) 총평 (요약)

결론부터 말하면, 현재 방식은 **이전 방식보다 현실성/설득력이 전반적으로 개선**되었습니다.

특히 다음이 강점입니다.
1. 역할 단위(OFF/DEF)로 니즈를 세분화해 액션 가능성이 높아짐
2. 단순 약점 탐지가 아니라 `usage - coverage` 미스매치를 반영해 실제 전술 수요를 반영함
3. 포지션 shortage 시 `G_/W_/B_` 접두로 "어디서 급한지"까지 표현함

다만 운영 안정성 측면에서는 아래 보완이 필요합니다.
- role/scheme 데이터가 없을 때 니즈가 비어버리는 fail-soft 경로
- threshold 하드코딩(55/0.62/0.35/0.18)으로 인한 리그 환경 변화 민감도
- 수비 스킴 share 부재 시 균등 fallback이 실제 팀 색을 희석

---

## 2) 현실성/설득력 평가

### 2-1. 왜 이전보다 현실적인가

- 과거의 "스타일 태그/로스터갭 태그/효율 보정" 다중 합성보다,
  현재는 역할별로 `점수(score)`, `인력층(coverage)`, `전술수요(usage)`를 같은 축에서 비교합니다.
- 이는 실무에서 코치/프런트가 보는 질문("우리 팀이 많이 쓰는 전술을 소화할 인력이 있나?")과 더 가깝습니다.

### 2-2. 설득력 포인트

- 공격: 선수별 상위 3개 역할 점수 집계 + 스타일 기반 usage 분배
- 수비: role profile 점수 + 스킴 가중 usage 집계
- `usage > coverage`면 가중치 추가, 반대면 일부 감산

즉 "못한다"만 보는 것이 아니라, "**많이 필요한데 못한다**"를 우선순위화해서 결과가 더 납득 가능합니다.

---

## 3) 병목/리스크 점검

### 3-1. 기능 리스크: 니즈 생성이 1개 경로에 과집중

현재 최종 needs는 사실상 role 기반 결과만 사용합니다.
- legacy style/roster-gap/efficiency 함수는 no-op 상태
- role_fit import 실패 시 needs는 빈 리스트로 반환

영향:
- 외부 role 데이터/모듈 이슈가 생기면 니즈가 통째로 약해질 수 있음
- 예전의 안전망(효율 하위권 방향성 태그 주입 등)이 사라져 최소 신호 보장이 약해짐

### 3-2. 모델링 리스크: 절대 임계치 하드코딩

다음 값들이 코드 상수로 박혀 있습니다.
- coverage good 기준: `sc >= 55.0`
- low_score 기준 중심: `0.62`
- low_cov 기준 중심: `0.35`
- 노이즈 컷오프: `w < 0.18`

영향:
- 리그 평균 실력 분포가 달라지면 태그 민감도가 과/소추정될 수 있음
- 시즌/리그 설정 변화에 자동 적응이 어려움

### 3-3. 설명 가능성 리스크: 수비 usage fallback 단순화

`defense_scheme_share`가 없으면 스킴을 균등 처리합니다.
영향:
- 실제로 드랍 위주 팀인데 블리츠/스위치 역할 usage가 과도 반영될 수 있음
- 결과 태그가 팀 전술 체감과 다르게 나와 설득력이 떨어질 수 있음

### 3-4. 성능(퍼포먼스) 병목 가능성

로직 규모 자체는 팀당 로테이션 8인 기준이라 절대 연산량은 크지 않지만,
아래는 상대적 hot spot입니다.
- 선수별 역할 점수 계산 + 정렬(top3) 루프
- 수비 역할 dot product + 정렬(top3) 루프

현재 팀 수(30) 기준으로는 실무 병목 가능성은 낮지만,
시뮬레이션 반복 호출이 많을 경우 캐시 여지가 있습니다.

## 4) 개선 권고안 (우선순위)

### P0 (안정성)
1. **Fail-soft 백업 니즈 복구**
   - role import 실패 또는 role_def 비정상(empty) 시,
     최소한의 `OFFENSE_UPGRADE` / `DEFENSE_UPGRADE` 대체 태그를 주입
   - 완전 무신호 상태를 방지

2. **Legacy no-op를 완전 삭제하거나, 백업 경로로 제한 부활**
   - 현재는 존재하지만 실제 의미가 거의 없어 유지보수 혼란 가능
   - "죽은 코드"라면 제거, "안전망"이라면 명시적 fallback으로 재도입

### P1 (품질)
3. **임계치 외부 설정화(config화)**
   - `55.0`, `0.62`, `0.35`, `0.18`, `0.08`, `0.20/0.15` 등을 설정 파일로 이동
   - 리그별/시즌별 튜닝 가능하게 변경

4. **수비 usage fallback 개선**
   - 균등 분배 대신 최근 경기 로그에서 추정한 약식 스킴 prior를 사용
   - 데이터 없을 때도 팀 정체성 훼손 최소화

### P2 (관측성/운영)
5. **needs 진단 로그/메트릭 추가**
   - 팀별 상위 3 태그와 `score/usage/coverage/delta`를 구조화 로그로 남김
   - "왜 이상한 태그가 떴는지" 운영 중 추적 비용 감소

6. **회귀 테스트 세트 구축**
   - 대표 팀 케이스(슈팅 부족형, 볼핸들 부족형, 빅 부족형, 수비 붕괴형) 골든셋 고정
   - threshold 조정 시 태그 변동 폭을 자동 검증

---

## 5) 최종 판단

- 방향성: **긍정적** (이전 방식 대비 더 농구적으로 타당하고 설명 가능)
- 즉시 리스크: **중간** (단일 경로 의존 + fallback 약화)
- 추천 액션: P0 2개 먼저 처리 후, P1 설정화까지 진행하면 실전 신뢰도가 크게 올라갈 것으로 판단
