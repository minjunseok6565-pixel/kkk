# 트레이드 관련 코드 현실성/설득력 검토 보고서 (현재 브랜치)

작성일: 2026-03-14  
대상 범위: `trades/`, `state_modules/state_trade.py`의 실행 로직(테스트 파일 제외)

## 검토 방식
- 트레이드 런타임 모듈을 우선 스캔(`trades/*`, `state_modules/state_trade.py`)한 뒤,
- 아래 기준으로 “현실성 부족/유저 설득력 부족/과도 단순화” 케이스를 선별했다.
  - 리그/팀 맥락 없이 고정값으로 의사결정하는가
  - 경계값에서 의사결정이 급변(불연속)하는가
  - 설명 가능한 근거보다 랜덤/휴리스틱 비중이 과한가
  - 실제 트레이드 시장에서 중요한 변수를 누락하는가

재검토 메모:
- 1~22 항목을 코드 재대조한 결과, **위치 오기는 확인되지 않았고** 대부분 실재한다.
- 다만 17번은 기존 표현이 다소 과장되어, “invalid 딜이 상위에 뜬다”가 아니라 “평가 단계 자체는 validate를 생략하므로 upstream 전제 의존이 크다”로 보정했다.

## 핵심 결론
현재 구현은 **결정성(Determinism)·안정성·설명가능성(Explainable heuristic)**을 우선한 구조다. 다만 그 대가로,
- 가치 산정(선수/픽/계약),
- 생성단 후보 필터,
- 협상 수락/카운터/자동종료,
- 오케스트레이션 전역 리듬
영역에서 **고정 상수·단일값 축약·임계치 분기**가 누적되어 현실 NBA 시장 대비 정보 손실이 크다.
(재검토 시 일부 표현은 강도를 조정했지만, 문제의 구조적 존재 자체는 유지됨)

특히 사용자 체감 관점에서 아래 패턴이 반복될 위험이 있다.
1. “왜 이 픽/선수가 이 값인지” 근거가 지나치게 단선적임.
2. “0.49 vs 0.51” 같은 경계에서 결과가 튀는 불연속.
3. “불가능한 딜이 상위로 보여 보였다가 나중에 탈락”하는 경험.
4. “시장 전체가 한꺼번에 과열/침체”하는 인공적 동조화.

## 상세 이슈



### 4) 의사결정 스케일이 `outgoing_total` 중심 + 최소값 6.0 고정 ✅
- **위치**: `trades/valuation/decision_policy.py`
- **관찰**:
  - 임계값/오버페이 허용량이 `scale = max(outgoing, 6.0)`에 비례.
- **왜 문제인가**:
  - `outgoing≈0` 구조에서 인위적 하한이 작동해 팀 성향과 무관한 경직 반응 유발.
  - 자산 타입/거래 구조에 따른 프리미엄 차이를 반영하기 어려움.
- **개선 방향**:
  - `max(outgoing, incoming, type_adjusted_floor)` 등 다축 스케일 도입.
  - 최소 스케일을 시즌/샐러리캡 연동 동적화.
- **예상 작업량**: 약 **2~4 MD** (스케일 함수 재설계 1, 파라미터/설명 메타 정비 0.5, 의사결정 회귀 검증 1~2.5).

### 5) COUNTER 결정이 `counter_rate > 0.5` 불연속 규칙 ✅
- **위치**: `trades/valuation/decision_policy.py`
- **관찰**:
  - deterministic 모드에서 `counter_rate`를 사실상 이진 임계치로 사용.
- **왜 문제인가**:
  - 0.49와 0.51이 전혀 다른 결과를 만들어 유저 체감이 “튀는” 문제 발생.
- **개선 방향**:
  1. deterministic이어도 회색지대 거리/압박/팀성향을 결합한 연속 함수 사용.
  2. sigmoid 기반 soft probability로 임계점 급변 완화.
- **예상 작업량**: 약 **1.5~3 MD** (연속 함수 도입 0.5, deterministic/stochastic 일관화 0.5, 경계값 회귀 테스트 0.5~2).

## 빠른 실행안 (2주 내)
1. 스왑 근사가치 도입 + pre-score 메타에 불확실성 기록.
2. 픽 기대값을 팀 기반 prior 분포로 전환(최소 버전).
3. counter 결정 함수를 연속형(sigmoid 기반)으로 교체.
4. 생성단에서 하드룰 위반 딜 사전 제외(부분 validate).
5. 시장 리듬 입력에 평균 외 분위수/분산 신호 1개 추가.

- **개선 방향**:
  - 리그 수급 지표 기반 동적 배율(예: 0.96~1.08 cap)로 전환.
- **예상 작업량**: 약 **2~4 MD** (수급 지표 산출 1, 동적 배율 적용 0.5, 상한/하한 튜닝 및 회귀 0.5~2.5).

### 1) 픽 기대값이 순위 1:1 매핑(로터리 분산 무시)
- **위치**: `trades/valuation/data_context.py`
- **관찰**:
  - `build_pick_expectations_from_standings()`가 `standings_order` 인덱스로 기대 픽 번호를 직접 부여.
  - 주석에도 `Ignores lottery variance`를 명시.
- **왜 문제인가**:
  - 로터리/타이브레이커/보호조건 상호작용이 실제 픽 가치의 핵심인데, 순위=픽 순번으로 단선화됨.
  - 시즌 중반 변동성이 큰데 `confidence` 고정값은 과신처럼 보일 수 있음.
- **개선 방향**:
  1. `PickExpectation`을 단일 숫자에서 분포(예: P(top4), range)로 확장.
  2. 최소한 1~14 구간은 시즌 시점 기반 분산 파라미터 적용.
  3. 보호조건(Top-N)과 기대분포 결합 EV를 일관 산출.
- **예상 작업량**: 약 **4~6 MD** (데이터모델 확장 1.5, 분포/EV 계산 2, 기존 가격/보호조건 연동 1, 회귀 테스트 0.5~1.5).

### 2) 기대 픽 번호 미상 시 `16.0` 고정 대체
- **위치**: `trades/valuation/market_pricing.py`
- **관찰**:
  - `_price_pick()`에서 기대 순번을 모르면 `exp_num = 16.0`으로 대체.
- **왜 문제인가**:
  - 모든 미상 케이스가 1라운드 중앙값으로 수렴해, 약팀/강팀 원소속 픽의 리스크 차이를 삭제함.
  - 유저 관점에서 “정보 없으면 다 비슷한 픽”으로 체감됨.
- **개선 방향**:
  1. 원소속 팀 전력 티어/최근 승률 기반 prior 분포 도입.
  2. 데이터 부족 시에도 팀 기반 prior(리빌딩팀은 더 이른 픽 분포) 적용.
- **예상 작업량**: 약 **2~3 MD** (팀 prior 정의 0.5, `_price_pick` fallback 치환 0.5, 튜닝/검증 1~2).

### 3) standings 기반 worst→best 정렬이 픽 기대 산정의 사실상 전부
- **위치**: `trades/valuation/service.py`
- **관찰**:
  - `win_pct`, `point_diff_pg`만으로 정렬해 픽 기대 입력 생성.
- **왜 문제인가**:
  - 일정 강도(SOS), 부상 복귀, 최근 로스터 변화, posture(탱킹/매수) 같은 핵심 맥락이 누락됨.
- **개선 방향**:
  - 최근 N경기 추세/부상 가중치/팀 방향성 보정치를 기대순번에 반영.
- **예상 작업량**: 약 **3~5 MD** (standings 입력 피처 확장 1.5, 가중 결합 로직 1, 컨텍스트 의존성 정리 0.5, 밸런스 튜닝/테스트 1~2).

### 11) 재정 패널티 연봉 프록시가 “최댓값” fallback
- **위치**: `trades/valuation/team_utility.py`

- **왜 문제인가**:
  - 프런트로드/백로드 계약에서 현재 부담 왜곡 위험.
- **개선 방향**:
  - 현재 시즌 연봉 우선, 없으면 시간가중 평균을 fallback으로 사용.
- **예상 작업량**: 약 **1~2 MD** (현재연도 우선 산식 반영 0.5, fallback 변경 0.25, 계약 케이스 회귀테스트 0.25~1.25).

### 12) Fit 엔진에서 unsupported need tag가 계산에서 제외
- **위치**: `trades/valuation/fit_engine.py`
- **관찰**:

- **개선 방향**:
  - 미지원 태그를 0점 처리 대신 경고/불확실성 패널티로 가시화.
- **예상 작업량**: 약 **1.5~3 MD** (unsupported 태그 패널티 정책 추가 0.5, breakdown/경고 메타 반영 0.5, 영향도 튜닝 0.5~2).

### 17) 생성단 점수평가에서 `validate=False` 경로 사용 (표현 보정)
- **위치**: `trades/generation/dealgen/scoring.py`
- **관찰**:
  - `evaluate_deal_for_team(..., validate=False)`로 양팀 평가 수행.
  - 현재 파이프라인(`core.py`)은 평가 전에 `repair_until_valid`를 통해 유효성 검사를 선행하므로, 즉시 invalid 딜이 대량 상위 노출되는 구조는 아님.
- **왜 문제인가**:
  - 평가 함수 자체는 hard-rule 재검증을 하지 않으므로, 상위 레이어의 validate 전제가 약해지거나 향후 경로가 추가되면 이상치가 점수화될 여지는 남는다.
  - 즉, 현재는 “즉시 UX 결함”보다는 “아키텍처 결합도/회귀 리스크” 성격이 더 강함.
- **개선 방향**:
  - 최소 하드룰 위반(샐러리/스테피엔/락) 스모크 검증을 평가 직전/직후에 얕게 두어 방어층을 추가.
- **예상 작업량**: 약 **1.5~3 MD** (얕은 하드룰 스모크 계층 추가 0.5~1, scoring 경로 결합 0.5, 회귀 테스트 0.5~1.5).

### 18) 딜 스코어 discard 임계가 절대값 중심

- **개선 방향**:
  - cap-share 또는 리그 분포 percentile 기반 상대 임계로 전환.
- **예상 작업량**: 약 **2~4 MD** (상대 임계 정의 0.5, config/스코어링 전파 0.5, 기존 세이브 호환 및 튜닝 1~3).

### 22) Deal generator 예산 스케일 posture 고정치 + 하한 1 강제

- **예상 작업량**: 약 **1.5~3 MD** (posture 연속함수화 0.5, budget 하한 정책 분기 0.5, 성능/행동 회귀 테스트 0.5~2).

## 정리
현재 트레이드 엔진은 구조 분리와 디버깅 친화성은 우수하지만, 현실성 측면에서는 **고정 휴리스틱 의존**이 커서 장기적으로 유저 설득력 저하 위험이 있다.

우선 P0 항목부터 적용하면, “왜 이 딜이 나왔는가 / 왜 이 반응이 나왔는가”에 대한 체감 일관성을 가장 빠르게 개선할 수 있다.
