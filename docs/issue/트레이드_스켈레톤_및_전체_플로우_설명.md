# 트레이드 시스템에서의 "스켈레톤" 의미와 전체 플로우

## 1) 스켈레톤의 정확한 의미
- 이 프로젝트에서 스켈레톤(skeleton)은 **거래 구조 템플릿(딜의 뼈대)** 을 뜻한다.
- 구현상 `SkeletonSpec`(식별자, 적용 모드, 타깃 티어, 우선순위, builder 함수, gate)을 통해 정의되고,
  BUY/SELL + target tier 라우팅 결과로 실제 빌더가 호출된다.
- 즉 스켈레톤은 "누구와 무엇을 대충 묶을까"를 정하는 것이 아니라,
  **딜 모양(예: picks_only, young_plus_pick, p4p_salary, 2-for-1 등)을 먼저 고정하는 단계**다.

## 2) 탐색→평가→실행 전체 단계 (누락 없이)

### A. 오케스트레이션 틱 진입
1. `run_trade_orchestration_tick()`이 날짜/리그지문/tick_nonce로 RNG seed를 만든다.
2. `build_trade_generation_tick_context()`로 틱 스코프 캐시(룰 컨텍스트, 팀 상황, 의사결정 컨텍스트, valuation provider, asset catalog)를 구성한다.
3. 트레이드 데드라인 하드스톱, 압박도 맵, AI-AI 체결 캡을 계산한다.
4. actor selection 이후 각 actor 팀에 대해 `DealGenerator.generate_for_team()`을 호출한다.

### B. 팀별 딜 생성 (DealGenerator)
1. 팀 posture(BUY/SELL) 판단 후 예산(budget)과 RNG를 만든다.
2. BUY면 `_generate_buy_mode()`, SELL이면 `_generate_sell_mode()`로 진입한다.
3. 중복/비용 제어용 상태(`seen_skeleton`, `seen_output`, eval cache, sweetener/fit-swap 시도 카운터)를 초기화한다.

### C. 거래 대상 탐색(타깃/매물/상대팀)
#### BUY
1. `select_targets_buy()`가 need_map을 준비한다.
2. PUBLIC listing 메타, 팀별 cooldown, banned player를 반영한다.
3. 후보 ref마다 need_similarity, listing_boost, pre_score를 계산한다.
4. listed/non-listed quota와 cap, tier2 확장/반복 캡을 적용해 최종 `TargetCandidate` 리스트를 만든다.

#### SELL
1. `select_targets_sell()`이 내 팀 outgoing catalog에서 매물 후보를 만든다.
2. lock/recent signing ban을 선필터링한다.
3. bucket 우선순위 + trade_request/public signal + score/timing/contract_pressure로 정렬한다.
4. head 구간만 셔플해 다양성을 주고 상위 `SellAssetCandidate`를 반환한다.
5. `select_buyers_for_sale_asset()`로 매물별 잠재 buyer 팀을 찾는다.

### D. 스켈레톤 단계
1. `build_offer_skeletons_buy/sell()`이 실행된다.
2. `classify_target_tier()`로 대상 티어를 분류한다.
3. `SkeletonRegistry.get_specs_for_mode_and_tier()`가 모드/티어/라우팅 설정으로 skeleton spec을 필터링한다.
4. 각 spec의 `build_fn`이 호출되어 base deal 후보를 만든다.
5. 공통 메타(`skeleton_id`, `domain`, `target_tier`, `arch_compat`)와 태그를 후보에 부착한다.
6. `apply_modifiers()`로 보호픽/스왑 등 변형을 추가한다.
7. shape 검증 후 beam width 범위로 1차 후보를 자른다.

### E. 변형 확장/유효화/평가
1. `expand_variants()`로 후보를 얕게 확장한다.
2. `_beam_select_candidates()`로 variant cap 내에서 선별한다.
3. 후보별 `repair_until_valid()` 수행:
   - validate 불합격을 고치기 위한 repair 루프 수행
   - 실패 원인에 따라 asset/player ban 정보 누적
4. 유효 후보는 `evaluate_and_score()`로 양팀 valuation+decision을 계산한다.
5. 필요 시 pick protection decorator, fit-swap counter, sweetener를 적용한다.
6. `_should_discard_prop()`로 비현실/극단 손해 딜을 제거한다.
7. 최종 dedupe 및 partner/target diversity 페널티를 반영해 proposal pool을 완성한다.

### F. valuation 내부 (팀 관점 평가)
1. `evaluate_deal_for_team()`이 호출되면(서비스 레이어):
   - 필요 시 `validate_deal()`로 하드 룰 검증
   - team situation/GM traits/decision context 생성
   - repo snapshot 기반 valuation provider 구성
2. 순수 valuation 파이프라인(시장가치→팀효용→패키지효과)으로 `TeamDealEvaluation` 산출.
3. `decision_policy`가 ACCEPT/COUNTER/REJECT를 결정한다.

### G. 프로모션(협상 세션 생성/자동 체결)
1. `promote_and_commit()`가 proposal을 score 순으로 정렬해 처리한다.
2. 인간 팀이 포함되면:
   - AI 측 verdict가 ACCEPT인 딜 위주로 협상 세션 생성/재사용
   - offer tone/probe/lowball 정책, leak 정책, relationship/cooldown 이벤트 적용
   - 필요 시 `create_committed_deal()`로 커밋 가능한 약정 생성
3. AI-AI인 경우:
   - auto_commit 여부, office_gate, 팀당 당일 다중트레이드 제한 점검
   - 통과하면 `league_service.execute_trade()`로 즉시 실행
   - 이후 market/memory에 executed effects 반영

### H. 실제 거래 실행(API 경로)
#### 메뉴 즉시 실행
1. `/api/trade/submit`:
   - deal canonicalize/validate
   - `apply_deal_to_db()` 호출
   - 무결성 검사 및 UI 캐시 갱신

#### 협상 기반 실행
1. `/api/trade/negotiation/commit`:
   - 세션/멱등성/팀 일치/validate
   - 상대팀 관점 valuation으로 ACCEPT/COUNTER/REJECT 판단
   - ACCEPT면 `create_committed_deal()` 생성
2. `/api/trade/submit-committed`:
   - `verify_committed_deal()`로 해시/락/만료 재검증
   - validate(allow_locked_by_deal_id)
   - `apply_deal_to_db()` 실행
   - `agreements.mark_executed()` 처리

### I. DB 반영과 후처리
1. `apply_deal_to_db()`는 player move를 수집/검증하고,
   `trade_exec_serial_lock`으로 틱과 직렬화한 뒤 `LeagueService.execute_trade()`를 수행한다.
2. 실행 성공 후 `apply_trade_executed_effects_to_state()`로 trade_market/trade_memory 파생 상태를 갱신한다.
3. 일부 후처리 실패는 거래 자체를 롤백하지 않고, transaction에 sync 실패 정보를 기록한다.

## 3) 트레이드 과정에서 스켈레톤의 역할 요약
- 스켈레톤은 **탐색 공간의 초기 구조를 결정하는 핵심 제약장치**다.
- 대상 탐색이 "누굴 데려올지"를 고르는 단계라면,
  스켈레톤은 "어떤 형태로 주고받을지"를 규정한다.
- 이후 repair/valuation/sweetener가 품질을 다듬지만,
  출발점(딜 모양)의 대부분은 스켈레톤에서 이미 정해진다.
- 따라서 스켈레톤은
  1) 탐색 폭발 방지,
  2) 리그 룰/현실감에 맞는 형태 유도,
  3) 티어/모드별 전략적 다양성 제공
  을 담당한다.
