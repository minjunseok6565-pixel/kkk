# 부상 기반 시장가/리스크 개선 브리핑 (이슈 7, 10 대응)

작성일: 2026-03-15
대상: `trades/valuation` (market pricing + team utility)

## 1) 결론 요약
- **당신의 방향은 매우 타당**합니다. 특히 “현재 부상”과 “부상 이력” 분리, 그리고 하드 임계값 대신 소프트 게이트를 쓰겠다는 점이 현재 코드 철학(결정적 + 설명가능 + 튜닝 가능)과 잘 맞습니다.
- 다만 현재 트레이드 valuation 파이프라인에는 **부상 정보 주입 경로가 없음**이 핵심 병목입니다. 먼저 데이터 컨텍스트에 injury SSOT를 연결해야, 시장가(이슈 7)와 팀 리스크 할인(이슈 10)을 안정적으로 확장할 수 있습니다.
- 구현 우선순위는
  1) 데이터 주입(현재 상태 + 이력),
  2) market stage에 “부상 가감” 추가,
  3) team stage risk 합성 확장,
  4) 설명 메타/튜닝 도구 확장
  순으로 가는 것이 가장 안전합니다.

---

## 2) 현재 코드 상태 진단

### 2-1) 선수 시장가(이슈 7)
현재 `_price_player()`는 사실상 아래 축이 중심입니다.
- OVR 현재가치(sigmoid)
- 상위 OVR 스타 보너스(softplus)
- 나이 기반 now/future 분해
- 포지션 희소성 배율
- 계약가치(공정연봉 대비)

즉, 부상 관련 신호는 시장가에 직접 반영되지 않습니다.

### 2-2) 리스크 할인(이슈 10)
현재 `_apply_risk_discount()`는
- `age_risk` + `term_risk` 두 축만 합성
- `risk_score = 0.65*age + 0.35*term`
형태여서, 부상 취약성/재발성/현재 OUT 상태/부위 중증도 같은 신호는 빠져 있습니다.

### 2-3) 데이터 모델 유연성
좋은 점은 `PlayerSnapshot`에 아래 확장 포인트가 이미 존재한다는 것입니다.
- `attrs: Dict[str, Any]`
- `meta: Dict[str, Any]`

즉 타입을 크게 깨지 않고 `meta["injury"]` 같은 구조를 주입할 수 있습니다.

### 2-4) 현재 병목: 데이터 컨텍스트에서 injury 미주입
`RepoValuationDataContext.get_player_snapshot()`는 선수/계약/agency_state는 붙이지만, injury SSOT(`player_injury_state`, `injury_events`)는 읽지 않습니다.

따라서 “계산식 개선” 이전에 “부상 정보 주입”이 선행되어야 합니다.

## 3) 부상 정보를 정확하게 받아오는 방법 (권장)

## SSOT 원천
- 현재 상태: `player_injury_state`
  - `status`, `out_until_date`, `body_part`, `severity` 등
- 이력(append-only): `injury_events`
  - `date`, `body_part`, `severity`, `duration_days`, `out_until_date`, `returning_until_date`

## valuation 주입용 최소 스키마(권장)
`PlayerSnapshot.meta["injury"]`에 아래를 채우는 방식을 권장합니다.

- `current`
  - `status`: HEALTHY/OUT/RETURNING
  - `days_to_return`: int (평가일 기준)
  - `body_part`: str
  - `severity`: int
- `history`
  - `recent_count_30d`, `recent_count_180d`, `recent_count_365d`
  - `critical_count_365d`
  - `same_part_repeat_365d_max` (동일 부위 최다 반복)
  - `avg_severity_365d`
- `health_credit_inputs`
  - `healthy_days_365d`
  - `availability_rate_365d`

핵심은 **원시 event를 매번 valuation에서 재집계하지 말고**, data_context 빌드 시 미리 집계해 `meta`로 넣는 것입니다(성능/설명성/캐시 안정성).

## 4) 당신 계획의 로직화 제안 (소프트 게이트 중심)

### 4-1) 시장가: 현재 부상 할인
요구사항 반영:
- 1개월 이내: 미미한 감소
- 1~6개월: 선형에 가까운 완만 감소
- 6개월 초과: 큰 감소
- 하드 경계 금지

권장 방식: **시그모이드 2개 + 램프 1개 혼합**
- `g_30 = sigmoid((d-30)/s30)`
- `g_180 = sigmoid((d-180)/s180)`
- `mid = clamp((d-30)/(180-30), 0, 1)`
- `discount_current = w1*g_30*mid + w2*g_180`

이렇게 하면 30일, 180일 근방에서 자연스럽게 곡률이 생기고 불연속이 사라집니다.

### 4-2) 시장가: 부상 이력 할인
요구사항 반영:
- 최근 부상 잦을수록 감소
- 치명적 부위(무릎/허리 등) 잦을수록 감소
- 동일 부위 반복 잦을수록 감소

권장 합성:
- `f_recent = soft_count(recent_count_180d)`
- `f_critical = soft_count(critical_count_365d)`
- `f_repeat = soft_count(same_part_repeat_365d_max - 1)`
- `history_risk = 1 - exp(-(a*f_recent + b*f_critical + c*f_repeat))`

`soft_count(x)=1-exp(-k*x)` 류를 쓰면 초반 민감, 후반 포화 형태가 나와 하드코딩 느낌이 줄어듭니다.

### 4-3) 리스크 할인(팀 효용): injury 축 확장
현재 age/term 합성에 아래를 추가:
- `inj_hist_risk` (치명 부위 빈도 + 최근 빈도 + 반복성)
- `inj_current_risk` (현재 OUT/RETURNING + 부위 치명도 + severity)

권장 예:
- `risk_score = blend(age_term_risk, inj_hist_risk, inj_current_risk)`
- blend는 가중합 + squash(sigmoid/tanh)로 0..1 고정

중요: 리스크 할인은 team stage 성향(`risk_discount_scale`)을 유지하되, injury 기반 입력축만 늘려야 팀별 성향과 자연스럽게 결합됩니다.

### 4-4) 건강 반대급부(업사이드)
“감점만 있는 시스템”을 피하려면:
- 시장가에서 `health_credit`를 작은 보너스로 추가
  - 예: 최근 365일 availability가 높고 치명 부상 이력이 없으면 +0~X%
- 팀 리스크 할인에서는
  - 매우 건강한 선수의 `risk_score`를 낮춰 결과적으로 factor가 높아지게 설계

보너스는 할인보다 약하게(예: 할인 max 18%, 보너스 max 6%) 두어 악용을 방지하는 것이 좋습니다.

## 5) 현실성 + 게임성 관점 추가 제안

## A. 치명 부위는 코드 상수로 박지 말고 "설정"으로
- `critical_body_parts`를 config로 분리
- 세이브/리그 성향별 튜닝 가능하게 유지

## B. severity와 기간의 역할 분리
- 시장가: 기간(days_to_return) 영향이 더 큼
- 리스크: severity와 치명 부위의 상호작용 영향이 더 큼

## C. 최근성 가중(half-life)
- 1년 이력이라도 최근 사건이 더 무겁게 반영되도록
- `exp(-lambda * age_days)` 가중 합 추천

## D. 설명가능성 메타 강화
valuation step meta에 아래를 남기면 UI/로그 설득력이 급상승합니다.
- `injury_current_discount`, `injury_history_discount`
- `critical_part_hits`, `same_part_repeat_max`
- `health_credit`

## E. 밸런스 안전장치
- 시장가 하한/상한 clamp 유지
- 단일 이벤트로 급락하지 않도록 포화함수 사용
- 특정 포지션/고OVR에 과벌점이 걸리지 않게 교차 테스트

---

## 6) 구현 가능성 판정

- **구현 가능성: 높음**
  - 이유: valuation 엔진은 이미 순수 함수 + config + step explain 구조로 되어 있어 확장 친화적입니다.
- **선행 작업 필요: injury 데이터 주입 경로 구축**
  - data_context에서 injury SSOT를 읽어 `PlayerSnapshot.meta`로 싣는 작업이 필수입니다.
- **리스크 포인트**
  - DB 조회 증가(특히 `injury_events`) → 집계 캐시 필요
  - 지나친 벌점으로 거래시장 경직 가능 → 보너스/포화/상한으로 완충

## 7) 추천 실행 순서 (현실적)
1. `data_context`에 injury aggregate 로더 추가
2. `market_pricing`에
   - CURRENT_INJURY_DISCOUNT
   - INJURY_HISTORY_DISCOUNT
   - HEALTH_CREDIT
   step 추가
3. `team_utility`의 `RISK_DISCOUNT`를 injury-aware로 확장
4. trade_rules/valuation config 노출(critical parts, gate smoothness, max caps)
5. 회귀 테스트
   - 경계(29~31일, 175~185일)
   - 건강선수 보너스
   - 치명부위 반복 부상
