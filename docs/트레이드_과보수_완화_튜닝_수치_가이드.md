# 트레이드 과보수 완화 튜닝 수치 가이드

목표: **트레이드 시스템 철학(현실성/안전장치)은 유지**하면서, 수치 조정만으로 “제안이 너무 안 나오는” 현상을 줄이기 위한 실무 가이드.

> 원칙
> - 규칙/구조를 갈아엎지 않고, 임계값·가중치·캡만 미세조정한다.
> - 한 번에 1~2개 축만 조정하고, 2~4주 시뮬레이션 로그로 효과를 확인한다.
> - “딜이 많이 나오는 것”보다 “저품질 딜 급증 없는 범위”를 우선한다.

---

## 0) 빠른 요약 (어디부터 만질지)

가장 영향도가 큰 순서(권장):

1. **노출량(프로액티브 listing) 축**
2. **의사결정 문턱(min_surplus / overpay / counter corridor) 축**
3. **DealGen 조기폐기(discard) 축**
4. **최종 게이트(office gate) 축**

---

## 1) 트레이드 블록 노출량(입구) 튜닝

### 1-1. 프로액티브 listing threshold 계열
- 관련 필드
  - `ai_proactive_listing_threshold_default` (기본 0.55)
  - `ai_proactive_listing_bucket_thresholds` (posture별 버킷 임계값)
  - `ai_proactive_listing_threshold_horizon_win_now_delta` (기본 -0.03)
  - `ai_proactive_listing_threshold_horizon_rebuild_delta` (기본 -0.05)
  - `ai_proactive_listing_threshold_urgency_cut` (기본 0.75)
  - `ai_proactive_listing_threshold_urgency_delta` (기본 -0.03)
  - `ai_proactive_listing_threshold_cooldown_active_delta` (기본 +0.05)

- 수치 ↑ / ↓ 의미
  - threshold 관련 값 **↑**: listing 통과가 어려워짐(노출 감소, 보수성↑)
  - threshold 관련 값 **↓**: listing 통과가 쉬워짐(노출 증가, 보수성↓)
  - `*_delta` 값은 부호가 중요:
    - 음수 delta 절대값을 키우면(예: -0.03→-0.06) 해당 상황에서 **완화**
    - 양수 delta를 키우면(예: +0.05→+0.08) 해당 상황에서 **강화**

- 과보수 완화 방향
  - `threshold_default`를 소폭 하향(예: 0.55→0.50)
  - `cooldown_active_delta`를 완화(예: +0.05→+0.02)

### 1-2. 팀/일일 listing cap 계열
- 관련 필드
  - `ai_proactive_listing_team_daily_cap` (기본 2)
  - `ai_proactive_listing_team_active_cap` (기본 4)
  - `ai_proactive_listing_player_cooldown_days` (기본 7)

- 수치 ↑ / ↓ 의미
  - daily/active cap **↑**: 팀당 노출량 증가(제안 풀 증가)
  - cooldown_days **↑**: 같은 선수 재노출 지연(보수성↑)
  - cooldown_days **↓**: 재노출 빨라짐(보수성↓)

- 과보수 완화 방향
  - daily cap +1, cooldown -2일 같은 보수적 완화부터 시작

## 2) 의사결정 문턱(수락/카운터) 튜닝

### 2-1. `min_surplus_required` (DecisionContext 산출)
- 현재 로직 요약
  - `lerp(-0.03, 0.10, eff_neg_tough)` 기반
  - `SELL/SOFT_SELL`이면 `+0.03`
  - `AGGRESSIVE_BUY`이면 `-0.02`

- 수치 ↑ / ↓ 의미
  - 요구 surplus가 **↑**: ACCEPT 어려워짐(보수성↑)
  - 요구 surplus가 **↓**: ACCEPT 쉬워짐(보수성↓)

- 과보수 완화 방향
  - 상단값(0.10) 또는 SELL 보정(+0.03)을 소폭 완화
  - 철학 유지 포인트: 음수로 크게 열지 말고, “양수 요구치만 덜 엄격”하게

### 2-2. `overpay_budget` (DecisionContext 산출)
- 현재 로직 요약
  - `0.18 * eff_win_now * urgency * buy_factor * (1 - 0.45*eff_neg_tough)`
  - clamp 상한 0.22

- 수치 ↑ / ↓ 의미
  - 계수/상한 **↑**: 손해 감수 범위 확대(완화)
  - 계수/상한 **↓**: 손해 감수 범위 축소(강화)

- 과보수 완화 방향
  - 상한 0.22를 소폭 상향하거나(예: 0.24), 주계수 0.18을 미세 상향
  - 단, 급격히 올리면 저품질 ACCEPT 증가 가능

### 2-3. `DecisionPolicyConfig.counter_corridor_ratio`
- 기본 0.06

- 수치 ↑ / ↓ 의미
  - **↑**: ACCEPT 근처 회색지대 확대 → REJECT 대신 COUNTER가 늘어 협상 지속성↑
  - **↓**: 회색지대 축소 → REJECT 직행 증가

- 과보수 완화 방향
  - 0.06→0.08 정도의 완만한 상향이 안전

### 2-4. `counter_overpay_fraction`
- 기본 0.65

- 수치 ↑ / ↓ 의미
  - **↑**: 음수 surplus 구간에서도 COUNTER 허용 폭 확대
  - **↓**: 빠른 REJECT 증가

## 3) Team Utility/fit 패널티 튜닝

### 3-1. fit below-threshold 패널티
- 관련 필드
  - `fit_below_threshold_strength` (기본 2.0)
  - `fit_below_threshold_floor` (기본 0.35)

- 수치 ↑ / ↓ 의미
  - strength **↑**: threshold 미달 시 급격히 감점(강화)
  - strength **↓**: 미달 패널티 완화
  - floor **↑**: 최저 multiplier를 높여 극단적 감점 완화
  - floor **↓**: 더 강한 감점 허용

- 과보수 완화 방향
  - strength 소폭 하향(2.0→1.6), floor 소폭 상향(0.35→0.45)

### 3-2. 리스크/재정 패널티 강도
- 관련 필드
  - `risk_discount_scale` (DecisionContext 산출)
  - `finance_penalty_scale` (DecisionContext 산출)
  - `finance_factor_floor` (기본 0.55)

- 수치 ↑ / ↓ 의미
  - `risk_discount_scale` **↑**: 리스크 감점 커짐
  - `finance_penalty_scale` **↑**: 재정 감점 커짐
  - `finance_factor_floor` **↑**: 최저 감점 한계 완화

- 과보수 완화 방향
  - scale을 크게 건드리기보다 floor를 조금 올려 “과도한 하방”만 막는 방식 권장

## 4) Package Effects(구조 패널티) 튜닝

### 4-1. hole/roster/cap-flex 계열
- 관련 필드
  - `hole_penalty_scale`, `hole_penalty_exponent`, `hole_penalty_cap_ratio`
  - `roster_excess_cap_ratio`
  - `cap_room_weight_base`, `cap_room_value_per_cap_fraction`, `cap_room_cost_exponent`, `cap_room_abs_cap`
  - `cap_ledger_score_scale`, `cap_ledger_abs_cap`

- 수치 ↑ / ↓ 의미
  - penalty scale/exponent/cap **↑**: 구조적 결손에 더 민감(보수성↑)
  - penalty scale/exponent/cap **↓**: 구조 패널티 완화(보수성↓)

- 과보수 완화 방향
  - 우선 `abs_cap` 계열만 소폭 낮추는 방식 권장(급격한 철학 붕괴 방지)

## 5) Deal Generator 후보 폐기/탐색량 튜닝

### 5-1. 조기 폐기(discard) 임계값
- 관련 필드
  - `discard_if_overpay_below` (기본 -18.0)
  - `discard_if_any_margin_below` (기본 -22.0)
  - `discard_if_reject_margin_below` (기본 -14.0)
  - `discard_if_both_margins_below` (기본 -10.0)

- 수치 ↑ / ↓ 의미
  - 임계값을 **덜 음수로 올리면**(예: -18→-15) 더 빨리 폐기(강화)
  - 임계값을 **더 음수로 내리면**(예: -18→-21) 폐기 완화

- 과보수 완화 방향
  - `discard_if_reject_margin_below`부터 완화(스윗너로 복구 가능한 딜 보존)

### 5-2. sweetener 복구 범위
- 관련 필드
  - `sweetener_max_deficit` (기본 10.0)
  - `sweetener_close_corridor_ratio` (기본 0.12)
  - `sweetener_close_cap` (기본 8.0)

- 수치 ↑ / ↓ 의미
  - deficit/corridor/cap **↑**: “조금 부족” 딜 복구 범위 확대
  - **↓**: 복구 범위 축소

- 과보수 완화 방향
  - `sweetener_close_cap` 또는 `max_deficit`를 소폭 상향

### 5-3. 타깃/파트너 다양성 캡
- 관련 필드
  - `max_partner_repeats` (기본 0=비활성)
  - `target_repeat_penalty` (기본 0.15)

- 수치 ↑ / ↓ 의미
  - penalty/cap 제약 **↑**: 한 상대 집중 억제(다양성↑, 성사율↓ 가능)
  - penalty/cap 제약 **↓**: 성사 가능 상대 집중 허용(성사율↑)

  - **↓**: 탐색량↓, 누락↑

- 과보수 완화 방향
  - 팀/선수 scan cap을 먼저 소폭 상향 후, 성능/시간 예산 확인

## 7) 오케스트레이션 최종 게이트 튜닝

### 7-1. office gate
- 관련 필드
  - `ai_ai_office_gate_min_score` (기본 0.25)
  - `ai_ai_office_gate_min_margin` (기본 -14.0)
  - `ai_ai_office_gate_overpay_slack` (기본 0.5)

- 수치 ↑ / ↓ 의미
  - `min_score` **↑**: veto 증가
  - `min_margin`을 **덜 음수로 올리면** veto 증가
  - `overpay_slack` **↑**: 일관성 검사 완화(veto 감소)

- 과보수 완화 방향
  - `min_score` 소폭 하향 또는 `overpay_slack` 소폭 상향부터 시작

## 8) “철학 유지형” 권장 튜닝 순서

1. **노출량 완화**: listing threshold/cap/cooldown 먼저 조정
2. **협상 지속성 완화**: counter corridor, counter_overpay_fraction 조정
3. **복구 기회 확대**: discard 임계값 일부 완화 + sweetener corridor 확장
4. **최종 veto 완화**: office gate를 마지막에 미세 조정

> 이렇게 하면 “비현실 딜 허용”이 아니라, **좋은 딜이 파이프라인 중간에서 과도하게 잘리는 현상**을 줄일 수 있습니다.

## 9) 체크리스트 (튜닝 실험 운영)

- [ ] 한 번에 1~2개 그룹만 변경
- [ ] 2~4주 시뮬레이션 후 아래 지표 비교
  - 팀당 생성 proposal 수
  - REJECT/COUNTER/ACCEPT 비율
  - office gate veto 건수
  - low-quality 체결(극단 margin) 비중
- [ ] 체결량만 보지 말고, 유저 체감(말이 되는 제안인지) 로그 샘플 확인
