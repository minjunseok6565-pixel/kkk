# 트레이드 잔여 단순화 이슈 코드 실사 노트

작성일: 2026-03-12  
대상: 항목 1,2,3,7,8 (4,5,6 제외)

## 목적
기존 `트레이드_잔여_단순화_확정_리포트.md`의 설명이 실제 코드와 일치하는지 확인하고,
현재 브랜치 기준으로 실제 문제가 어디서/어떻게 발생하는지 근거를 남긴다.

---

## 1) 픽 기대값: standings 선형 매핑 고정

### 코드 확인
- `build_pick_expectations_from_standings()`는 standings 인덱스를 그대로 `expected_pick_number = index + 1`로 넣는다.
- 함수 주석에 `Ignores lottery variance; that's for later.`가 직접 명시되어 있다.
- `build_repo_valuation_data_context()`는 외부 기대값이 없고 standings가 주어지면 위 함수를 자동 호출해 기대값을 만든다.

### 현재 프로젝트 기준 실제 문제
- 픽 가치 계산 엔진(`market_pricing`)은 보호규칙(TOP_N)과 연도할인을 적용할 수 있지만,
  그 입력인 `expected_pick_number` 자체가 standings 선형값이면 상류 입력 분산이 부족하다.
- 따라서 "보호규칙 보정이 있더라도" 출발점이 평균화되어 로터리/점프 리스크를 충분히 반영하지 못한다.
- 또 기대값이 없을 때 `_price_pick()`이 `16.0`(중간픽)으로 폴백하므로, 정보 공백 상황에서는 가치가 더 평탄해진다.

---

## 2) 패키지 효과: archetype/계약부담 축약

### 코드 확인
- `_primary_archetype_tag()`는 role_fit 다중 정보가 있어도 최고 점수 태그 하나만 선택한다.
- 주석에 `This is intentionally simple`이 명시되어 있다.
- `_commitment_metric()`은 커미트먼트를 `salary_now * remaining_years`로 계산한다.
- 주석에 `Options/partial guarantees are ignored in v1.`가 명시되어 있다.

### 현재 프로젝트 기준 실제 문제
- 다기능 선수(예: 온볼+오프볼 겸용)의 조합가치가 단일 태그 버킷으로 축약되어,
  패키지 내 "보완"과 "중복"의 세밀한 차이를 놓칠 수 있다.
- 계약 리스크에서도 팀옵션/부분보장/비보장 구조가 commitment 계산에서 사라져,
  같은 연봉/연차라도 실제 위험도 차이가 충분히 분리되지 않는다.

## 3) fit-swap 적합도: 단순 가중 오버랩

### 코드 확인
- `_need_fit_score()`는 `sum(need_weight * supply[tag]) / sum(need_weight)` 형태의 단순 평균이다.
- `maybe_apply_fit_swap()`는 swap-out 대상 선정과 replacement 후보 개선 판단에 이 점수를 직접 사용한다.

### 현재 프로젝트 기준 실제 문제
- 적합도 산식이 태그별 독립 합산이라 태그 간 상호작용(예: 볼핸들러 + 슈터 조합) 정보를 담지 못한다.
- 결과적으로 fit-swap 단계에서 "태그 점수는 개선"인데 실제 로스터 조합 효율은 낮은 교체가 상위에 남을 수 있다.
- 최근 코드에서 horizon/posture 기반의 2차 랭킹(`_fit_swap_primary_score`)이 추가됐지만,
  진입 필터(최소 개선 여부)는 여전히 `_need_fit_score` 하나에 의존한다.

## 7) CAP_FLEX: commitment 선형 근사

### 코드 확인
- `_cap_flex_adjustment()`의 미래 유연성 파트(B)는 `delta_commitment = in_commit - out_commit`을 사용한다.
- `in_commit/out_commit`은 `_commitment_metric()` 합산으로 계산된다.
- `_commitment_metric()`은 options/partial guarantees 무시 정책을 주석으로 명시한다.

### 현재 프로젝트 기준 실제 문제
- CAP_FLEX 평가가 계약 조항 구조를 무시한 선형 commitment에 기반하므로,
  실제 캡 유연성에서 중요한 "탈출 가능성"(옵션/보장 여부) 구분이 약하다.
- 즉, 동일 총액처럼 보이는 계약군의 미래 리스크 프로파일을 충분히 분해하지 못한다.

---

## 8) 패키지 중복 분류: 단일 archetype 축 압축

### 코드 확인
- `_diminishing_returns()`는 인바운드 선수들을 `_primary_archetype_tag()` 결과로 버킷화한다.
- 버킷 내 선수 수가 많을수록 `diminishing_factors`로 체감 페널티를 부여한다.

### 현재 프로젝트 기준 실제 문제
- 중복 판정의 축이 단일 태그라 다차원 역할 차이(수비 타입/슈팅 중력/온볼 분배)가 반영되지 않는다.
- 따라서 실제로는 상호보완적인 조합도 같은 태그로 묶이면 과대 페널티를 받을 수 있다.
- 반대로 태그만 다르면 실질 중복인데도 페널티가 약해지는 케이스가 남는다.

## 정리
현재 브랜치에서 1,2,3,7,8 항목은 문서 설명이 대체로 맞고,
코드상으로도 "의도된 단순화" 주석/구현이 직접 확인된다.
다만 1번은 `market_pricing`이 보호규칙(TOP_N)을 처리하므로,
"보호를 전혀 반영하지 않는다"가 아니라
"보호 계산의 입력 기대값 생성이 단순하다"로 해석하는 것이 정확하다.
