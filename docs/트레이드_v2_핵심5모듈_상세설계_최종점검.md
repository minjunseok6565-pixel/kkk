# 트레이드 v2 핵심 5모듈 상세설계 최종 점검 리포트

작성일: 2026-03-10  
검토 대상: `docs/트레이드_v2_핵심5모듈_상세설계.md`  
연계 기준: `docs/트레이드_개편_신규시스템_구현구상.md`, `docs/트레이드_잔여_단순화_확정_리포트.md`

## 0) 총평

결론부터 말하면, 현재 상세설계는 **방향성은 매우 정확**하고 (이슈 1/2/3/7/8의 핵심 병목을 정확히 겨냥), SSOT/중복금지/안전전환 원칙도 잘 잡혀 있다.  
다만, 실제 코드베이스와 대조했을 때 **즉시 보완이 필요한 설계 공백 7개(MUST-FIX)**가 있다.

- 핵심: "없는 값을 가정하지 않는다" 원칙을 지키려면, 일부 함수 시그니처/입력 계약을 더 보수적으로 잠가야 한다.
- 핵심: "중복 구현 금지"를 지키려면, 기존 provider/terms/cap_model과 정확히 접합되는 인터페이스를 먼저 확정해야 한다.

아래 항목을 반영하면, 설계가 목표 달성(가정값 0건 + 중복 계산 0건 + 단계적 전환)을 훨씬 안정적으로 달성할 수 있다.

## 1) MUST-FIX (출시 전 필수 보완)

## 1-1. `context_v2`가 사용할 "자산 유니버스 입력"이 정의되지 않음

현재 `ValuationDataProvider`는 `get_*_snapshot(id)` 형태의 **개별 조회 API만 제공**한다. 즉 provider 자체로는 "리그 전체 players/picks/swaps/contracts"를 열거할 수 없다.  
그런데 상세설계의 `ValuationContextV2`는 팀별 매핑/전체 매핑을 전제로 한다.

**문제점**
- builder가 어떤 ID 집합을 기준으로 context를 구성할지 불명확.
- 구현자가 provider 내부 맵(`draft_picks_map`)에 직접 접근하면 Protocol 추상화가 깨짐.

**보완안(필수)**
- `build_valuation_context_v2(...)` 시그니처에 최소 한 가지를 추가:
  1) `asset_ids: Mapping[str, Sequence[str]]` (players/picks/swaps/fixed), 또는
  2) `deal: Deal` 기반으로 필요한 자산만 수집(권장), 또는
  3) provider protocol 확장(`list_pick_ids()` 등) — 영향이 크므로 비권장.

---

## 1-2. `collect_v1_v2_diff()` 시그니처가 비교 대상을 혼동함

현재 설계안은 `collect_v1_v2_diff(v1_ctx: ValuationDataProvider, v2_ctx: ValuationContextV2)`인데, 이는 **v1 계산 결과가 아니라 입력 provider**를 받는다.  
이 상태로는 "계산결과 diff만" 수집한다는 원칙과 충돌한다.

**보완안(필수)**
- diff 함수 입력을 `v1_result`/`v2_result` 형태로 분리:
  - 예: `collect_v1_v2_diff(v1_metrics: Mapping[str, float], v2_ctx: ValuationContextV2)`
  - 또는 deal/team 단위 결과 객체를 직접 입력.
- 원천 데이터 차이 로그는 별도 telemetry로 분리.

---

## 1-3. `pick_distribution.py`의 핵심 타입(`DraftLotteryRules`, `PickTermsProvider`)이 현 코드에 없음

상세설계는 해당 타입을 전제로 하지만 현재 코드베이스에는 동일 타입/프로토콜이 없다.

**문제점**
- 구현 시 임의 타입 추가 가능성이 높아져 SSOT가 분산될 위험.

**보완안(필수)**
- 먼저 `trades/valuation/types_v2.py`(또는 `pick_distribution.py` 내부)에 최소 Protocol/TypedDict를 명시.
- `DraftLotteryRules`는 시즌별 odds/로터리 슬롯 구조를 담는 **순수 데이터 구조**로 고정.
- `PickTermsProvider`는 R2 전용으로 뒤로 미루고, R1 단계에서는 완전히 제거(함수 인자 자체 제거 또는 `NotUsed` 표시).

---

## 1-4. `cap_ledger.py`의 payroll fallback 정의가 현재 입력 시그니처와 충돌

상세설계는 "팀 payroll 입력이 없으면 player snapshot salary 합산"을 말하지만, 제시된 함수 시그니처 입력은 `team_contract_textures` 중심이다.  
이 구조에서는 player snapshot 접근 경로가 없다.

**보완안(필수)**
- 아래 중 하나를 명시:
  1) `team_payroll_map`을 **R1 필수 입력**으로 승격(가장 단순),
  2) `team_players: Mapping[str, Sequence[PlayerSnapshot]]` 추가,
  3) `context_v2`가 payroll 계산 후 `cap_ledger`에만 전달.
- 가정 금지 원칙상, payroll 미존재 시 `source_coverage=false` + 중립치 처리로 고정.

---

## 1-5. 계약 texture의 "부분보장/비보장 반영"은 현재 실존 필드와 미스매치 가능성 높음

현재 `ContractSnapshot`의 정규 필드는 `salary_by_year`, `options`, `meta` 중심이고, 부분보장/비보장 표준 키는 코드상 SSOT로 정의되어 있지 않다.

**문제점**
- 구현자가 임의 meta key를 가정할 가능성.

**보완안(필수)**
- R1에서는 부분보장/비보장 지표를 **선택 지표**로 강등.
- 반영 가능한 키셋을 문서에 명시(예: `meta["guarantee_by_year"]`가 실제 존재할 때만).
- 키 미존재시 반드시 `MISSING_CONTRACT_DETAIL` + `source_coverage["guarantee"]=false`.

---

## 1-6. posture enum 허용 집합을 문서에서 고정해야 함

현재 프로젝트 posture는 실사용상 `AGGRESSIVE_BUY`, `SOFT_BUY`, `STAND_PAT`, `SOFT_SELL`, `SELL`로 수렴한다.  
설계 문서에는 "기존 decision/generation과 동일 집합"만 있고 구체 집합이 없다.

**보완안(필수)**
- `cap_ledger` 문서에 허용 posture enum을 명시 고정.
- 미허용 값은 `UNKNOWN_POSTURE` note + `STAND_PAT` 폴백 규칙 확정.

---

## 1-7. `TradeValuationConfig` 타입은 현재 코드베이스에 실존하지 않음

상세설계의 `build_valuation_context_v2(...)` 시그니처에 `config: TradeValuationConfig`가 포함되어 있으나,
현재 저장소에는 해당 타입 정의가 없다.

**문제점**
- 구현 단계에서 임의 Config 스키마를 즉석에서 만들면 기존 valuation config 체계(`MarketPricingConfig`, `TeamUtilityConfig`, `PackageEffectsConfig`, `DecisionPolicyConfig`)와 중복/충돌할 위험이 큼.

**보완안(필수)**
- R1에서는 `config`를 신규 단일 타입으로 강제하지 말고, 아래 둘 중 하나로 고정:
  1) 기존 config 묶음 사용(예: `ValuationV2ConfigBundle` 내부에 기존 Config 참조),
  2) 최소 `Mapping[str, Any]` + 키 화이트리스트.
- R2에서 정말 통합 타입이 필요하면 `types_v2.py`에 정식 정의하되,
  "기존 Config와의 관계(래핑 vs 대체)"를 먼저 문서로 확정 후 도입.

## 2) 이슈 1/2/3/7/8 달성 관점 점검

## 2-1. 이슈 1(픽 기대값): 달성 가능성 높음, 단 R1 경계 재정의 필요

좋은 점:
- 단일 기대순번 -> 분포(PMF/CDF, 분산, tail) 전환은 문제 원인을 정확히 찌름.
- 보호/스왑의 의미론을 `protection.py`/`pick_settlement.py`에 의존하도록 둔 점이 맞음.

보완 필요:
- R1에서 "standings+existing pick/swap+TOP_N"만 다루고, 나머지는 플래그와 fallback으로 봉인해야 함.
- `SwapSnapshot.year/round`가 Optional 이므로 시나리오 계산 전에 coverage gate 필요.

## 2-2. 이슈 2(역할/계약 질감): 방향 정확, 지표 필수/선택 경계 더 엄격히 필요

좋은 점:
- 역할을 다차원화하고, 계약은 options/remaining schedule 기반으로 분리하는 설계가 정확.

보완 필요:
- role_texture는 현재 데이터에서 확실한 축(creation/spacing/rim/defense)만 필수화한 것이 옳다.
- contract_texture는 guarantee/non-guarantee를 R1 필수로 두면 가정이 들어올 가능성이 크므로 선택화 필요.

## 2-3. 이슈 3(fit-swap): 5모듈만으로는 "완전 해결"이 아니라 "기초 체력 확보"

사실관계:
- 기존 구현구상서의 핵심 모듈은 `lineup_ecosystem.py`까지 포함했지만, 이번 5모듈 범위에는 없다.

해석:
- 5모듈은 fit-swap 고도화의 기반 데이터(역할 texture, 중복 정보)를 제공하는 단계로는 충분.
- 다만 이슈 3을 100% 닫으려면 후속 단계에서 fit_swap 경로 교체가 반드시 필요.

권고:
- 본 설계 문서 완료 기준에 "fit_swap 통합 단계는 후속 스코프"라는 문구를 명시.

## 2-4. 이슈 7(CAP_FLEX): 목표 달성 가능, 단 입력 누락시 중립처리 규칙 명문화 필요

좋은 점:
- commitment 단일 선형식을 분해해 운영축으로 분리한 방향이 정확.

보완 필요:
- payroll/cap/apron/optionality 입력이 일부라도 빠지면 공격적 추정 금지.
- `source_coverage` 기반으로 점수 영향 축소(또는 0) 규칙을 수식 수준에서 명시해야 안전.

## 2-5. 이슈 8(패키지 중복): 역할 texture 도입으로 해결 기반 확보

좋은 점:
- 단일 archetype 축을 버리고 다차원 중복으로 가려는 방향이 정확.

주의점:
- 실제 중복 계산 로직은 `package_effects` 리팩터 단계에서 완성됨.
- 즉 5모듈 구현만으로는 이슈 8이 "완결"되지는 않고, "중복 판단 입력"을 개선하는 단계임을 문서에 명시하는 것이 정확.

## 3) "없는 값을 가정했는지" 점검 결과

판정: **대체로 안전하나, 아래 4개는 가정 유입 위험 존재**

1. `DraftLotteryRules` 실체 미정 상태에서 odds 구조를 임의 생성할 위험  
2. `PickTermsProvider` 미정 상태에서 확장 약정값을 임의 가정할 위험  
3. 계약 `meta`의 부분보장/비보장 키를 임의 가정할 위험  
4. cap_ledger에서 payroll 미입력 시 내부 추정으로 대체할 위험

=> 위 MUST-FIX를 반영하면 "가정값 사용 0건" 목표에 근접 가능.

---

## 4) "기존 값 중복 생성" 점검 결과

판정: **원칙은 잘 잡혔고, 실행 시 아래 중복 위험만 통제하면 됨**

1. remaining years/salary 재계산 중복  
- `contracts/terms.py` 결과만 소비하도록 강제 필요.

2. protection 파서 중복  
- `normalize_protection()`을 반드시 재사용.

3. cap 수식 중복  
- `cap_model.py` 외 수식 구현 금지.

4. fit supply 추출 중복  
- `FitEngine.compute_player_supply_vector()` 외 경로 금지.

5. pick/swap 기대값 이중 반영  
- v2 on 시 `PICK_PROTECTION_EXPECTATION`/`SWAP_EXERCISE_PROB` 기존 보정을 완전히 끄는 스위치 필요.

## 5) 설계 문서에 바로 반영할 수정 제안(요약)

1. `context_v2` builder 입력에 자산 유니버스 정의 추가(필수).  
2. `collect_v1_v2_diff()` 입력을 "계산 결과" 기준으로 수정(필수).  
3. `DraftLotteryRules`/`PickTermsProvider` 타입 정의 선행 또는 R1에서 제거(필수).  
4. `cap_ledger` payroll 입력 계약을 명시적으로 고정(필수).  
5. contract meta 키셋(부분보장/비보장) 존재 기반 처리로 명문화(필수).  
6. posture enum 허용 집합 및 폴백 규칙 명시(필수).  
7. `TradeValuationConfig`를 기존 config 체계로 대체하거나, 신규 생성 시 래퍼 타입으로 스코프 제한(필수).  
8. 이슈 3/8의 "완결 단계"가 package_effects/fit_swap 이관까지임을 DoD에 명시(권장).

## 5-1) "없는 값을 가정" 항목별 처리 전략 (기존값 대체 vs 신규 생성)

아래는 `docs/트레이드_개편_신규시스템_구현구상.md`의 목표(이슈 1/2/3/7/8)를 유지하면서,
상세설계에서 실존하지 않는 값을 어떻게 처리할지에 대한 **실행 결정표**다.

| 항목(현재 가정) | 목표 이슈 | R1 결정 | 이유 |
|---|---:|---|---|
| `TradeValuationConfig` | 공통(1/2/3/7/8) | **기존값 대체** | 기존 `MarketPricingConfig`/`TeamUtilityConfig`/`PackageEffectsConfig`/`DecisionPolicyConfig` 조합으로 충분. 새 통합 Config를 바로 만들면 중복 위험이 큼. |
| `DraftLotteryRules` (미정 타입) | 1 | **신규 생성(최소 타입)** | 현 코드에 동일 SSOT가 없으므로, `pick_distribution.py`에서만 쓰는 순수 데이터 타입(odds table)로 최소 정의 필요. |
| `PickTermsProvider` (R2) | 1 | **R1 제거, R2 신규** | R1 목표는 기존 pick/swap/protection만으로도 달성 가능. 조기 도입 시 없는 입력을 가정할 유인이 커짐. |
| `team payroll` 미입력 fallback 값 | 7 | **기존값 대체** | 이미 player snapshot의 `salary_amount`, contract schedule(`contracts/terms.py`)가 존재. 없으면 추정하지 않고 coverage false 처리. |
| `contract` 부분보장/비보장 표준 키 | 2,7 | **R1 기존값 대체 + 제한적 신규** | R1은 `ContractSnapshot.options`/`salary_by_year` 기반만 필수. 보장 세부키는 실제 meta에 존재할 때만 소비(미존재시 미계산). |
| `asset universe` 전체 목록 | 공통 | **기존값 대체** | provider 직접 열거 API가 없으므로, deal 기반 자산 집합(또는 상위 orchestration이 주입한 ID 목록)을 입력으로 받는 방식이 안전. |

원칙:
- R1은 가능한 한 **기존 실존 값으로 대체**한다.
- 신규 생성이 필요한 값은 `DraftLotteryRules`처럼 "대체 가능한 기존 SSOT가 없는" 경우로 한정한다.
- 신규 생성 시에도 계산 로직은 기존 SSOT(`protection.py`, `pick_settlement.py`, `contracts/terms.py`, `cap_model.py`)를 소비만 해야 한다.

## 6) 최종 판정

- 현재 상세설계는 **핵심 방향이 올바르고, 프로젝트의 SSOT 철학과도 정렬**되어 있다.  
- 다만 위 MUST-FIX 7개를 보완하지 않으면 구현 시점에
  - 없는 값 가정,
  - provider 우회 접근,
  - legacy/v2 이중반영
  이 발생할 가능성이 있다.

**따라서 권고 판정은 "승인(조건부)"**: MUST-FIX 반영 후 구현 착수 권장.
