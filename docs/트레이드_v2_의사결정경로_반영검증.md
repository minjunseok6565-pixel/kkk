# 트레이드 v2 의사결정 경로 반영 검증 (현재 패치 기준)

작성일: 2026-03-10

## 1) 결론

현재 패치 기준으로 **실제 의사결정 주 경로에 v2 반영이 이루어졌음**.

- 기본 플래그 동작이 ON으로 변경되어, 인자 미지정/환경변수 미지정 시에도 `context_v2` 경로가 활성화됨.
- service의 주 경로(`evaluate_deal_for_team`)에서 stage(`shadow/gradual/full`)에 따라 실제 평가 파이프라인에 점진 반영됨.
- package_effects의 핵심 두 축이 모두 texture 기반으로 동작함:
  - Diminishing returns: `RoleTexture` matrix/overlap
  - CAP_FLEX future commitment: `ContractTexture` 기반 합산

## 2) 경로 검증 상세

### 2-1. 기본 플래그 ON

`service.py`에서 `use_valuation_context_v2` 해석 시 기본값이 `default=True`로 적용됨.

### 2-2. 주 경로(stage) 연동

- `use_v2`가 켜지면 `stage`에 따라 `PackageEffectsConfig`가 service에서 생성되어 evaluator에 주입됨.
  - `gradual`: texture 가중치 완충
  - `full`: 기본 texture 가중치
- evaluator 호출 시 `package_config=package_cfg`가 전달되어 실제 package 계산에 반영됨.

### 2-3. context_v2 실제 호출 및 진단 기록

- `build_valuation_context_v2(...)`가 service 주 경로에서 직접 호출됨.
- 결과 diagnostics/diff가 `evaluation.meta["context_v2"]`에 기록됨.
- 예외 발생 시도 서비스 실패 대신 `context_v2.error` 메타로 폴백.

### 2-4. package_effects 실제 의사결정 반영

- `apply()`에서 diminishing returns는 `_diminishing_returns_texture(...)`를 사용.
- `DIMINISHING_RETURNS` step 메타에 `pairwise_overlap`를 포함.
- `_cap_flex_adjustment()`는 `_build_player_contract_textures()`를 통해 `ContractTexture`를 만들고,
  future commitment를 texture 결합 점수로 계산.

## 3) 검증 판정

- **반영성**: 완료 (주 경로 연결됨)
- **기본 활성화**: 완료 (default on)
- **점진 반영 제어**: 완료 (`shadow/gradual/full`)
- **레거시 대비 안전성**: 양호 (stage별 완충 + diagnostics + error 폴백)


## 4) "v1 완전 은닉 + 전면 v2 작동" 달성 여부 점검

요청 의도 기준으로 판정하면, **아직 전면 v2 100% 전환은 미달성**이다.

### 4-1. 달성된 부분

- package 의사결정 핵심(중복/체감, CAP_FLEX future commitment)은 v2 texture 기반으로 동작한다.
  - diminishing: `RoleTexture` overlap matrix
  - CAP_FLEX future: `ContractTexture` 결합 점수
- service 기본 플래그가 ON이라 context_v2 계산/진단은 기본적으로 수행된다.

### 4-2. 아직 v1 레거시가 남아 있는 부분

1. **valuation provider는 여전히 v1 `data_context` 기반**
   - service 주 경로가 `build_repo_valuation_data_context(...)`를 사용한다.

2. **market pricing의 픽 평가는 v1 `PickExpectation.expected_pick_number` 경로 중심**
   - `market_pricing.py`가 `expected_pick_number`를 계속 핵심 입력으로 사용한다.
   - 즉 pick distribution v2를 시장가 산식의 SSOT로 완전 전환한 상태는 아니다.

3. **service stage 기본값이 `shadow`**
   - context_v2 자체는 빌드되지만, 기본 stage는 "진단/메타 중심" 의미를 가진다.
   - (현재 package_effects는 texture 경로가 직접 사용되므로 체감 반영은 있으나,
     아키텍처 관점의 전면 v2 전환/레거시 은닉과는 다르다.)

4. **레거시 helper 코드 잔존**
   - `_commitment_metric()` 함수 정의가 파일에 남아 있다(현재 미사용).
   - 미사용 레거시 코드 정리 단계가 필요하다.

### 4-3. 최종 판정

- **현재 상태**: 하이브리드(v2 핵심 반영 + v1 인프라/산식 일부 공존)
- **요청 목표("v1 완전 레거시화, 전면 v2") 기준**: **미완료**

### 4-4. 전면 v2 전환을 위한 즉시 후속 작업

1. service/provider를 `context_v2` 중심 입력으로 승격하고, v1 `data_context` 의존 경로를 옵트아웃 처리
2. `market_pricing` 입력을 `PickExpectation` 중심에서 `PickDistributionBundle` 중심으로 교체
3. `valuation_context_v2_stage` 기본값을 `full`로 상향하고 `shadow`를 디버그 전용으로 축소
4. `_commitment_metric` 등 미사용 레거시 helper 삭제 + 관련 테스트 정리


## 5) 최신 패치 반영 후 재검증 (v2 전면동작 관점)

이번 패치에서 다음이 추가 적용됨.

1. service 경로에서 **v2 provider adapter**를 사용해, 평가 시 pick 입력이 `context_v2.pick_distributions`를 우선 사용하도록 전환
2. market_pricing이 `PickExpectation`만 보던 경로에서 벗어나, `PickDistributionBundle`을 1순위 입력으로 사용
3. `valuation_context_v2_stage` 기본값을 `full`로 상향, `shadow`는 texture 가중치 0의 디버그 전용 모드로 축소
4. `_commitment_metric` 레거시 helper 삭제

### 5-1. 전면 v2 판정(업데이트)

- **의사결정 핵심 경로(패키지/캡/픽 가격입력)**: v2 중심으로 전환됨
- **v1 경로 상태**:
  - snapshot 로딩용 base provider는 내부 위임으로 남아 있으나,
    평가 입력(픽 기대/분포)은 adapter를 통해 v2 산출물을 우선 사용
- **종합 판정**: 실운영 의사결정 경로 기준으로는 사실상 v2 전면 동작에 근접/달성

### 5-2. 잔여 기술 부채(권장)

- `build_repo_valuation_data_context`를 snapshot loader로 쓰는 내부 위임층을
  장기적으로 `context_v2` native provider로 완전 교체하면 구조 일관성이 더 높아짐.
