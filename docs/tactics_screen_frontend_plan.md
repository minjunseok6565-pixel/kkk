# 전술 설정 화면(`tactics-screen`) 프론트엔드 개선 기획안 v1.0

## 1) 문서 목적
- 다음 작업에서 **실제 프론트엔드 구현**을 바로 시작할 수 있도록, 전술 화면의 정보 구조/레이아웃/비주얼/인터랙션/검증 기준을 확정한다.
- 본 기획안은 **상업용 게임 UI 퀄리티**를 목표로 하며, "툴"이 아닌 "코칭 커맨드 센터" 경험을 제공한다.
- **신규 API 추가 없이** 기존 조회 데이터만 사용한다.

---

## 2) 작업 범위(고정)

### 포함
- `static/NBA.html` 내 `#tactics-screen` 섹션 구조 개선
- `static/NBA.css` 내 전술 화면 전용 스타일 개선
- `static/NBA.js` 내 전술 화면 렌더/상태 계산/UI 피드백 로직 개선

### 제외
- `#start-screen`, `#team-screen`, `#main-screen`, `#schedule-screen`, `#training-screen`, `#standings-screen`, `#college-screen`, `#medical-screen` 등 **타 화면 변경 금지**
- 신규 백엔드 API, 신규 DB 스키마, 기존 API 응답 구조 변경

---

## 3) 디자인 목표 & UX 원칙
1. **의사결정 중심 UI**: 입력 요소 나열이 아니라 "전술 결정 → 영향 확인" 흐름을 만든다.
2. **즉시 피드백**: 역할/출전시간 변경 시 우측 인사이트가 즉시 갱신된다.
3. **프로덕션 완성도**: 간격 체계, 타이포, 상태색, 호버/포커스 일관성 확보.
4. **읽기 쉬운 데이터 밀도**: 한 화면에서 핵심 정보를 보되, 과밀하지 않게 구획화.
5. **확장 가능 구조**: 추후 저장 API가 붙어도 구조를 재작업하지 않도록 설계.

## 4) 정보 구조(IA)

전술 화면을 다음 4개 레이어로 재구성한다.

### A. Top Hero (상단 요약 바)
- 팀명 / 시즌 문맥(가능한 기존 값 재사용)
- 현재 공격/수비 스킴 배지
- 핵심 지표(요약 칩)
  - 총 배정 출전시간(목표 240)
  - 스타터 평균 MIN
  - 벤치 평균 MIN
  - 역할 다양성 지수(단순 분산 기반)
- 액션 버튼
  - `저장`(현재 구조상 실제 저장 미구현이면 disabled + 툴팁)
  - `되돌리기`(초기 draft로 복귀)

### B. Scheme Preset Strip (스킴 프리셋 영역)
- 기존 버튼형(`공격 스킴`, `수비 스킴`)을 카드형 선택 UI로 개선
- 각 카드 항목:
  - 스킴명(사용자 친화 라벨)
  - 한 줄 설명
  - "추천 역할 조합" 미리보기(기존 role key 조합 기반)
- 선택 시 하단 라인업의 수비 역할 호환성 즉시 재평가

### C. Lineup Workbench (핵심 편집 영역)
- 좌측: **선발 5인 카드형 테이블**
- 우측: **로테이션 5인 카드형 테이블**
- 각 행 컬럼:
  - 선수(셀렉트)
  - 공격 역할(셀렉트)
  - 수비 역할(셀렉트)
  - 출전시간(숫자 + 슬라이더 동기화)
  - 상태 아이콘(적합/주의/충돌)
- 행 단위 상호작용:
  - 변경 시 하이라이트 애니메이션 600~900ms
  - 수비 역할 중복 시 즉시 경고 + 원복

### D. Insight & Bench Panel (우측 인사이트/하단 로스터)
- 우측 인사이트 패널(스크롤 고정 가능):
  - 출전시간 분포 막대(선발/벤치)
  - 포지션 커버리지(가능한 경우)
  - 역할 중복/결손 경고 리스트
  - "현재 전술의 강점/리스크" 규칙 기반 메시지
- 하단 로스터 리스트:
  - 현재와 동일한 데이터 소스 사용
  - 선택 가능한 후보군 가시성 강화(검색/필터는 선택)

## 5) 데이터 소스 & API 재사용 규칙

## 사용 가능한 기존 조회 데이터(현재 코드 기준)
- 팀 로스터: `GET /api/team-detail/{teamId}`
- 현재 게임 문맥(날짜 등): `GET /api/state/summary` (필요 시)
- 전술 화면 상태: 프론트 draft(`state.tacticsDraft`) + 로스터(`state.rosterRows`)

### 신규 API 금지 원칙
- 추가 endpoint 생성 금지
- 서버 저장 기능이 없으면 "로컬 임시 적용" UX로 명시
- 필요한 계산은 JS에서 파생 지표로 생성

### 파생 지표(프론트 계산)
- `totalMinutes = starters+rotation minutes 합`
- `minutesDelta = 240 - totalMinutes`
- `roleDiversity = 고유 공격역할 수 / 전체 슬롯`
- `defenseRoleConflicts = 중복 role 개수`
- `benchCreationRisk = 벤치 볼핸들러/creator role 부족 여부(룰 기반)`

## 6) UI 컴포넌트 사양

## 6-1. 스킴 카드(SchemeCard)
- 상태: default / hover / selected / disabled
- 내용: 제목, 설명, 태그 2~3개, 선택 체크 아이콘
- 접근성: 키보드 포커스 가능, `aria-pressed`

## 6-2. 전술 행 카드(TacticsRow)
- 기본 높이: 64~72px
- 내부 5분할 그리드
- 상태 배지:
  - `OK`(녹색)
  - `WARN`(앰버)
  - `ERROR`(레드)

## 6-3. 요약 KPI 칩(KPIChip)
- 숫자 + 라벨
- 조건부 색상:
  - 정상(중립)
  - 목표 이탈(경고)

## 6-4. 경고 피드(AlertFeed)
- 우선순위 정렬: ERROR > WARN > INFO
- 최대 5개 노출, 나머지는 "더보기"

## 8) 인터랙션 상세

1. 스킴 선택
- 카드 클릭 → 선택 상태 반영
- 수비 스킴 변경 시 허용 role 목록 업데이트
- 기존 role이 비허용이면 자동 치환 + 알림 토스트

2. 출전시간 편집
- 숫자 입력/슬라이더 동기화
- 0~48 clamp
- 총합 240 기준 실시간 색상 피드백

3. 역할 중복 제어
- 현재처럼 "수비 역할 중복 금지" 유지
- alert() 대신 인라인 에러/토스트로 교체

4. 변경사항 추적
- 화면 진입 시 snapshot 저장
- 변경 발생 시 `변경됨` 배지
- `되돌리기` 버튼으로 snapshot 복귀

## 9) 구현 지침(파일 단위)

## `static/NBA.html`
- `#tactics-screen` 내부를 다음 구조로 재배치
  - `tactics-hero`
  - `tactics-scheme-strip`
  - `tactics-workbench`(lineup + insight)
  - `tactics-roster-box`
- id는 기존 JS 연동을 고려해 유지 가능한 것은 유지
- 신규 class는 전술 화면 네임스페이스 접두(`tactics-`) 사용

## `static/NBA.css`
- 전술 화면 전용 블록을 독립 섹션으로 정리
- 기존 공통 `.btn`는 재사용하되, 전술 전용 변형 클래스 추가
- 다크 테마 카드/칩/배지/경고 스타일 정의
- hover/focus-visible/disabled 상태 명시

## `static/NBA.js`
- 기존 함수 확장 중심(전면 재작성 지양)
  - `renderTacticsScreen()` : 요약/인사이트 계산 + 렌더
  - `bindLineupEvents()` : 인라인 검증/피드백
  - 파생 계산 함수 분리(예: `computeTacticsInsights`)
- 기존 API 호출 경로 변경 금지
- 타 화면 로직 분기 변경 금지

## 10) 품질 기준(Definition of Done)

### 기능
- 스킴 변경, 선발/로테이션 편집, 출전시간 편집이 기존처럼 동작
- 수비 역할 중복 제약이 정상 적용
- 출전시간 합계 및 경고가 정확히 계산

### UI/UX
- 전술 화면만 시각적으로 고급화되고, 타 화면 스타일은 영향 없음
- 1920x1080 기준 한 화면에서 핵심 정보 확인 가능
- 키보드 탭 이동/포커스가 시각적으로 명확

### 기술
- 콘솔 에러 없음
- 기존 화면 전환 흐름(showTacticsScreen/Back) 정상
- 불필요한 전역 변수/하드코딩 증가 최소화
