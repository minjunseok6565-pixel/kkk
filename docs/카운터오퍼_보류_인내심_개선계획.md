# 트레이드 카운터 오퍼 개선 계획

## 1) 목표
- 기존 프로젝트가 이미 계산/저장하고 있는 값만 최대한 재사용해, 카운터 오퍼의 현실감과 협상 종료 로직을 개선한다.
- 개선 포인트는 3가지다.
  1. `COUNTER` 판정 시 즉시 카운터 vs 협상 보류(유예) 분기
  2. 보류 후 며칠 관찰 뒤 `REJECT` 또는 `COUNTER` 재결정
  3. AI 전용 인내심(`patience budget`)으로 무한 카운터 루프 방지

---

## 2) 구현 원칙
- **SSOT 유지**: 가치평가/합법성 검증/카운터 생성은 기존 경로를 그대로 사용한다.
- **상태머신 확장 최소화**: 세션 `phase/status`를 크게 바꾸지 않고, 필요한 필드만 `session` 메타에 추가한다.
- **기존 신호 재사용 우선**: GM 성향, 관계치, 오퍼 메타, 데드라인 압력 등 이미 있는 값을 우선 활용한다.
- **튜닝 가능성 확보**: 하드코딩 대신 `OrchestrationConfig` 또는 counter config 쪽 파라미터로 노출한다.

---

## 3) 필요한 값과 "어디서 끌어올지"

아래 표는 구현에 필요한 신호와 현재 소스(파일/필드)를 명시한다.

| 분류 | 필요 값 | 소스(현재 프로젝트) | 사용 목적 |
|---|---|---|---|
| 가치판단 | `decision.verdict`, `required_surplus`, `overpay_allowed`, `decision.meta.corridor`, `decision.reasons` | `app/api/routes/trades.py`의 commit 평가 결과 (`evaluate_deal_for_team`) | COUNTER 내부의 즉시/보류 점수 계산 |
| 거래 밸류 | `eval_summary.net_surplus`, `incoming_total`, `outgoing_total` | `app/api/routes/trades.py` 평가 요약 | 제안이 경계선인지(애매함) 판단 |
| 관계 | `session.relationship.trust/fatigue/promises_broken` | `trades/negotiation_store.py` 세션 스키마 | 보류 확률/인내심 상한 계산 |
| GM 성향 | `NegotiationToughness`, `RelationshipSensitivity` | `gm_profiles` → `decision_context.py`(GMTradeTraits) | 팀별 협상 스타일 반영 |
| 팀 성향(기존) | `patience` | `data/gm_profiles.json`, `data/team_situation.py` | 인내심 cap의 기본치/보정치 |
| 시간 맥락 | `deadline_pressure` | `DecisionContext.deadline_pressure`, `offer_meta.deadline_pressure` | 보류 기간/종료 성향 조절 |
| 제안 톤/주도 | `offer_tone`, `initiator_team_id`, `proposer_team_id`, `recipient_team_id` | `trades/orchestration/promotion.py`에서 `offer_meta` 저장 | AI가 먼저 제안한 세션 cap 가산 |
| 자동종료 인프라 | `auto_end` 구조체, `mark_auto_ended`, `evaluate_and_maybe_end` | `trades/negotiation_store.py`, `trades/orchestration/ai_end_policy.py` | 보류 만료/종료 처리 재사용 |
| 현재 카운터 생성기 | `build_counter_offer(...)` | `trades/counter_offer/init.py`, `builder.py` | 즉시 카운터 또는 보류 후 카운터 생성 |

### 3-1) 세션에 새로 저장할 최소 필드
기존 `session` dict에 아래를 추가한다.

- `counter_control` (dict)
  - `defer_active: bool`
  - `deferred_until: YYYY-MM-DD | null`
  - `defer_reason: str | null` (예: `WAIT_MARKET_SIGNAL`, `DEADLINE_MONITOR`)
  - `ai_patience_used: float`
  - `ai_patience_cap: float`
  - `counter_rounds: int`
  - `last_counter_action_at: ISO datetime | null`

> 주의: 팀 운영 성향의 `patience`(team_situation용)와 이름 충돌을 피하려면, 세션 내부 키는 `ai_patience_*`로 고정한다.

## 4) 의사결정 설계

## 4-1) COUNTER 내부 분기: 즉시 카운터 vs 보류
`verdict == COUNTER`일 때 바로 `build_counter_offer`를 호출하지 않고, 먼저 `defer_score`를 계산한다.

예시 점수식(초안):

```text
defer_score =
  + w_margin * near_threshold_score
  + w_fatigue * norm(fatigue)
  + w_low_trust * norm(-trust)
  + w_broken * norm(promises_broken)
  + w_tough * eff_neg_tough
  - w_deadline * deadline_pressure
```

- `near_threshold_score`: `required_surplus - net_surplus`가 작을수록 높게
- `deadline_pressure`가 높으면 보류보다 즉시 결정 성향
- threshold 초과 시 `DEFER`, 아니면 기존처럼 즉시 카운터 생성

## 4-2) 보류 상태 동작
- `DEFER`면 세션을 당장 닫지 않고 `defer_active=true`, `deferred_until=today+N` 저장
- `phase`는 신규 `COUNTER_DEFERRED` 또는 기존 `COUNTER_PENDING` + defer flag 중 택1
  - 권장: 신규 phase(`COUNTER_DEFERRED`)로 UI/로그 가독성 확보
- 보류 기간 동안 커밋 호출 시 메시지로 "검토 중" 반환(거절 아님)

## 4-3) 보류 만료 후 재평가
`today >= deferred_until`이면 다음 중 하나를 수행
1. 최신 제안을 다시 평가
2. 결과가 `COUNTER`면 카운터 생성 시도
3. 카운터 생성 실패/점수 미달 시 `REJECT`

## 5) AI 인내심(patience budget) 설계

## 5-1) 인내심 상한(`ai_patience_cap`) 계산
기본식(초안):

```text
cap = base_cap
    + k_profile * profile_patience
    + k_rel_pos * max(trust, 0)
    - k_rel_neg * fatigue
    - k_broken * promises_broken
    + k_initiator * is_ai_initiated
```

- `profile_patience`: `gm_profiles`의 `patience` 재사용
- `is_ai_initiated`: `offer_meta.initiator_team_id != user_team_id`면 1
  - 즉, AI가 먼저 던진 딜이면 cap를 높인다(요구사항 반영)

## 5-2) 인내심 소모(`ai_patience_used`) 규칙
카운터 라운드마다 소모:
- 기본 `+1.0`
- 유저 역카운터가 AI 직전 카운터와 큰 차이면 추가 `+delta`
- `defer`를 선택한 라운드는 `+0.5` (가벼운 소모)

## 5-3) 폭발 조건
- `ai_patience_used >= ai_patience_cap`이면 협상 종료
- 종료 방식은 우선 `REJECTED`(명시 거절) 권장
  - 또는 `EXPIRED_BY_AI`를 재사용하되 reason을 `PATIENCE_EXHAUSTED`로 분리

## 6) 구현 단계(현실적인 순서)

### Phase A (MVP: 1차 배포)
1. 세션 스키마 확장 (`counter_control`)
2. commit 경로에서 `COUNTER` 내부 분기 함수 추가
3. 보류 상태 저장 + 보류 메시지 반환
4. 보류 만료 시 재평가/거절

### Phase B (2차)
1. AI 인내심 cap/used 도입
2. 라운드별 소모 반영
3. cap 초과 시 즉시 종료

### Phase C (3차)
1. 튜닝 파라미터를 config로 노출
2. 텔레메트리 추가(보류율, 인내심 종료율, 평균 카운터 라운드)
3. 팀 성향별 A/B 튜닝

## 8) 리스크 및 완화
- 리스크: 보류가 잦으면 유저가 답답함
  - 완화: deadline 근처는 보류 페널티, 보류 최대 횟수 제한
- 리스크: phase 증가로 UI 분기 누락
  - 완화: `can_open/can_commit` 조건에 신규 phase 명시 + 회귀 테스트
- 리스크: 인내심 파라미터 과민
  - 완화: 팀 공통 기본값 + 작은 범위로 점진 튜닝

## 9) 성공 지표 (ROI 측정)
- 평균 협상 라운드 수(무한루프 감소 여부)
- `counter -> counter -> counter` 장기 루프 비율
- 보류 후 최종 결과(거절/카운터/성사) 분포
- 유저 체감 지표(세션당 이탈률, 협상 완료율)

---

## 10) 최종 결론
- 본 개선안은 현재 코드베이스와의 **정합성이 높고 구현 난이도가 관리 가능한 편**이다.
- 특히 기존 `relationship`, `GM traits`, `offer_meta`, `auto_end` 인프라를 재활용하므로 신규 시스템 대비 개발 효율이 높다.
- 권장 전략은 **MVP(보류 분기) -> 인내심 도입 -> 튜닝/텔레메트리**의 3단계 점진 배포다.
