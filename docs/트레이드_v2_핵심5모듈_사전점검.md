## 0) 결론 요약

판정: **"방향성은 매우 좋지만, 현재 문서 상태 그대로는 목표(이슈 1/2/3/7/8의 100% 달성) 보장이 안 됨"**.

핵심 이유:
1. **이슈 3(라인업 생태계 fit) 구현 책임이 5모듈 범위에서 사라짐**
   - 구현구상서는 `lineup_ecosystem.py`를 핵심 모듈로 명시했지만, 5모듈 상세설계에서는 제외됨.
2. **일부 입력값을 "있다"고 전제한 구간이 있음**
   - `team_payroll_map`, 팀별 계약 texture 묶음, 시즌별 standings/lottery 입력의 공급 경로가 불명확.
3. **기존 시스템과 중복 계산 위험이 남아 있음**
   - pick 가치 보정, fit/need 계산, package_effects의 기존 경로와 v2 경로 동시 적용 방지 규칙이 더 구체화돼야 함.

즉, **현재 설계는 "95% 수준"의 골격은 맞지만, 구현 직전 체크리스트 기준으로는 차단 이슈가 존재**.

## 1) 목표 정합성 점검 (이슈 1/2/3/7/8)

### 1-1. 이슈 1 (픽 기대값) — 대체로 충족, 단 입력 계약 보강 필요

좋은 점:
- 단일 기대순번에서 PMF/CDF/분산 기반으로 전환하는 방향이 정확함.
- 보호는 SSOT 정규화(`normalize_protection`)를 사용하도록 제한해 중복 구현 리스크를 낮춤.
- `DraftLotteryRules`를 이미 프로젝트가 보유하고 있어 "없는 룰 엔진 가정"이 아님.

보강 필요:
- `build_pick_distributions_from_standings()` 입력으로 `standings_order_worst_to_best`를 요구하는데, **상위 orchestrator가 이를 어디서 항상 공급하는지 계약이 문서화되지 않음**.
- `season_rules`를 외부에서 넘기는 구조인데, 시즌 미매칭 시 fallback 정책(중립 PMF, 에러, telemetry)이 상세설계에 고정돼야 함.

### 1-2. 이슈 2 (역할/계약 질감) — 충족 가능, 단 경계 정의 추가 필요

좋은 점:
- 역할은 `role_fit` + `fit_engine` 공급벡터 재사용으로 설계되어 중복 구현 방지 원칙과 일치.
- 계약은 `contracts/terms.py`를 명시적으로 SSOT로 고정해 기존 선형 근사식 대체 방향이 타당.

보강 필요:
- `ContractTexture.matching_utility` 산식의 입력 경계가 모호함. 새 규칙을 만들면 salary matching SSOT와 어긋날 수 있으므로, `cap_model`/기존 salary matching 파라미터 참조 범위를 문서에 명시해야 함.
- "부분보장/비보장 제외"는 R1에서는 합리적이나, 이 경우 `source_coverage` reason code를 강제해야 계약 디테일 누락이 설명 가능해짐.

### 1-3. 이슈 3 (fit-swap 적합도) — **현재 설계만으로는 미충족 위험 높음**

핵심 갭:
- 구현구상 문서에는 `lineup_ecosystem.py`가 포함되어 있었지만, 5모듈 상세설계에서는 제외됨.
- 현행 `fit_swap.py`의 핵심 `_need_fit_score()`는 여전히 단순 weighted overlap이라, **대체 모듈/연결점이 없으면 이슈 3이 남음**.

권고:
- 선택지 A: 5모듈 범위에 `lineup_ecosystem.py`를 다시 포함(실질적으로 6모듈).
- 선택지 B: `role_texture.py`에서 생태계 점수까지 포함하되, 함수 계약에 `lineup unit` 입력과 출력(`ecosystem_fit`)을 명시.
- 어떤 선택이든 `fit_swap.py` 교체 포인트(함수명/호출부)를 설계 문서에 고정해야 함.

### 1-4. 이슈 7 (CAP_FLEX) — 방향 정확, 입력 공급선 명확화 필요

좋은 점:
- CAP_FLEX를 package의 부수항이 아니라 별도 ledger 차원으로 분리한 점이 적절함.
- `cap_model.py` 조회만 허용한 원칙은 SSOT 보존에 맞음.

보강 필요:
- `build_cap_ledgers()` 필수 입력인 `team_payroll_map`이 **현재 valuation data provider 계약에 없음**.
- 팀 payroll 추출 책임(누가, 어느 단계에서, 어떤 소스에서)을 `context_v2` 설계에 명시하지 않으면 구현 시 "임시 계산"이 생길 가능성이 큼.

### 1-5. 이슈 8 (패키지 중복) — 역할 texture만으로는 일부 미달 가능

좋은 점:
- 단일 archetype 버킷을 해체하려는 방향은 정확함.

보강 필요:
- 이슈 8은 기능 중복 + 로테이션 충돌 + connector 가치까지 포함해야 하는데, 현재 5모듈 중 이를 최종 합성하는 책임 모듈이 불명확.
- `package_effects` 교체 시점에서 "기존 diminishing 로직 제거"와 "신규 다차원 overlap 적용"의 동시성 조건을 추가해야 이중 반영을 방지할 수 있음.

## 2) "없는 값 가정" 여부 점검

아래 항목은 **현 설계가 입력 존재를 암묵 가정하는 위험 지점**이다.

1. `ValuationContextV2.contracts` 전체 맵
   - `ValuationDataProvider` protocol에는 전체 contract map 조회 API가 없다.
   - 현재 provider는 개별 player snapshot 내 contract attach 또는 내부 `contracts_map`만 제공한다.
   - 조치: `list_contract_snapshots()` 같은 명시 API 추가, 또는 `context_v2`가 player traversal로 계약을 수집한다는 규약 명시.

2. `team_payroll_map`
   - cap_ledger 필수 입력으로 설계됐지만 provider 표준 계약엔 없음.
   - 조치: context builder 입력 필수값으로 강제하거나, 별도 `PayrollProvider` protocol 정의.

3. `standings_order_worst_to_best` 상시 제공
   - pick_distribution 필수 입력인데, valuation service 체인에서 누락 가능성이 있음.
   - 조치: 누락 시 `source_coverage=false` + `MISSING_INPUT_STANDINGS_ORDER`를 강제하고, 중립 fallback을 문서화.

4. `DraftLotteryRules` 시즌 매칭
   - 시즌별 rules registry는 있으나, 요청 시즌 룰 부재 시 동작이 설계에 고정되지 않음.
   - 조치: 시즌 미존재 시 nearest/기본/에러 중 1개를 정책으로 고정하고 telemetry code를 명시.

## 3) "기존 값/로직 중복 생성" 위험 점검

### 3-1. pick 경로 이중 보정 위험 (높음)
- 현행 `market_pricing.py`에는 `PICK_PROTECTION_EXPECTATION`, `SWAP_EXERCISE_PROB` 단계가 존재.
- v2 distribution 적용 후 기존 단계를 끄지 않으면 동일 의미를 2회 반영할 가능성이 큼.
- 상세설계의 "비활성화" 문장을 구현 체크리스트의 **테스트 가능한 불변조건**으로 승격 필요.

### 3-2. fit 계산 중복 위험 (중간~높음)
- `fit_engine.py`는 이미 `role_fit/attrs -> supply` 계산을 SSOT로 제공.
- `role_texture.py`에서 유사 추출 로직을 직접 쓰면 중복/드리프트 발생.
- 조치: role_texture는 "벡터 변환만" 수행하고 공급벡터 추출은 fit_engine 위임을 함수 시그니처/테스트로 고정.

### 3-3. 계약 잔여연차/잔여금 중복 계산 위험 (중간)
- `contracts/terms.py`가 이미 남은 연차/총액 SSOT.
- contract_texture에서 산식 편의상 자체 계산을 넣으면 package_effects 과거 문제를 반복.
- 조치: remaining years/total을 반환하는 내부 helper도 금지하고 SSOT 함수 직접 호출 원칙 유지.

### 3-4. CAP/APRON 계산 중복 위험 (중간)
- `cap_model.py` 외부에서 시즌 cap 추정식을 넣으면 중복.
- 조치: cap_ledger는 `CapModel.numbers_for_season()` 또는 env accessor만 사용하도록 제한.

## 4) 구현 전 반드시 문서에 추가할 "차단 해소 항목" (필수)

1. **이슈 3 책임 복원**
   - `lineup_ecosystem.py`를 5모듈 범위에 포함하거나, 동등 기능을 어느 모듈이 맡는지 명시.

2. **입력 공급 계약 명문화**
   - `team_payroll_map`, standings, season_rules, contract list의 공급자/누락 시 fallback reason code를 표준화.

3. **dual-read 이중반영 차단 매트릭스 추가**
   - pick/role/contract/cap 각각에 대해 "v1 off 스위치"를 표로 고정.

4. **package_effects 치환 단계의 원자성 규칙**
   - 신규 component on 시 `_primary_archetype_tag`, `_commitment_metric`, `_cap_flex_adjustment`의 어떤 분기를 동시에 off할지 명시.

5. **coverage reason code 사전 확정**
   - 최소 코드셋 예: `MISSING_INPUT_STANDINGS_ORDER`, `MISSING_INPUT_TEAM_PAYROLL`, `MISSING_INPUT_CONTRACT_SNAPSHOT`, `MISSING_INPUT_ROLE_FIT`, `UNSUPPORTED_RULE_PROTECTION`.

## 5) 최종 판정

- 현재 설계는 **핵심 방향(SSOT 준수, 무가정, 중복방지, 단계전환)** 측면에서 매우 우수하다.
- 다만 아래 3가지를 보강해야 "우리 목표 100% 달성 가능" 판정을 줄 수 있다.
  1. 이슈 3의 구현 책임 복원(모듈 경계 재확정)
  2. 입력 공급 계약(특히 payroll/standings/season rules) 명문화
  3. v1/v2 이중반영 차단을 테스트 가능한 불변조건으로 격상

위 3가지만 반영되면, 5모듈(또는 6모듈 보정) 기반으로 이슈 1/2/3/7/8을 안정적으로 닫을 수 있다.
