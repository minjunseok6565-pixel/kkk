# 트레이드 시장 비현실 동작 점검 브리핑 (2026-03-08)

## 결론 요약
- **슈퍼스타(예: 웸반야마급) 공개 매물에 제안이 적은 원인**은 단일 원인이라기보다, 고가 자산에 불리한 후보 랭킹/스켈레톤/검증 예산 구조가 중첩된 결과로 보입니다.

## 2) 슈퍼스타 매물에 오퍼가 적은 구조적 원인

### 2-1. BUY 타깃 탐색이 need-tag index 중심 + 좁은 스캔 폭 ✅
BUY 타깃은 `incoming_by_need_tag` 인덱스에서 팀 need 기반으로만 훑습니다.
태그별로 `need_n`을 채울 때 `scan_limit = need_n * 3` 상한을 두기 때문에,
슈퍼스타가 해당 태그 상위 구간에 걸리지 않으면 아예 후보 풀에 거의 진입하지 못할 수 있습니다.

- 변경 후 동작: BUY 탐색은 `incoming_by_need_tag` 하드 게이트 대신 `incoming_all_players`(리그 전체 incoming 풀) 기반으로 후보를 수집하고, `need_similarity`를 순위 가중치로 반영합니다. 또한 `scan_limit = need_n * 3`가 제거되어 후순위 후보도 cap 범위 내에서 검토됩니다.

### 2-2. 타깃 rank에서 연봉 페널티가 고연봉 스타에 불리 ✅
타깃 점수식에 `rank -= 0.015 * salary_m` 항이 있어, 고연봉 스타는 need 적합도가 높아도 랭크가 쉽게 깎입니다.
여기에 `incoming refs` 정렬 자체가 tag_strength 중심이므로, "절대 가치가 큰 선수"가 "해당 팀 need에 정확히 맞는 중급 자산"보다 뒤로 밀릴 여지가 큽니다.

- 변경 후 동작: BUY rank/pre-score에서 `salary_m` 직접 페널티를 제거하고, `contract_gap_cap_share(기대 cap share - 실제 cap share)` 기반 계약 점수로 대체했습니다.
- 변경 후 동작: 최종 점수는 `player_core(순수가치) + contract_term(계약가치×팀상황민감도) + listing_boost + jitter`로 계산되며, `contract_term`은 `tanh(gap/softness)` 포화 함수와 apron/posture/deadline multiplier를 함께 반영합니다.
- 변경 후 동작: 따라서 고연봉 자체는 감점 근거가 아니고, 기대 가치 대비 과/저평가 계약인지가 핵심 판정 기준으로 작동합니다.

### 2-3. 초기 오퍼 스켈레톤이 고가 자산 상향 매칭에 제한적 ✅
BUY 스켈레톤은 기본적으로
- picks_only (흡수 가능 시에만)
- young_plus_pick (선수 1 + 소수 픽)
- p4p_salary
- consolidate 2-for-1
위주이며,
딜 shape 상한(`max_assets_per_side`, `max_players_per_side`, `max_picks_per_side`)도 타이트합니다.
결과적으로 "스타 1명"에 필요한 대형 패키지 공간이 충분치 않을 수 있습니다.

- 변경 후 동작: BUY 스켈레톤 라우트가 `ROLE/STARTER/HIGH_STARTER(STAR)` tier별 allowlist로 확장되어,
  `player_swap.three_for_one_upgrade`, `player_swap.star_lateral_plus_delta`,
  `timeline.bluechip_plus_first_plus_swap` 같은 고가 자산용 패키지 조합이 기본 활성화됩니다.
- 변경 후 동작: tier 라우팅은 `STAR -> HIGH_STARTER`로 매핑되어 고가 타깃이 role 중심의 협소한 스켈레톤에 갇히지 않고,
  3-for-1/대형 가치 교환형 아키타입까지 초기 오퍼 단계에서 탐색합니다.

### 2-4. 평가/판정에서 고가 딜은 reject/counter로 잘 소거됨
스켈레톤으로 만든 안들이 평가 단계에서 양쪽 surplus threshold를 넘기기 어렵고,
결정 정책은 `required_surplus`, `overpay_budget`, `counter corridor` 기반이므로,
고가 딜은 소거되거나 counter 루프로 머무는 빈도가 높아질 수 있습니다.

### 2-5. 추가 지적 타당성 검토 (2026-03-08)
아래 3개 지적은 **코드 기준으로 타당**합니다.

- **딜 shape 상한이 고가치 대형딜 탐색을 조기 차단** ✅
  - 생성기 기본 설정이 `max_assets_per_side=9`, `max_players_per_side=4`, `max_players_moved_total=7`,
    `max_picks_per_side=4`로 완화되었고,
    `_shape_ok()`에서 이를 하드 컷으로 적용합니다.
  - 변경 후에는 슈퍼스타 딜에서 자주 필요한 3-for-1/4-for-2류 선수 패키지가
    shape 검증 이전에 일괄 탈락하는 비율이 유의미하게 감소합니다.

- **incoming 인덱싱/스캔 방식이 유니콘 노출을 줄일 가능성** ✅
  - incoming 태그 인덱스는 선수 `supply`의 상위 4개 태그만 보고, 그중 `>=0.55`만 유지합니다.
    멀티툴 선수처럼 강점이 넓게 분산된 케이스는 인덱스 진입 태그 수가 줄어들 수 있습니다.
  - BUY 타깃 스캔은 태그별 `scan_limit = need_n * 3`까지만 확인합니다.
    앞쪽 후보가 동일팀/밴/쿨다운으로 탈락하면 뒤쪽의 적합 후보를 보지 못할 수 있습니다.

  - 변경 후 동작: incoming 노출은 단일 need-tag 인덱스 의존을 줄이고, 전역 incoming 인덱스 + 팀/선수 스캔 cap(`teams_cap`, `players_cap`, `iteration_cap`) 구조로 바뀌어 유니콘/후순위 자산의 탐색 진입 가능성이 높아졌습니다.

- **trade block listing은 강제 노출이 아니라 soft rank boost** ⚠️ (부분 수정)
  - BUY 타깃 선택에서 public listing은 후보 포함의 필수 조건이 아니고,
    랭크에 `interest_boost`를 더하되 `buy_target_listing_interest_cap` 상한으로 제한됩니다.
  - 즉 listing은 “탐색 강제”가 아니라 “정렬 가중치” 성격이라, 다른 제약과 겹치면 체감 노출 효과가 약해질 수 있습니다.

  - 변경 후 동작: BUY 타깃 탐색이 전역 incoming 풀 기반으로 재구성되면서,
    active public listing 선수는 listed lane으로 분리 수집되어 merge 단계의 `listed_min_quota` 내에서 우선 진입합니다.
    다만 최종 타깃은 `listed_max_share`/팀·선수 cap/쿨다운·밴 등 공통 제약을 함께 적용하므로,
    listing은 “절대 강제 노출”이라기보다 “후보 진입 우선권 + 랭크 가중치”로 동작합니다.

### 2-6. 슈퍼스타가 트레이드 블록에 너무 쉽게 올라오는 구조적 원인 (신규)

아래는 “시즌 초반에 SGA/앤써니 에드워즈급이 public listing에 뜬다”는 관측과 맞물리는 **상향 노출 편향**입니다.

- **AI proactive listing이 매 tick·매 AI 팀에 자동 적용됨** ✅
  - tick 루프에서 각 AI actor가 제안 생성을 마칠 때마다 `apply_ai_proactive_listings(...)`가 호출됩니다.
  - 즉 "실제 시장 반응(오퍼 부족/요청)"이 없어도, 정책상 후보만 있으면 시즌 초부터 기계적으로 listing이 누적될 수 있습니다.

  - 변경 후 동작: proactive listing 평가 주기를 cadence로 분리해 `WEEKLY` 설정 시 앵커 요일 + 7일 간격으로만 평가되며, 팀별 `last_eval_at` 메타를 기준으로 재평가를 제한합니다.

- **proactive 후보 필터에 ‘슈퍼스타 보호’ 개념이 사실상 없음** ⚠️ (부분)
  - proactive 단계에서 제외되는 주요 조건은 `asset lock`, `recent signing ban`, `cooldown` 정도이며,
    `market.total` 상위권/팀 프랜차이즈 코어/올-NBA급 같은 보호 규칙은 없습니다.
  - 따라서 선수가 `SURPLUS_*`/`CONSOLIDATE` 버킷에만 들어오면 고가치 자산도 listing 후보로 열립니다.

- **버킷 분류가 ‘절대 가치’보다 ‘팀 적합도/중복도’ 중심이라 스타도 후보화 가능** ✅
  - `SURPLUS_LOW_FIT`는 팀-선수 fit 기준, `SURPLUS_REDUNDANT`는 공급-니즈 중복 기준으로 잡습니다.
  - 이 경로는 선수 절대 가치(리그 최상위 자산인지)로 컷하지 않기 때문에,
    "가치가 매우 높지만 현재 로스터 문맥에서 중복/저적합"으로 계산된 스타가 후보군에 섞일 수 있습니다.

  - 변경 후 동작: SELL outgoing 분류는 `SURPLUS_LOW_FIT`/`SURPLUS_REDUNDANT`를 제거하고 `SURPLUS_EXPENDABLE` 단일 버킷으로 재편됐습니다.
    `trade_block_score` 계산 시 `core_proxy`(팀 내 `market.now` 랭크 + 농구가치 기반), `identity_risk_proxy`, `dependence_risk` 보호 신호를 강하게 감점하며,
    `core_proxy >= 0.82` 또는 `identity_risk_proxy >= 0.78` 또는 `dependence_risk >= 0.68`이면 `hard_protected`로 강제 제외됩니다.
    즉, 단순 저적합/중복만으로는 코어 스타가 버킷에 진입하기 어렵고, “대체 가능성 + 보호 신호 미충족”을 함께 만족해야 후보화됩니다.

- **팀별 proactive 공급량이 시즌 초 과잉 노출을 만들기 쉬움** ✅
  - 기본값이 `team_daily_cap=2`, `team_active_cap=4`라서 AI 팀 수가 많으면 리그 전체 listing 공급이 빠르게 증가합니다.
  - SELL/SOFT_SELL 자세에서는 TTL도 길어(`12/7일`) 한 번 올라온 매물이 꽤 오래 남아,
    "블록에 스타가 자주 보인다"는 체감이 강화됩니다.

  - 변경 후 동작: 버킷 진입만으로 즉시 상장되지 않고 `surplus_score >= 팀상황/버킷별 threshold`를 통과한 선수만 proactive listing 대상이 됩니다. threshold는 posture/horizon/urgency/cooldown에 따라 조정되어 팀별 공급량이 더 보수적으로 제어됩니다.

- **cooldown(7일)이 짧아 ‘재상장 루프’를 충분히 막지 못함**
  - 선수 단위 proactive cooldown은 기본 7일로, 시즌 초 주간 단위로 재상장이 가능한 수준입니다.
  - 특히 팀 posture 변동이 잦거나 active slot이 비는 경우, 같은 급의 핵심 선수가 반복 노출될 여지가 큽니다.

정리하면, 현재는 **"슈퍼스타를 원천적으로 잘 안 내놓는 보호장치"보다 "버킷 충족 시 자동 상장" 메커니즘이 강한 구조**입니다.
그래서 고가치 선수가 생각보다 자주 블록에 보이고, 반대로 listing 이후 관심은 rank boost 수준에 머물러 폭발적으로 증폭되지 않는 비대칭이 발생합니다.


## 3) 지금 증상과 코드의 대응관계
- **"웸반야마급 매물인데 오퍼가 거의 안 들어온다"**
  - need-tag 중심 탐색 + 스캔 상한 + 연봉 패널티 + 제한된 스켈레톤 조합으로,
    초고가 자산이 후보/유효딜까지 도달하기 어렵다.

- **"SGA/에드워즈급이 시즌 초부터 블록에 자주 뜬다"**
  - AI proactive listing이 tick 단위로 자동 실행되고,
    후보 필터가 절대가치 기반 슈퍼스타 보호 없이 버킷/락/쿨다운 중심이라 고가치 자산이 상장 후보에 들어올 수 있다.
