# NBA 시뮬레이션 게임 메인 화면 UI 고도화 기획안 (상업 출시 품질)

## 0) 문서 목적/범위
- 목적: 현재 메인 화면(팀 로스터 + 선수 상세 진입 흐름)을 상업용 게임 수준의 프론트엔드 품질로 고도화하기 위한 실행 가능한 기획/설계 기준을 제공한다.
- 적용 범위: `static/NBA.html`, `static/NBA.css`, `static/NBA.js`.
- 비적용 범위: 백엔드 API 추가/수정, 타 화면(UI), DB 스키마 변경.
- 원칙: 기존 조회 API와 현재 응답 데이터만 활용하며, 필요한 추가 정보는 프론트엔드에서 파생 계산한다.

---

## 1) 제품 목표와 성공 기준

### 1-1. 제품 목표
1. "누가 봐도 완성도 높은 상업용 게임 UI" 인상을 제공한다.
2. 사용자 행동(선수 탐색 → 비교 → 상세 확인 → 의사결정)의 비용을 낮춘다.
3. 일관된 시각 시스템으로 "급히 만든 화면" 인상을 제거한다.

### 1-2. 성공 KPI (런칭 후 계측 기준)
- 로스터 화면에서 선수 상세 진입까지 평균 클릭 수: 기존 대비 20% 감소.
- 로스터 화면 체류 시간: 기존 대비 15% 증가(탐색성 상승).
- 상호작용 실패율(잘못된 클릭/스크롤 혼선): 기존 대비 30% 감소.
- 주관 평가(내부 QA): "상업 품질" 5점 척도 평균 4.3 이상.

## 2) 핵심 UX 컨셉

### 2-1. 콘셉트 문장
**"Front Office Command Center"**
- 데이터는 촘촘하지만, 시선 흐름은 단순하게.
- 화려함보다 정보 위계와 반응성으로 고급감을 만든다.

### 2-2. 경험 원칙
1. **정보 위계 우선**: 중요한 수치(OVR/컨디션/계약 리스크)를 전면 배치.
2. **즉시성**: Hover/선택/정렬 반응이 200ms 내 인지 가능해야 함.
3. **예측 가능성**: 색/아이콘/배지 의미가 화면 전체에서 불변.
4. **의사결정 중심**: 장식보다 비교 가능한 구조를 우선.

---

## 3) 정보 구조(IA) 재설계

## 3-1. 메인 화면 3영역 구조
1. **상단 글로벌 바**
   - 팀명, 시즌, 현재 뷰 이름, 뒤로/메인 액션.
2. **요약 인사이트 바**
   - 평균 나이, 평균 OVR, 총 연봉, 캡 여유, 부상/불만 카운트.
3. **핵심 작업 영역**
   - 좌: 로스터 테이블(탐색 중심)
   - 우: 선택 선수 퀵 패널(핵심 정보 + 상세 진입)

> 현재 상세 페이지 이동 흐름은 유지하되, 메인에서 의사결정 밀도를 높인다.

### 3-2. 사용자 플로우
1. 필터(포지션/상태) 적용
2. 정렬(OVR/연봉/컨디션)
3. 선수 행 선택
4. 우측 퀵 패널에서 핵심 검토
5. 상세 진입

## 4) UI 시스템 가이드 (상업 품질 기준)

### 4-1. 디자인 토큰
- Spacing: `4 / 8 / 12 / 16 / 24 / 32 / 40`
- Radius: `6 / 10 / 14`
- Shadow:
  - Level 1: `0 2px 10px rgba(15,23,42,0.08)`
  - Level 2: `0 8px 24px rgba(15,23,42,0.12)`
- Border: `1px solid rgba(15,23,42,0.08)`

### 4-2. 타이포그래피
- Heading XL(페이지 타이틀): 36/44, 700
- Heading L(섹션): 24/32, 700
- Body M(기본 본문): 16/24, 500
- Body S(테이블/보조): 14/20, 500
- Number (재무/지표): tabular-nums 적용

### 4-3. 컬러 체계
- Neutral 배경: cool gray 톤 2~3단계
- Team Primary: 팀 아이덴티티 포인트 1개
- Semantic:
  - Positive: 건강/양호
  - Warning: 주의/피로
  - Danger: 부상/불만/계약 리스크
- 금지: 동일 의미에 서로 다른 색 사용

### 4-4. 모션 규격
- Hover: 120~160ms
- Select/Expand: 180~220ms
- Easing: ease-out 기반 단일 통일
- 금지: 과한 bounce, 무거운 패럴랙스

## 5) 메인 화면 상세 기획

## 5-1. 상단 글로벌 바
구성 요소:
- 좌측: Team Avatar + 팀명 + 시즌 텍스트
- 중앙: 현재 화면 경로(Breadcrumb 형태)
- 우측: 메인/저장/설정 등 고정 액션

상태:
- 스크롤 시 상단 고정(sticky)
- 라이트 블러 배경으로 콘텐츠와 분리

### 5-2. 팀 인사이트 요약 바 (신규)
카드 5개 고정:
1. 평균 OVR
2. 평균 나이
3. 총 연봉 / 샐러리캡 여유
4. 부상/주의 선수 수
5. 불만/요청 선수 수

데이터 소스:
- 기존 로스터 조회 결과의 집계(프론트 계산)

### 5-3. 로스터 테이블 고도화
권장 컬럼:
1. 선수 (아바타/이름/포지션)
2. OVR
3. 역할(Starter/Rotation/Bench)
4. 컨디션 (수치 + 바)
5. 건강 (단기/장기 요약)
6. 계약 (연봉, 만료)
7. 리스크 배지 (부상, 불만, 고연봉 대비 저효율)

인터랙션:
- 행 Hover: 배경 + 우측 액션 아이콘 노출
- 행 Select: Team Primary 2px 인셋 강조
- 정렬: 컬럼 헤더 토글(오름/내림)
- 필터: 포지션, 상태, 나이대

표현 규칙:
- 퍼센트는 숫자 우선 + 색상 보조
- 연봉은 통화 포맷 고정
- 컬럼 정렬은 데이터 타입별 일관성 유지

### 5-4. 우측 선수 퀵 패널 (메인 전용)
구성:
- 선수 식별(이름, 포지션, 나이, 키/몸무게)
- 핵심 KPI 4칸(OVR, 경기력, 건강, 계약 효율)
- 강점 TOP3 / 약점 TOP2 (ATTR 파생)
- 상세 페이지 이동 CTA 버튼

효과:
- 전체 페이지 이동 전에 1차 판단 가능
- 상세 화면 의존도를 낮춰 탐색 피로 감소

### 5-5. 빈 상태/로딩/오류 화면
- Empty: "조건에 맞는 선수가 없습니다" + 필터 초기화 버튼
- Loading: 스켈레톤(행 8개)
- Error: 재시도 버튼 + 원인 문구(사용자 친화)

## 6) 기존 API만으로 구현 가능한 파생 지표 정의

> 아래 지표는 신규 API 없이 현재 조회값 조합으로 계산한다.

1. **Contract Efficiency Index**
   - 예시: `OVR / 연봉(백만 달러)` 정규화
2. **Durability Risk**
   - 예시: `나이 + 부상 관련 ATTR + 체력` 가중합
3. **Form Index**
   - 예시: 경기력%, 컨디션%, 출전 상태를 0~100 스케일로 통합
4. **Role Fit Score**
   - 포지션별 중요 ATTR 가중 평균
5. **Alert Flag**
   - 낮은 컨디션, 불만 존재, 저효율 고연봉 조건 충족 시 배지 부여

주의:
- 계산 로직은 화면 내 툴팁으로 간단 노출(블랙박스 인상 방지)

## 7) HTML/CSS/JS 구현 지침

### 7-1. HTML 구조 원칙
- 시맨틱 태그 사용(`header`, `main`, `section`, `aside`, `table`)
- 데이터 영역과 컨트롤 영역 명확 분리
- 접근성: 버튼/토글/정렬 헤더에 aria 속성 부여

### 7-2. CSS 아키텍처
- BEM 또는 컴포넌트 접두사 규칙 고정
- 색/간격/폰트는 CSS 변수로 중앙화
- 반응형 브레이크포인트 3단계
  - Desktop(≥1440)
  - Laptop(1024~1439)
  - Tablet(768~1023)

### 7-3. JS 상태 관리 원칙
- 단일 `viewModel` 객체로 필터/정렬/선택 상태 관리
- 렌더 함수 분리
  - `renderSummary()`
  - `renderRosterTable()`
  - `renderQuickPanel()`
- 이벤트 위임 기반으로 성능/유지보수성 확보

---

## 8) 품질 기준(Definition of Done)

### 8-1. 비주얼 품질
- 컴포넌트 간 간격 오차 ±2px 이내
- 텍스트 대비 WCAG AA 충족(핵심 텍스트)
- 아이콘/배지 크기 규칙 준수

### 8-2. 상호작용 품질
- 정렬/필터 반응시간 체감 200ms 내
- Hover/Select 상태 누락 0건
- 키보드 탐색(탭 이동) 가능

### 8-3. 데이터 품질
- 요약바 수치와 테이블 집계값 불일치 0건
- 포맷 규칙(연봉, % 등) 위반 0건

## 9) 구현 단계 로드맵 (메인 화면 한정)

### Phase 1: 구조/시스템 정비 (1~2일)
- 레이아웃 골격 재구성
- 디자인 토큰/CSS 변수 정의
- 공통 카드/배지/버튼 컴포넌트화

### Phase 2: 데이터 인터랙션 (2~3일)
- 요약 인사이트 바 구현
- 테이블 정렬/필터/선택 고도화
- 우측 퀵 패널 연결

### Phase 3: 폴리싱/QA (1~2일)
- 모션/상태 전수 점검
- 빈/오류/로딩 화면 적용
- 픽셀/접근성/성능 검수

## 10) "바이브 코딩 티" 제거 체크리스트

1. 동일 컴포넌트의 높이/패딩이 화면마다 달라지지 않는가?
2. 숫자 정렬(특히 연봉/퍼센트)이 한 줄로 깔끔히 맞는가?
3. 상태 표현이 색상에만 의존하지 않는가?
4. Hover/Active/Disabled 상태가 모두 정의되어 있는가?
5. 긴 데이터(이름/숫자)에서 줄바꿈/잘림 규칙이 일관적인가?
6. 로딩/빈 화면/에러 화면이 존재하는가?
7. 모바일/저해상도에서 정보 손실 없이 축약되는가?

## 11) 다음 작업자(프론트 구현) 전달 템플릿

- 작업 파일: `static/NBA.html`, `static/NBA.css`, `static/NBA.js`만 수정.
- 금지 사항: 신규 API 생성, 백엔드 로직 변경, 타 화면 수정.
- 필수 산출물:
  1. 메인 화면 리디자인 반영
  2. 정렬/필터/선택 인터랙션 구현
  3. 요약 인사이트 바 + 퀵 패널 구현
  4. 로딩/빈/에러 상태 구현
- 검수 기준: 본 문서 8장(Definition of Done) 100% 충족.
