# 트레이드 보수성 튜닝 가이드

이 문서는 `docs_trade_valuation_review.md`에서 언급한 항목을 **실제 구현 로직 기준**으로 풀어쓴 실무 튜닝 가이드입니다.

- 대상 독자: PM/기획/운영 + 개발자
- 목표: "어느 수치를 올리거나 내리면 제안량/성사율이 어떻게 바뀌는지"를 빠르게 판단

---

## 0) 전체 흐름 한눈에 보기

트레이드 판정은 대략 아래 순서로 계산됩니다.

1. `orchestration/listing_policy.py`에서 트레이드 블록 노출(프로액티브 listing) 필터
2. `decision_context.py`에서 팀 성향/상황을 knob로 변환
3. `team_utility.py`에서 선수 가치에 fit/risk/finance 배율 적용
4. `package_effects.py`에서 딜 묶음 페널티/보정 적용
5. `decision_policy.py`에서 `required_surplus`, `overpay`, `counter corridor`로 최종 판정
6. `dealgen/scoring.py`/`dealgen/sweetener.py`에서 너무 나쁜 후보 early discard + near-miss 복구
7. `orchestration/policy.py`의 office gate에서 최종 승격 전 안전 컷

즉, **한 군데만 보수적인 게 아니라 여러 레이어가 겹쳐서 보수화**될 수 있습니다.

## 1) 즉시 확인(P0): 의도-구현 불일치

## 1-1. **`min_surplus_required`** 음수 허용 의도 소실

### 실제 구현

- 컨텍스트 생성 단계(의도):

```python
# decision_context.py
min_surplus_required = lerp(-0.03, 0.10, eff_neg_tough)
if posture in ("SELL", "SOFT_SELL"):
    min_surplus_required += 0.03
elif posture == "AGGRESSIVE_BUY":
    min_surplus_required -= 0.02
```

- 판정 단계(현재 동작):

```python
# trades/valuation/decision_policy.py
min_surplus_ratio = max(0.0, knobs.min_surplus_required)
required_surplus = min_surplus_ratio * scale
```

즉, context에서 음수까지 열어 둔 값을 decision에서 다시 0 이상으로 잘라냅니다.

### 수치 조정 효과

- **음수 허용 유지(절단 제거/완화)**
  - 공격적 BUY 팀이 "약간 손해" 딜도 받아들일 여지가 생김
  - ACCEPT 증가, COUNTER/REJECT 감소 가능
- **현행 유지(`max(0.0, ...)`)**
  - 공격적 BUY라도 최소 본전 이상만 허용 → 체결률이 낮아질 수 있음

### NBA 팬 버전 비유

"윈나우 팀이 우승 퍼즐 마지막 조각을 위해 1라운드 픽을 조금 더 써도 오케이"라는 프런트 철학을 세워놨는데,
실제 결재 시스템에서 "무조건 본전 이상만"으로 막아둔 상태에 가깝습니다.

## 2) 의사결정 문턱 계열 튜닝

## 2-1. **`min_surplus_required`** (필요 이득률)

### 실제 구현

```python
# decision_context.py
min_surplus_required = lerp(-0.03, 0.10, eff_neg_tough)

# decision_policy.py
required_surplus = min_surplus_ratio * scale
scale = max(outgoing_total, 6.0)
```

`outgoing_total`이 큰 딜일수록 required_surplus 절대값이 커집니다.

### 수치 조정 효과

- **값을 낮추면**: ACCEPT 문턱 하락 → 큰 딜 포함 성사율 상승
- **값을 높이면**: "이득 크게 아니면 안 함" → 거래량 감소, COUNTER/REJECT 증가

### NBA 팬 버전 비유

같은 "+5점" 이득 기준도 소형 트레이드에서는 충분하지만,
초대형 딜에서는 "최소 +15점은 돼야" 같은 식으로 바뀌는 구조입니다.

---

## 2-2. **`overpay_budget`** (손해 허용폭)

### 실제 구현

```python
# decision_context.py
overpay_budget = 0.18 * eff_win_now * urgency * buy_factor * (1.0 - 0.45 * eff_neg_tough)
overpay_budget = clamp(overpay_budget, 0.0, 0.22)

# decision_policy.py
overpay_allowed = overpay_ratio * scale
overpay_floor = -overpay_allowed
```

### 수치 조정 효과

- **상향**: "조금 손해지만 필요해서 감수" 가능한 구간 확대
- **하향**: 손해 허용 거의 없음 → REJECT 증가

### NBA 팬 버전 비유

마감 직전 윈나우 팀이 "벤치 한 장 더 얹어도 당장 로스터 업그레이드"를 허용하느냐의 문제입니다.

## 2-3. **`counter_rate`** + **`counter_corridor_ratio`**

### 실제 구현

```python
# decision_context.py
counter_rate = lerp(0.35, 0.80, eff_neg_tough) * lerp(1.0, 0.70, urgency)

# decision_policy.py
counter_corridor_ratio = 0.06
in_corridor = (accept_threshold - corridor) <= net < accept_threshold

# deterministic mode
return counter_rate > 0.5
```

### 수치 조정 효과

- **counter_rate↑**: 회색지대에서 ACCEPT 대신 COUNTER 빈도 증가
- **counter_corridor_ratio↑**: COUNTER 검토 구간 자체 확대
- **stochastic_counter=True**: 매번 같은 결론 대신 확률적 반응

### NBA 팬 버전 비유

GM이 "바로 오케이"보다 "한 장만 더"를 더 자주 말하게 되는 설정입니다.

---

## 2-4. **`counter_overpay_fraction`**

### 실제 구현

```python
# decision_policy.py
if allow_counter and (
    in_corridor
    or (
        within_overpay and net < 0
        and abs(net) <= cfg.counter_overpay_fraction * overpay_allowed
    )
):
    return DealVerdict.COUNTER
```

즉, 순손해(`net < 0`)여도 `overpay_allowed`의 일정 비율 이내면 REJECT 대신 COUNTER를 허용합니다.

### 수치 조정 효과

- **상향**: 음수 surplus 구간에서도 재협상 여지가 넓어져 COUNTER 증가
- **하향**: "손해가 보이면 즉시 거절" 성향 강화

### NBA 팬 버전 비유

단장이 "지금 오퍼는 살짝 마이너스지만, 픽 보호조건만 바꾸면 성립 가능"이라고
테이블을 유지할지, 바로 전화를 끊을지를 정하는 다이얼입니다.

## 3) 선수 가치 페널티 계열 튜닝

## 3-1. Fit 이중 페널티

### 실제 구현

```python
# fit_engine.py
fit_factor_floor = 0.70
fit_below_threshold_floor = 0.35
fit_below_threshold_strength = 2.0

multiplier = fit_factor * threshold_penalty
```

fit score가 임계치 아래면 `FIT_FACTOR`에 더해 `FIT_BELOW_THRESHOLD_PENALTY`가 한 번 더 곱해집니다.

### 수치 조정 효과

- **threshold 완화 / strength 완화 / floor 상향**
  - 미스핏 선수의 가치 급락 완화
  - near-miss 딜이 살아남아 협상 지속 가능
- **반대로 강화**
  - 팀 색깔은 선명해지지만 거래량은 급감 가능

### NBA 팬 버전 비유

"우리 시스템에 100% 맞아야만 영입"인지,
"80% 맞아도 재능 있으면 받아서 코치가 맞춘다"인지의 차이입니다.

## 3-2. Risk discount

### 실제 구현

```python
# team_utility.py
risk_score = 0.65 * age_risk + 0.35 * term_risk
factor = 1.0 - scale * 0.35 * risk_score
factor = clamp(factor, 0.60, 1.00)
```

### 수치 조정 효과

- **`risk_discount_scale`** **상향**: 고령/장기계약 선수 더 싸게 평가
- **`risk_discount_scale`** **하향**: 베테랑 매물 거래가 쉬워짐
- **`risk_factor_floor`** **상향**: 최악 상황에서도 과도한 하락 방지

### NBA 팬 버전 비유

"34살 베테랑 + 3년 계약"을 얼마나 무서워하느냐(부상/노쇠 리스크 가격표) 조절입니다.

## 3-3. Finance penalty

### 실제 구현

```python
# decision_context.py
finance_penalty_scale = lerp(0.40, 1.60, eff_fin_cons)
if apron_status == "ABOVE_2ND_APRON":
    eff_fin_cons = max(eff_fin_cons, 0.75)

# team_utility.py
finance_score = (1 - finance_term_weight) * salary_burden + finance_term_weight * term
factor = 1.0 - scale * 0.45 * finance_score
factor = clamp(factor, 0.55, 1.00)
```

### 수치 조정 효과

- **`finance_penalty_scale`** **상향**: 고액/장기계약 매물 거래 급감
- **`finance_penalty_scale`** **하향**: 샐러리 정리형·대형 계약 교환 성사율 상승
- **`finance_factor_floor`** **상향**: 재정 이유로 가치가 과도하게 깎이는 현상 완화

### NBA 팬 버전 비유

"좋은 선수지만 연봉이 너무 비싸서 패스"를 얼마나 세게 적용할지 정하는 다이얼입니다.

## 4) 타임라인(현재/미래) 가중치 튜닝

## 4-1. **`w_now`** / **`w_future`**

### 실제 구현

```python
# decision_context.py
w_now = clamp01(0.15 + 0.75 * eff_win_now + 0.10 * urgency)
w_future = clamp01(1.0 - w_now)
```

한 축이 올라가면 다른 축이 자동 하락(상보 구조)합니다.

### 수치 조정 효과

- **`w_now`** 성향 강화: 즉시전력 가치↑, 픽/유망주 가치↓
- **`w_now`** 완화: 미래자산도 협상 테이블에 남음
- 양 축 하한 보강: 극단값에서 한쪽이 "사실상 0점" 되는 현상 완화

### NBA 팬 버전 비유

리빌딩 팀은 "내년보다 3년 뒤"를 보고,
우승권 팀은 "지금 당장 플옵 로테"를 보는데,
이 균형이 너무 극단이면 서로 대화가 안 됩니다.

## 5) 패키지(묶음) 페널티 튜닝

## 5-1. 핵심 노브

### 실제 구현(기본값)

```python
# package_effects.py
roster_excess_waste_rate = 0.85
hole_penalty_scale = 0.22
hole_penalty_cap_ratio = 0.18
cap_room_weight_base = 0.35
cap_room_abs_cap = 22.0
```

### 수치 조정 효과

- **`roster_excess_waste_rate`** **하향**: 2-for-1, 벤치 패키지 딜 숨통
- **`hole_penalty_scale`**/`cap_ratio` **하향**: 특정 포지션 공백 공포 완화
- **`cap_room_weight_base`**/**`cap_room_abs_cap`** **하향**: 캡스페이스 사용 페널티 완화

### NBA 팬 버전 비유

"선수 3명 받는다고 무조건 좋아지는 건 아님"(로테이션 과밀),
"센터 내보내면 리바운드 망함"(포지션 구멍),
"캡공간도 자산"(FA 총알)이라는 현실을 얼마나 강하게 반영할지 조절하는 파트입니다.

## 5-2. 구조 강도 노브(`exponent`/`ledger`) 

### 실제 구현

```python
# package_effects.py
p = (abs(delta) ** cfg.hole_penalty_exponent) * cfg.hole_penalty_scale
texture_penalties[axis] = (
    (abs(dv) ** cfg.hole_penalty_exponent)
    * cfg.hole_penalty_scale
    * cfg.hole_texture_parallel_weight
)

raw_fut = w_need * ledger_raw * cfg.cap_ledger_score_scale
```

`hole_penalty_exponent`는 포지션 결손 패널티의 "기울기"를, `cap_ledger_score_scale`은 미래 캡 레저 점수의 영향력을 키웁니다.

### 수치 조정 효과

- **`hole_penalty_exponent` 상향**: 결손이 커질수록 패널티가 비선형으로 급증
- **`hole_penalty_exponent` 하향**: 중간 수준 결손의 과벌점 완화
- **`cap_ledger_score_scale` 상향**: 미래 캡 유연성(ledger) 가중치 강화
- **`cap_ledger_score_scale` 하향**: 현재 전력 중심 의사결정으로 이동

### NBA 팬 버전 비유

"빅맨 한 명 빠진 구멍"을 단순 감점으로 볼지,
"플레이오프에서 바로 약점 공략당할 치명타"로 볼지를 정하는 감도 조절입니다.

## 6) 생성 단계(discard) 튜닝

## 6-1. 조기 폐기 임계치

### 실제 구현

```python
# dealgen/types.py
discard_if_overpay_below = -18.0
discard_if_any_margin_below = -22.0
discard_if_reject_margin_below = -14.0
discard_if_both_margins_below = -10.0

# dealgen/scoring.py
mb = buyer_net - buyer_required
ms = seller_net - seller_required
if mb < discard_if_overpay_below: discard
...
```

### 수치 조정 효과

- 절대값을 **키워 완화**(예: **-18 → -20**): 더 많은 near-miss 후보 유지
- 절대값을 **줄여 강화**(예: **-18 → -15**): 엉터리 오퍼 제거는 빨라지나 제안량 감소

### NBA 팬 버전 비유

"첫 전화에서 조건 조금 안 맞는 딜"을 바로 끊을지,
아니면 "2라픽 하나 더 얹으면 될 수도"라고 다음 통화를 할지 정하는 기준입니다.

## 6-2. Sweetener 복구 구간

### 실제 구현

```python
# dealgen/sweetener.py
scale = max(team_eval.outgoing_total, 6.0)
close = max(floor, ratio * scale)
if cap > 0.0:
    close = min(cap, close)
if base_max > 0.0:
    close = min(base_max, close)
```

그리고 `abs(margin) <= close`인 near-miss만 pick/swap sweetener로 복구 시도를 진행합니다.

### 수치 조정 효과

- **`sweetener_max_deficit`/`sweetener_close_cap` 상향**: "조금 부족" 범위 확대 → 복구 시도 증가
- **`sweetener_close_corridor_ratio` 상향**: 딜 스케일이 큰 건에서 복구 창 확대
- **하향**: 복구 시도 감소, 초기 판정 결과가 그대로 확정될 확률 증가

### NBA 팬 버전 비유

원안이 살짝 부족할 때 "2라운드픽 + 스왑권 하나"로 타협점을 찾는 GM 협상력을
어디까지 허용할지 정하는 옵션입니다.

## 6-3. 타깃/상대 다양성 제약

### 실제 구현

```python
# dealgen/core.py
return replace(prop, score=float(prop.score) - pen * float(c))

# final output
if c >= cap_i:
    continue
```

`target_repeat_penalty`는 같은 타깃 반복 제안을 soft penalty로 낮추고,
`max_partner_repeats`는 같은 상대팀 반복을 hard cap으로 차단합니다.

### 수치 조정 효과

- **`target_repeat_penalty` 상향**: 다양성↑, 상위안이 분산됨
- **`max_partner_repeats` 활성/하향**: 특정 거래 상대 쏠림 방지
- 반대로 완화하면 성사 가능성이 높은 상대에 제안이 집중되어 단기 성사율은 오를 수 있음

### NBA 팬 버전 비유

마감일에 "한 팀하고만 10통화"할지,
"리그 전체에 동시에 찔러보며 시장가를 확인"할지를 정하는 운영 철학입니다.

## 6-4. BUY 타깃 검색량(스캔 캡)

### 실제 구현

```python
# dealgen/utils.py
intensity = max(0.0, min(1.0, 0.65 * d + 0.35 * u))
shaped = _smoothstep01(intensity)

teams_cap = base_teams + round(deadline_bonus * shaped)
players_cap = base_players + round(deadline_bonus_players * shaped)
non_listed_quota = base_non_listed + round(non_listed_bonus * intensity)
```

`buy_target_listed_min_quota`, `buy_target_non_listed_base_quota`,
`buy_target_max_teams_scanned_base`, `buy_target_max_players_scanned_base`,
`buy_target_retrieval_iteration_cap`이 후보풀 크기를 직접 좌우합니다.

### 수치 조정 효과

- **quota/scan cap 상향**: 제안 재료 증가(탐색비용·시간 증가)
- **하향**: 계산은 가벼워지지만 near-fit 타깃 누락 가능성 증가

### NBA 팬 버전 비유

스카우트팀이 "관심 선수 10명만 보고 끝"낼지,
"30명까지 넓게 훑고 마지막에 추릴지"를 정하는 리소스 배분입니다.

## 7) 트레이드 블록 노출량(입구) 튜닝

## 7-1. 프로액티브 listing threshold

### 실제 구현

```python
# orchestration/listing_policy.py
base = bucket_threshold_or_default
if horizon == "WIN_NOW" and bucket == "SURPLUS_EXPENDABLE":
    base += ai_proactive_listing_threshold_horizon_win_now_delta
if urgency >= ai_proactive_listing_threshold_urgency_cut:
    base += ai_proactive_listing_threshold_urgency_delta
if cooldown_active:
    base += ai_proactive_listing_threshold_cooldown_active_delta

return clamp(base, ai_proactive_listing_threshold_min, ai_proactive_listing_threshold_max)
```

### 수치 조정 효과

- **기본 threshold 하향**: 블록 노출 선수 증가 → 초기 제안 풀 확대
- **`cooldown_active_delta` 하향**: 냉각기 중에도 재노출 기회 회복
- **urgency/horizon delta 절대값 상향(음수 방향)**: 상황형 완화 강화

### NBA 팬 버전 비유

"아직 시장에 안 내놓은 선수"를 내부적으로 더 빨리 블록에 올릴지,
끝까지 비공개 자산으로 묶어둘지를 정하는 프런트 공개 전략입니다.

## 7-2. 팀/일일 listing cap + 선수 cooldown

### 실제 구현

```python
# orchestration/listing_policy.py
remaining = min(
    max(0, team_active_cap - len(active_pids)),
    max(0, team_daily_cap - already_today),
)
if remaining <= 0:
    return []
```

여기에 `ai_proactive_listing_player_cooldown_days`가 개별 선수 재등록 간격을 제한합니다.

### 수치 조정 효과

- **team_daily/active cap 상향**: 팀당 공개 매물 증가
- **player_cooldown_days 하향**: 동일 선수 재노출 속도 증가

### NBA 팬 버전 비유

단장이 하루에 시장에 내놓는 카드 수(전화 돌리는 횟수)를 늘릴지,
한 번 거절당한 선수를 며칠 뒤 다시 꺼낼지를 정하는 운영 리듬입니다.

## 8) 오케스트레이션 최종 게이트(office gate)

### 실제 구현

```python
# orchestration/policy.py
if score < config.ai_ai_office_gate_min_score:
    return False

margin = net_surplus - required_surplus
if margin < -(overpay_allowed + slack):
    return False
if margin < config.ai_ai_office_gate_min_margin:
    return False
```

### 수치 조정 효과

- **`min_score` 상향**: 최종 veto 증가(품질 보수화)
- **`min_margin`을 덜 음수로 상향**: 적자 딜 컷 강화
- **`overpay_slack` 상향**: decision/eval 불일치 허용폭 확대 → veto 감소

### NBA 팬 버전 비유

현업 단장끼리 합의한 딜이라도,
리그 사무국이 "이건 내부 기준상 너무 무리"라며 마지막 도장 전에 돌려보내는 안전장치입니다.

## 9) 운영 추천: 튜닝 순서

1. **P0**: **`min_surplus_required`** 의도-구현 불일치 먼저 정리
2. **P1**: 입구(프로액티브 listing threshold/cap/cooldown) 완화
3. **P2**: decision 문턱(**`min_surplus_required`**, **`overpay_budget`**, **`counter_rate`**, **`counter_overpay_fraction`**) 완화
4. **P3**: discard/sweetener/buy retrieval로 near-miss 복구량 확대
5. **P4**: package/office gate를 마지막 미세조정

한 번에 다 바꾸기보다, **10~20%**씩 단계 조정 + 모니터링이 안전합니다.

---

## 10) 모니터링 체크리스트

- 팀당 제안 생성 수
- ACCEPT / COUNTER / REJECT 비율
- `net_surplus - required_surplus` 분포 (0 근처 적체 여부)
- REJECT reason 상위 코드 (`FIT_FAILS`, `INSUFFICIENT_SURPLUS`, `PACKAGE_EFFECTS`)
- posture(SELL/BUY)별 성사율
- office gate veto 사유(`SCORE_TOO_LOW`, `MARGIN_TOO_NEGATIVE`) 비중

**해석 팁**
- 0 근처 음수에 대량 몰림: 문턱/튜닝 문제일 가능성 높음
- 큰 음수에 대량 몰림: 자산평가 자체(시장가/패키지) 재점검 필요
