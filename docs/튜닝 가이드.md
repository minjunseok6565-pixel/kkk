# 트레이드 보수성 튜닝 가이드

이 문서는 `docs_trade_valuation_review.md`에서 언급한 항목을 **실제 구현 로직 기준**으로 풀어쓴 실무 튜닝 가이드입니다.

- 대상 독자: PM/기획/운영 + 개발자
- 목표: "어느 수치를 올리거나 내리면 제안량/성사율이 어떻게 바뀌는지"를 빠르게 판단

---

## 0) 전체 흐름 한눈에 보기

트레이드 판정은 대략 아래 순서로 계산됩니다.

1. `orchestration/listing_policy.py`에서 트레이드 블록 노출(프로액티브 listing) 필터
2. `decision_context.py`에서 팀 성향/상황을 knob로 변환
3. `team_utility.py`에서 선수 가치에 fit/risk/finance 배율 적용
4. `package_effects.py`에서 딜 묶음 페널티/보정 적용
5. `decision_policy.py`에서 `required_surplus`, `overpay`, `counter corridor`로 최종 판정
6. `dealgen/scoring.py`/`dealgen/sweetener.py`에서 너무 나쁜 후보 early discard + near-miss 복구
7. `orchestration/policy.py`의 office gate에서 최종 승격 전 안전 컷

즉, **한 군데만 보수적인 게 아니라 여러 레이어가 겹쳐서 보수화**될 수 있습니다.

## 1) 즉시 확인(P0): 의도-구현 불일치

## 1-1. **`min_surplus_required`** 음수 허용 의도 소실

### 실제 구현

- 컨텍스트 생성 단계(의도):

```python
# decision_context.py
min_surplus_required = lerp(-0.03, 0.10, eff_neg_tough)
if posture in ("SELL", "SOFT_SELL"):
    min_surplus_required += 0.03
elif posture == "AGGRESSIVE_BUY":
    min_surplus_required -= 0.02
```

- 판정 단계(현재 동작):

```python
# trades/valuation/decision_policy.py
min_surplus_ratio = max(0.0, knobs.min_surplus_required)
required_surplus = min_surplus_ratio * scale
```

즉, context에서 음수까지 열어 둔 값을 decision에서 다시 0 이상으로 잘라냅니다.

### 수치 조정 효과

- **음수 허용 유지(절단 제거/완화)**
  - 공격적 BUY 팀이 "약간 손해" 딜도 받아들일 여지가 생김
  - ACCEPT 증가, COUNTER/REJECT 감소 가능
- **현행 유지(`max(0.0, ...)`)**
  - 공격적 BUY라도 최소 본전 이상만 허용 → 체결률이 낮아질 수 있음

### NBA 팬 버전 비유

"윈나우 팀이 우승 퍼즐 마지막 조각을 위해 1라운드 픽을 조금 더 써도 오케이"라는 프런트 철학을 세워놨는데,
실제 결재 시스템에서 "무조건 본전 이상만"으로 막아둔 상태에 가깝습니다.

### 열람/수정 파일(정확 경로)

## 2-1. **`min_surplus_required`** (필요 이득률)

### 실제 구현

```python
# decision_context.py
min_surplus_required = lerp(-0.03, 0.10, eff_neg_tough)

# decision_policy.py
required_surplus = min_surplus_ratio * scale
scale = max(outgoing_total, 6.0)
```

`outgoing_total`이 큰 딜일수록 required_surplus 절대값이 커집니다.

### 수치 조정 효과

- **튜닝 가능한 숫자 ① `lerp(-0.03, 0.10, eff_neg_tough)`의 하한 `-0.03`**
  - `-0.03 → -0.05`: 공격적 BUY 저점이 더 열려, 소규모 딜(`scale=6`) 기준 `required_surplus`가 `-0.18 → -0.30`까지 허용 가능(단, 현재 `max(0.0, ...)` 유지 시 체감 제한).
  - `-0.03 → -0.01`: "손해 감수" 폭 축소, 공격적 BUY도 사실상 본전 위주로 수렴.
- **튜닝 가능한 숫자 ② `lerp(-0.03, 0.10, eff_neg_tough)`의 상한 `0.10`**
  - `0.10 → 0.12`: tough 협상팀의 요구 이득률 강화. 예: `scale=20`에서 필요 surplus `2.0 → 2.4`.
  - `0.10 → 0.07`: tough 팀도 완화되어 대형 딜의 ACCEPT/Counter 전환 증가.
- **튜닝 가능한 숫자 ③ `scale = max(outgoing_total, 6.0)`의 바닥 `6.0`**
  - `6.0 → 8.0`: 소형 딜에도 최소 문턱을 더 크게 부과(미니멈 required_surplus 상승) → 잡딜 감소.
  - `6.0 → 4.0`: 소형 딜 문턱 완화 → 로테이션/벤치 급 딜 성사율 상승.
- **튜닝 가능한 숫자 ④ posture 보정치 `+0.03`(SELL), `-0.02`(AGGRESSIVE_BUY)**
  - `+0.03` 상향: SELL 팀은 "프리미엄 없으면 안 판다"가 더 강해짐.
  - `-0.02` 절대값 상향(예: `-0.04`): AGGRESSIVE_BUY가 더 공격적으로 수용.

### NBA 팬 버전 비유

같은 "+5점" 이득 기준도 소형 트레이드에서는 충분하지만,
초대형 딜에서는 "최소 +15점은 돼야" 같은 식으로 바뀌는 구조입니다.

## 2-2. **`overpay_budget`** (손해 허용폭)

### 실제 구현

```python
# decision_context.py
overpay_budget = 0.18 * eff_win_now * urgency * buy_factor * (1.0 - 0.45 * eff_neg_tough)
overpay_budget = clamp(overpay_budget, 0.0, 0.22)

# decision_policy.py
overpay_allowed = overpay_ratio * scale
overpay_floor = -overpay_allowed
```

### 수치 조정 효과

- **튜닝 가능한 숫자 ① 기본 계수 `0.18`**
  - `0.18 → 0.24`: urgency·win-now가 높을 때 overpay 예산이 전반적으로 확대.
  - `0.18 → 0.12`: 전체 팀군의 손해 허용 기조가 즉시 보수화.
- **튜닝 가능한 숫자 ② toughness 감쇠 `0.45`**
  - `0.45 → 0.60`: tough 팀에서 overpay_budget이 더 빠르게 축소(강경 협상 강화).
  - `0.45 → 0.25`: tough 성향이어도 윈나우 급한 상황이면 손해 감수 여지 유지.
- **튜닝 가능한 숫자 ③ clamp 상한 `0.22`**
  - `0.22 → 0.30`: 극단 urgency에서 허용 적자 폭 상단 개방.
  - `0.22 → 0.15`: 아무리 급해도 큰 적자는 못 보게 하드 캡 강화.

### NBA 팬 버전 비유

마감 직전 윈나우 팀이 "벤치 한 장 더 얹어도 당장 로스터 업그레이드"를 허용하느냐의 문제입니다.

## 2-3. **`counter_rate`** + **`counter_corridor_ratio`**

### 실제 구현

```python
# decision_context.py
counter_rate = lerp(0.35, 0.80, eff_neg_tough) * lerp(1.0, 0.70, urgency)

# decision_policy.py
counter_corridor_ratio = 0.06
in_corridor = (accept_threshold - corridor) <= net < accept_threshold

# deterministic mode
return counter_rate > 0.5
```

### 수치 조정 효과

- **튜닝 가능한 숫자 ① `counter_rate` lerp 하한/상한 `0.35 ~ 0.80`**
  - 하한 `0.35 → 0.45`: 온건 팀도 기본 COUNTER 성향 증가.
  - 상한 `0.80 → 0.65`: 강경 팀의 과도한 COUNTER 연쇄를 완화.
- **튜닝 가능한 숫자 ② urgency 감쇠 `lerp(1.0, 0.70, urgency)`의 `0.70`**
  - `0.70 → 0.50`: 급할수록 COUNTER보다 ACCEPT/REJECT로 빨리 결론.
  - `0.70 → 0.85`: 급해도 COUNTER 관성 유지.
- **튜닝 가능한 숫자 ③ clamp `0.10 ~ 0.95`**
  - 최소 `0.10` 상향 시 COUNTER 거의 안 나오는 팀 감소.
  - 최대 `0.95` 하향 시 "거의 무조건 COUNTER" 팀 억제.
- **튜닝 가능한 숫자 ④ `counter_corridor_ratio=0.06`**
  - `0.06 → 0.10`: accept threshold 직하단 10% 폭까지 COUNTER 검토.
  - `0.06 → 0.03`: 애매한 딜도 빠르게 ACCEPT/REJECT로 양분.
- **결정모드 스위치: deterministic에서 `counter_rate > 0.5` 컷**
  - 컷 `0.5 → 0.6`: COUNTER 발동이 줄어 보다 단호한 응답.
  - 컷 `0.5 → 0.4`: COUNTER 발동 증가.

### NBA 팬 버전 비유

GM이 "바로 오케이"보다 "한 장만 더"를 더 자주 말하게 되는 설정입니다.

## 2-4. **`counter_overpay_fraction`**

### 실제 구현

```python
# decision_policy.py
if allow_counter and (
    in_corridor
    or (
        within_overpay and net < 0
        and abs(net) <= cfg.counter_overpay_fraction * overpay_allowed
    )
):
    return DealVerdict.COUNTER
```

즉, 순손해(`net < 0`)여도 `overpay_allowed`의 일정 비율 이내면 REJECT 대신 COUNTER를 허용합니다.

### 수치 조정 효과

- **튜닝 가능한 숫자: `counter_overpay_fraction` (기본적으로 `overpay_allowed` 대비 비율)**
  - `0.25 → 0.40`: 허용 손해의 40% 구간까지는 REJECT 대신 COUNTER 유도.
  - `0.25 → 0.10`: 아주 얕은 음수만 재협상, 대부분 즉시 REJECT.
- **연동 포인트: `overpay_allowed` 자체가 커지면 동일 fraction도 절대 허용폭 확대**
  - 예: `overpay_allowed=8`, fraction `0.25`면 `abs(net) <= 2`까지 COUNTER.
  - 예: `overpay_allowed=12`, fraction `0.25`면 `abs(net) <= 3`까지 COUNTER.

### NBA 팬 버전 비유

단장이 "지금 오퍼는 살짝 마이너스지만, 픽 보호조건만 바꾸면 성립 가능"이라고
테이블을 유지할지, 바로 전화를 끊을지를 정하는 다이얼입니다.

## 3-1. Fit 이중 페널티

### 실제 구현

```python
# fit_engine.py
fit_factor_floor = 0.90
fit_below_threshold_floor = 0.60
fit_below_threshold_strength = 1.1

multiplier = fit_factor * threshold_penalty
```

fit score가 임계치 아래면 `FIT_FACTOR`에 더해 `FIT_BELOW_THRESHOLD_PENALTY`가 한 번 더 곱해집니다.

### 수치 조정 효과

- **튜닝 가능한 숫자 ① `min_fit_threshold = lerp(0.45, 0.70, eff_fit_strict)` (`decision_context.py`)**
  - `0.70 → 0.75`: 엄격 팀의 임계치가 더 높아져, `fit_score` 0.70~0.75 구간 선수도 추가 패널티 대상이 됨(미스핏 컷 강화).
  - `0.70 → 0.60`: 엄격 팀도 임계치 하향으로 near-miss 선수 재협상 여지 증가.
  - `0.45 → 0.35`: 느슨한 팀은 사실상 대부분 선수에 threshold penalty 미적용.
- **튜닝 가능한 숫자 ② `fit_scale = lerp(0.60, 1.80, eff_fit_strict)` (`decision_context.py`)**
  - `1.80 → 2.20`: fit 점수 차이가 가치 배수에 더 크게 반영되어, 맞는 선수/안 맞는 선수의 가격 격차 확대.
  - `1.80 → 1.20`: fit 영향 완화로 순수 탤런트/자산 가치가 상대적으로 더 중요해짐.
  - `0.60 → 0.40`: 느슨한 팀은 fit에 거의 둔감해져 거래 폭 확대.
- **튜닝 가능한 숫자 ③ `fit_below_threshold_strength` (`fit_engine.py`, 기본 `1.1`)**
  - `1.1 → 1.6`: threshold 미달 시 할인 기울기 급상승(낮은 fit 점수에서 가치 급락).
  - `1.1 → 0.7`: 임계치 아래라도 완만하게 할인되어 REJECT 대신 COUNTER로 남는 케이스 증가.
- **튜닝 가능한 숫자 ④ `fit_below_threshold_floor` (`fit_engine.py`, 기본 `0.60`)**
  - `0.60 → 0.45`: 미스핏 하한을 더 낮춰 "안 맞으면 크게 깎는다"를 강화.
  - `0.60 → 0.75`: 최악 미스핏도 75% 가치까지 보존해 과도한 붕괴 방지.
- **튜닝 가능한 숫자 ⑤ `fit_factor_floor/cap` (`fit_engine.py`, 기본 `0.90/1.35`)**
  - floor `0.90 → 0.80`: 저적합 선수의 기본 할인 폭 확대.
  - cap `1.35 → 1.50`: 고적합 선수 프리미엄 확대(핏 좋은 롤플레이어 가격 상승).

### NBA 팬 버전 비유

"우리 시스템에 100% 맞아야만 영입"인지,
"80% 맞아도 재능 있으면 받아서 코치가 맞춘다"인지의 차이입니다.

## 3-2. Risk discount

### 실제 구현

```python
# team_utility.py
risk_score = 0.65 * age_risk + 0.35 * term_risk
factor = 1.0 - scale * 0.35 * risk_score
factor = clamp(factor, 0.60, 1.00)
```

### 수치 조정 효과

- **튜닝 가능한 숫자 ① `risk_discount_scale = lerp(0.35, 0.08, eff_risk_tol)` (`decision_context.py`)**
  - 상단 `0.35 → 0.45`: 리스크 회피 팀에서 고령/장기계약 할인 강화(베테랑 수요 감소).
  - 상단 `0.35 → 0.25`: 리스크 회피 팀도 할인 완화.
  - 하단 `0.08 → 0.15`: 리스크 선호 팀조차 일정 수준 할인 유지(공격적 매수 억제).
- **튜닝 가능한 숫자 ② 연령/계약 가중치 `0.65/0.35` (`team_utility.py`)**
  - `0.65/0.35 → 0.80/0.20`: 나이 리스크 중심 평가(단기 고액 계약엔 상대적으로 관대).
  - `0.65/0.35 → 0.50/0.50`: 장기계약 리스크 반영 확대(젊어도 긴 계약이면 감점).
- **튜닝 가능한 숫자 ③ 계수 `factor = 1 - scale * 0.35 * risk_score`의 `0.35`**
  - `0.35 → 0.50`: 동일 risk_score에서도 할인폭 확대.
  - `0.35 → 0.20`: 리스크 할인 민감도 완화.
- **튜닝 가능한 숫자 ④ clamp 하한 `risk_factor_floor=0.60`**
  - `0.60 → 0.70`: 어떤 리스크 조합에서도 가치 70% 이상 보존.
  - `0.60 → 0.50`: 극단 리스크 자산을 더 강하게 디스카운트.

### NBA 팬 버전 비유

"34살 베테랑 + 3년 계약"을 얼마나 무서워하느냐(부상/노쇠 리스크 가격표) 조절입니다.

## 3-3. Finance penalty

### 실제 구현

```python
# decision_context.py
finance_penalty_scale = lerp(0.40, 1.60, eff_fin_cons)
if apron_status == "ABOVE_2ND_APRON":
    eff_fin_cons = max(eff_fin_cons, 0.75)

# team_utility.py
finance_score = (1 - finance_term_weight) * salary_burden + finance_term_weight * term
factor = 1.0 - scale * 0.45 * finance_score
factor = clamp(factor, 0.55, 1.00)
```

### 수치 조정 효과

- **튜닝 가능한 숫자 ① `finance_penalty_scale = lerp(0.40, 1.60, eff_fin_cons)` (`decision_context.py`)**
  - 상단 `1.60 → 2.00`: 재정 보수 팀의 고액 계약 기피 급강화(샐러리 덤프 성사 난도 상승).
  - 상단 `1.60 → 1.20`: 재정 보수 팀도 거래 가능 폭 확대.
  - 하단 `0.40 → 0.20`: 재정 완화 팀은 캡 부담을 거의 무시하고 전력 업사이드 우선.
- **튜닝 가능한 숫자 ② 2nd apron 보정 `eff_fin_cons = max(eff_fin_cons, 0.75)`**
  - `0.75 → 0.90`: 2nd apron 팀은 사실상 항상 강한 재정 페널티 적용.
  - `0.75 → 0.60`: apron 위 팀도 상황 따라 유연한 딜 허용.
- **튜닝 가능한 숫자 ③ `finance_term_weight=0.35` (`team_utility.py`)**
  - `0.35 → 0.55`: 계약 "길이" 비중 확대(연봉이 낮아도 긴 계약 감점).
  - `0.35 → 0.20`: 당장 연봉 부담 중심으로 평가(기간 감점 완화).
- **튜닝 가능한 숫자 ④ 계수 `factor = 1 - scale * 0.45 * finance_score`의 `0.45`**
  - `0.45 → 0.60`: 동일 재정 점수에서도 할인폭 확대.
  - `0.45 → 0.30`: 재정 페널티 완화.
- **튜닝 가능한 숫자 ⑤ clamp 하한 `finance_factor_floor=0.55`**
  - `0.55 → 0.70`: 재정 사유로 가치가 반토막 나는 현상 억제.
  - `0.55 → 0.45`: 재정 리스크가 큰 자산을 더 공격적으로 할인.

### NBA 팬 버전 비유

"좋은 선수지만 연봉이 너무 비싸서 패스"를 얼마나 세게 적용할지 정하는 다이얼입니다.

## 4) 타임라인(현재/미래) 가중치 튜닝

### 열람/수정 파일(정확 경로)

- `decision_context.py`


## 4-1. **`w_now`** / **`w_future`**

### 실제 구현

```python
# decision_context.py
w_now = clamp01(0.15 + 0.75 * eff_win_now + 0.10 * urgency)
w_future = clamp01(1.0 - w_now)
```

한 축이 올라가면 다른 축이 자동 하락(상보 구조)합니다.

### 수치 조정 효과

- **튜닝 가능한 숫자 ① 베이스/계수 `0.15 + 0.75*eff_win_now + 0.10*urgency`**
  - `0.75 → 0.90`: win-now 성향이 결과에 더 직접 반영되어 현재가치 편향 강화.
  - `0.75 → 0.60`: 팀 성향 차이를 완화해 중도적 가중치로 수렴.
  - `0.10 → 0.20`: 데드라인 임박(urgency↑) 시 현재가치 쏠림 급증.
- **튜닝 가능한 숫자 ② 하한 베이스 `0.15`**
  - `0.15 → 0.25`: 리빌딩 팀도 현재 기여를 최소 25% 반영.
  - `0.15 → 0.05`: 미래 편향 팀의 장기자산 선호 극대화.
- **튜닝 가능한 구조 ③ `w_future = 1.0 - w_now` 상보식**
  - 상보식 유지: 한쪽 강화 시 다른쪽 자동 약화(명확하지만 극단 발생 가능).
  - 별도 floor 도입(예: 두 축 모두 최소 0.20): 극단적 0점화 방지.

### NBA 팬 버전 비유

리빌딩 팀은 "내년보다 3년 뒤"를 보고,
우승권 팀은 "지금 당장 플옵 로테"를 보는데,
이 균형이 너무 극단이면 서로 대화가 안 됩니다.

## 5-2. 구조 강도 노브(`exponent`/`ledger`)

### NBA 팬 버전 비유

"빅맨 한 명 빠진 구멍"을 단순 감점으로 볼지,

아니면 "2라픽 하나 더 얹으면 될 수도"라고 다음 통화를 할지 정하는 기준입니다.

## 6-2. Sweetener 복구 구간

### 실제 구현

```python
# dealgen/sweetener.py
scale = max(team_eval.outgoing_total, 6.0)
close = max(floor, ratio * scale)
if cap > 0.0:
    close = min(cap, close)
if base_max > 0.0:
    close = min(base_max, close)
```

그리고 `abs(margin) <= close`인 near-miss만 pick/swap sweetener로 복구 시도를 진행합니다.

### 수치 조정 효과

- **`scale = max(outgoing_total, 6.0)`의 바닥 `6.0`**
  - `6.0 → 8.0`: 소형 딜도 sweetener 복구 판정 범위가 넓어짐.
  - `6.0 → 4.0`: 소형 딜은 더 엄격하게 원안 품질을 요구.
- **`sweetener_close_corridor_ratio`**
  - `0.20 → 0.30`: 대형 딜일수록 복구 허용폭 크게 증가.
  - `0.20 → 0.10`: ratio 기반 복구창 축소.
- **`sweetener_close_floor` / `sweetener_close_cap` / `sweetener_max_deficit`**
  - floor 상향: 저스케일 딜의 최소 복구창 보장.
  - cap·max_deficit 상향: 고스케일 딜의 복구 상한 확대.
  - cap·max_deficit 하향: 무리한 sweetener 루프 억제.

### NBA 팬 버전 비유

원안이 살짝 부족할 때 "2라운드픽 + 스왑권 하나"로 타협점을 찾는 GM 협상력을
어디까지 허용할지 정하는 옵션입니다.

## 7-1. 프로액티브 listing threshold

### 실제 구현

```python
# orchestration/listing_policy.py

if horizon == "WIN_NOW" and bucket == "SURPLUS_EXPENDABLE":
    base += ai_proactive_listing_threshold_horizon_win_now_delta
```

- **`ai_proactive_listing_threshold_cooldown_active_delta`**
  - 양수 확대: 최근 노출 선수 재등장 억제.
  - 0 또는 음수: cooldown 중 재노출 허용 확대.

"아직 시장에 안 내놓은 선수"를 내부적으로 더 빨리 블록에 올릴지,

## 8) 오케스트레이션 최종 게이트(office gate)

## 9) 운영 추천: 튜닝 순서

1. **P0**: **`min_surplus_required`** 의도-구현 불일치 먼저 정리
2. **P1**: 입구(프로액티브 listing threshold/cap/cooldown) 완화
3. **P2**: decision 문턱(**`min_surplus_required`**, **`overpay_budget`**, **`counter_rate`**, **`counter_overpay_fraction`**) 완화
4. **P3**: discard/sweetener/buy retrieval로 near-miss 복구량 확대
5. **P4**: package/office gate를 마지막 미세조정

한 번에 다 바꾸기보다, **10~20%**씩 단계 조정 + 모니터링이 안전합니다.

- 팀당 제안 생성 수
- ACCEPT / COUNTER / REJECT 비율
- `net_surplus - required_surplus` 분포 (0 근처 적체 여부)
- REJECT reason 상위 코드 (`FIT_FAILS`, `INSUFFICIENT_SURPLUS`, `PACKAGE_EFFECTS`)
- posture(SELL/BUY)별 성사율
- office gate veto 사유(`SCORE_TOO_LOW`, `MARGIN_TOO_NEGATIVE`) 비중

**해석 팁**
- 0 근처 음수에 대량 몰림: 문턱/튜닝 문제일 가능성 높음
- 큰 음수에 대량 몰림: 자산평가 자체(시장가/패키지) 재점검 필요
