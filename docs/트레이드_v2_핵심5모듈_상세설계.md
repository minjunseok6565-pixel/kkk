## 0) 문서 목적

본 문서는 아래 5개 파일을 **현재 코드베이스의 실존 SSOT 입력만으로** 구현 가능한 수준으로 상세 설계한다.

- `trades/valuation/context_v2.py`
- `trades/valuation/pick_distribution.py`
- `trades/valuation/contract_texture.py`
- `trades/valuation/role_texture.py`
- `trades/valuation/cap_ledger.py`

설계 목표:
1. 무가정(없는 값 생성 금지)
2. 중복금지(단일 계산 책임 강제)
3. 안전전환(v1/v2 dual-read + feature flag)

---

## 1) 전역 설계 원칙 (필수)

### 1-1. 입력 소스 화이트리스트 (강제)

v2 핵심 5모듈은 아래 입력 외 사용 금지:

- Snapshot/조회: `ValuationDataProvider` (`trades/valuation/types.py`, `trades/valuation/data_context.py`)
- 계약 스케줄/잔여연차: `contracts/terms.py`
- 보호 정규화: `trades/protection.py`
- 보호/스왑 정산 의미론: `trades/pick_settlement.py` semantics
- 보호 스키마 규칙: `trades/rules/builtin/pick_protection_schema_rule.py`
- fit 공급 벡터 추출: `trades/valuation/fit_engine.py`
- 시즌 cap/apron 수치: `cap_model.py`

### 1-2. 없는 값 처리 규약

- 값이 없으면 억지 추정 금지
- 처리 규약:
  - 수치: 0.0 또는 미계산 중립치
  - 상태: `source_coverage=false`
  - 설명: `MISSING_INPUT_*` / `UNSUPPORTED_RULE_*` reason flag

### 1-3. 단일 계산 책임 (중복 금지)

| 주제 | SSOT | v2 허용 범위 | 금지 |
|---|---|---|---|
| 계약 일정/remaining years | `contracts/terms.py` | 결과 소비 | schedule 재계산 |
| 옵션 입력/상태 | `ContractSnapshot.options` | texture 점수화 | parsed 옵션 파서 강제 |
| 보호 정규화 | `trades/protection.py` | normalize 결과 소비 | 보호 파서 재작성 |
| 보호/스왑 정산 의미 | `trades/pick_settlement.py` | 시나리오 모델링 시 의미 참조 | 정산 룰 재정의 |
| 시즌 cap/apron 수치 | `cap_model.py` | 값 조회/정규화 | cap 수식 재구현 |
| 선수 supply 벡터 | `fit_engine.py` | texture 변환 | supply 추출 로직 재작성 |
| snapshot 로딩 | `ValuationDataProvider` | provider 호출 | repo 직접 재조회 |

### 1-4. 런타임 안전 전환 원칙

- feature flag off: 기존 동작 100% 동일
- feature flag on(단계적): pick → role/contract → cap
- dual-read에서 `collect_v1_v2_diff()`는 **계산결과 diff만** 기록 (원천 데이터 재해석 금지)

## 2) 파일별 상세 설계

## 2.1 `trades/valuation/context_v2.py`

### A. 역할

`context_v2.py`는 v2 valuation 오케스트레이터다.

- `ValuationDataProvider`에서 스냅샷 조회
- 팀별 `DecisionContext`와 결합
- 하위 4모듈(`pick_distribution`, `role_texture`, `contract_texture`, `cap_ledger`) 조립 호출
- v1/v2 결과 diff telemetry 기록

### B. 비역할

- 원천 데이터 파싱/정규화 로직 자체 구현 금지
- repo 직접 접근 금지(반드시 provider 경유)
- 하위 모듈 공식 복제 금지

### C. 원천 매핑표 (필수)

- player/contract/pick/swap snapshot → `RepoValuationDataContext` (`ValuationDataProvider`)
- 계약 스케줄 해석 → `contracts/terms.py`
- protection normalize → `trades/protection.py`
- cap/apron 시즌값 → `cap_model.py`
- fit supply 추출 → `fit_engine.py`

### D. 핵심 타입

```python
@dataclass(frozen=True)
class ValuationContextV2:
    players: Mapping[str, PlayerSnapshot]
    picks: Mapping[str, PickSnapshot]
    swaps: Mapping[str, SwapSnapshot]
    contracts: Mapping[str, ContractSnapshot]
    pick_distributions: Mapping[str, PickDistributionBundle]
    role_textures: Mapping[str, RoleTexture]
    contract_textures: Mapping[str, ContractTexture]
    cap_ledgers: Mapping[str, CapLedgerView]
    diagnostics: ContextDiagnostics
```

### E. 공개 함수

```python
def build_valuation_context_v2(
    provider: ValuationDataProvider,
    decision_context_by_team: Mapping[str, DecisionContext],
    *,
    current_season_year: int,
    current_date_iso: str,
    config: TradeValuationConfig,
    dual_read: bool = False,
) -> ValuationContextV2


def collect_v1_v2_diff(
    v1_ctx: ValuationDataProvider,
    v2_ctx: ValuationContextV2,
) -> ContextDiffReport
```

### F. diff 최소 지표(고정)

초기 dual-read는 아래 3개만 강제:
- `pick_ev_delta`
- `contract_burden_delta`
- `cap_flex_delta`

### G. 예상 작업량

- 구조/adapter: 2 MD
- diff/telemetry: 2 MD
- 호환 안정화: 1~2 MD

합계: **5~6 MD**

## 2.2 `trades/valuation/pick_distribution.py`

### A. 역할

단일 기대순번(`expected_pick_number`)을 분포형 자산으로 전환한다.

### B. 비역할

- 전력예측/부상예측/외부 API 확률 도입 금지
- 보호 파서 재구현 금지
- settlement 규정 재정의 금지

### C. 입력 계약 (R1/R2 분리)

- **R1 필수(현재 실존)**
  - `PickSnapshot` 목록
  - `SwapSnapshot` 목록
  - `standings_order_worst_to_best`
  - `DraftLotteryRules`(시즌 룰 테이블)
- **R2 선택(도입 시에만 사용)**
  - 확장 약정 provider (`terms_provider`)

### D. 핵심 타입

```python
@dataclass(frozen=True)
class PickDistributionBundle:
    pmf: Mapping[int, float]
    cdf: Mapping[int, float]
    ev_pick: float
    variance: float
    scenario_notes: tuple[str, ...]
    compat_expected_pick_number: float | None
    p10_pick: int | None = None
    p50_pick: int | None = None
    p90_pick: int | None = None
    tail_upside_prob: float | None = None
    tail_downside_prob: float | None = None
    source_coverage: Mapping[str, bool] = field(default_factory=dict)
```

### E. 공개 함수

```python
def build_pick_distributions_from_standings(
    *,
    picks: Sequence[PickSnapshot],
    swaps: Sequence[SwapSnapshot],
    standings_order_worst_to_best: Sequence[str],
    season_rules: DraftLotteryRules,
    terms_provider: Optional[PickTermsProvider] = None,
) -> dict[str, PickDistributionBundle]
```

### F. 내부 규칙

1. PMF 합=1.0 검증(±1e-9)
2. 보호는 `normalize_protection()` 경유 결과만 반영
3. `terms_provider is None`이면 `PickSnapshot.protection + SwapSnapshot`만 사용
4. `TOP_N` 외 보호 유형은 `UNSUPPORTED_RULE_PROTECTION` 플래그 + 중립 fallback
5. v2 적용 시 pick/swap 기존 근사 보정과 이중 적용 금지

### G. market_pricing 연동 규칙

- 1차 호환: `compat_expected_pick_number`로 기존 `PickExpectation.expected_pick_number` 주입
- v2 본연 경로 on 시 아래 기존 단계 비활성화:
  - `PICK_PROTECTION_EXPECTATION`
  - `SWAP_EXERCISE_PROB`

### H. 필수/선택 출력 경계

- 필수: `pmf`, `ev_pick`, `variance`, `scenario_notes`
- 선택: `p10/p50/p90`, `tail_*` (입력 충분 시)

### I. 예상 작업량

- PMF 엔진: 2~3 MD
- 보호/스왑 시나리오: 3 MD
- pricing adapter/회귀: 2~3 MD

합계: **7~9 MD**

## 2.3 `trades/valuation/role_texture.py`

### A. 역할

단일 archetype 태그를 대체해 역할을 다차원 texture로 변환한다.

### B. 비역할

- lineup touches/minutes를 실측 없이 추정해 필수값으로 사용 금지
- `fit_engine` 공급벡터 추출 로직 재구현 금지

### C. 입력 우선순위(고정)

1. `player.meta.role_fit`
2. `player.attrs.role_fit`
3. 둘 다 없으면 축별 0.0 + `source_coverage=false`

추가로 `FitEngine.compute_player_supply_vector()` 결과가 있으면 adapter로 주입 사용.

### D. 핵심 타입

```python
@dataclass(frozen=True)
class RoleTexture:
    creation_proxy: float
    spacing_proxy: float
    rim_pressure_proxy: float
    defense_proxy: float
    connector_index: float
    source_coverage: Mapping[str, bool]
    notes: tuple[str, ...] = tuple()
```

### E. 공개 함수

```python
def build_role_textures(
    players: Sequence[PlayerSnapshot],
    *,
    fit_engine: FitEngine,
    injected_supply: Mapping[str, Mapping[str, float]] | None = None,
) -> dict[str, RoleTexture]
```

### F. 지표 경계

- 필수 우선축: `creation/spacing/rim_pressure/defense`
- 선택축: lineup/touches/minutes 파생치(5모듈 R1 범위에서 필수 금지)

### G. 예상 작업량

- 스키마/매핑: 2 MD
- coverage/fallback: 1~2 MD
- adapter 연결: 1~2 MD

합계: **4~6 MD**

## 2.4 `trades/valuation/contract_texture.py`

### A. 역할

`salary * years` 단일근사 대신 계약 질감(texture)으로 평가한다.

### B. 비역할

- 계약 파싱/옵션 파싱 재구현 금지
- cap 룰 엔진 재구현 금지

### C. 입력 계약

- 필수 입력: `ContractSnapshot` (`options` 포함), `current_season_year`
- 선택 입력: `salary_cap`(없으면 cap-share 지표는 coverage false)
- `parsed_option_terms` 같은 신규 파서 입력은 사용하지 않음

### D. 핵심 타입

```python
@dataclass(frozen=True)
class ContractTexture:
    guaranteed_commitment: float
    control_direction: float
    reversibility: float
    trigger_risk: float
    matching_utility: float
    toxic_risk: float
    notes: tuple[str, ...]
    source_coverage: Mapping[str, bool]
```

### E. 공개 함수

```python
def build_contract_textures(
    contracts: Sequence[ContractSnapshot],
    *,
    current_season_year: int,
    salary_cap: float | None,
) -> dict[str, ContractTexture]
```

### F. 내부 산식 규칙

- remaining years / remaining salary total: 반드시 `contracts/terms.py` 사용
- `control_direction`: `ContractSnapshot.options`의 TO/PO/ETO 매핑 사용
- 부분보장/비보장은 `meta`에 실제 키가 있을 때만 반영
- 누락 시 `MISSING_CONTRACT_DETAIL` note + `source_coverage=false`

### G. legacy 교체 포인트

- `package_effects`의 `_commitment_metric()` 경로는 v2 안정화 후 제거

### H. 예상 작업량

- 산식/정규화: 2~3 MD
- 옵션/누락 처리: 2 MD
- package/cap adapter: 1~2 MD

합계: **5~7 MD**

## 2.5 `trades/valuation/cap_ledger.py`

### A. 역할

CAP_FLEX를 재무 운영축으로 분리 계산한다.

### B. 비역할

- CBA 절대 규정 판정기 구현 금지
- cap/apron 수식 재구현 금지(`cap_model.py` 조회만 허용)

### C. 입력 계약

- 팀별 `ContractTexture` 집계
- 팀별 posture(enum 문자열; 기존 decision/generation 계층과 동일 집합)
- cap/apron 시즌 값(`cap_model.py`)
- 팀 payroll이 별도 입력으로 없으면 player snapshot salary 합산만 사용

### D. 핵심 타입

```python
@dataclass(frozen=True)
class CapLedgerView:
    current_operating_room: float
    future_locked_money: float
    optionality_profile: float
    liquidity_profile: float
    apron_pressure: float
    extension_pressure: float | None
    posture_adjusted_flex: float
    source_coverage: Mapping[str, bool]
    ledger_notes: tuple[str, ...]
```

### E. 공개 함수

```python
def build_cap_ledgers(
    team_contract_textures: Mapping[str, Sequence[ContractTexture]],
    *,
    posture_by_team: Mapping[str, str],
    current_season_year: int,
    cap_model: CapModel,
    team_payroll_map: Mapping[str, float] | None = None,
) -> dict[str, CapLedgerView]


def score_cap_flex_delta(
    before: CapLedgerView,
    after: CapLedgerView,
    *,
    posture: str,

```

### F. 필수/선택 지표 경계

- 필수: cap/apron 상대지표(`current_operating_room`, `apron_pressure` 등)
- 선택: `extension_pressure` (입력 충분 시만 계산)

### G. legacy 교체 포인트

- `_cap_flex_adjustment()`의 선형 `delta_commitment` 경로 제거
- `score_cap_flex_delta(before, after)`로 교체

### H. 예상 작업량

- 집계/정규화: 2~3 MD
- posture 가중: 1~2 MD
- valuation 연동/회귀: 1~2 MD

합계: **4~7 MD**

## 3) 파일 간 데이터 흐름 계약

```text
ValuationDataProvider(data_context SSOT)
   -> context_v2(조립)
      -> pick_distribution
      -> role_texture
      -> contract_texture
      -> cap_ledger
   -> valuation adapter
      -> market_pricing v2
      -> package_effects v2
```

규칙:
- 상위 모듈은 하위 모듈 내부 공식을 알면 안 된다.
- 하위 모듈끼리 직접 호출하지 않는다(조립은 `context_v2`만).
- `types_v2.py`는 기존 `types.py`와 의미 중복 타입 생성 금지(래퍼/확장 타입만 허용).

## 5) 테스트/검증 설계

### 5-1. 단위 테스트

- `pick_distribution`
  - PMF 합=1 검증
  - `TOP_N` 보호 경계 케이스
  - terms provider 미주입 fallback 검증
- `role_texture`
  - role_fit 없음 fallback + coverage false 검증
  - 필수 4축 범위(0~1) 검증
- `contract_texture`
  - TO/PO/ETO 방향성 검증
  - remaining years/salary total이 `contracts/terms.py`와 동일함 검증
- `cap_ledger`
  - cap_model 입력 기반 apron pressure 계산 검증
  - posture별 delta 민감도 차이 검증

### 5-2. 통합 테스트

- 동일 입력에서 v1/v2 diff snapshot
- 주요 시나리오:
  - 탱킹팀 + 보호픽
  - 컨텐더 + apron 근접
  - 옵션 다수 포함 계약 패키지

### 5-3. 운영 telemetry

- `v2_missing_input_rate`
- `v1_v2_rank_flip_rate`
- `pick_tail_risk_activation_rate`
- `cap_posture_override_rate`
