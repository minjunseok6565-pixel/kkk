# 현재 스켈레톤 로직 정밀 분석 (BUY/SELL 공통)

이 문서는 **현재 코드 그대로**를 기준으로, 트레이드 생성기에서 스켈레톤 단계가 언제 시작되고, 스켈레톤 템플릿(=registry spec)을 어떤 순서로 검토/통과/탈락시키는지 누락 없이 설명한다.

---

## 1) 스켈레톤 단계 진입 시점

스켈레톤 단계는 `trades/generation/dealgen/core.py`의 메인 생성 루프에서 들어간다.

- BUY 경로:
  1. `select_targets_buy(...)`로 타깃 목록을 만든다.
  2. 타깃별로 seller를 확정한 뒤 `build_offer_skeletons_buy(...)`를 호출한다.
  3. 이 호출 시점이 BUY 스켈레톤 단계 진입 지점이다.

- SELL 경로:
  1. `select_targets_sell(...)`로 매물(`sale_assets`)을 만든다.
  2. 각 매물마다 `select_buyers_for_sale_asset(...)`로 buyer 후보 + `match_tag`를 만든다.
  3. buyer 후보별로 `build_offer_skeletons_sell(...)`를 호출한다.
  4. 이 호출 시점이 SELL 스켈레톤 단계 진입 지점이다.

즉, 실제 진입 조건은 “타깃/매물 선택이 끝났고, 예산/풀캡/쿨다운 조건을 통과한 상태에서 skeleton builder 함수가 호출될 때”다.

---

## 2) 진입 직전의 상위 가드(스켈레톤 단계 전)

스켈레톤 함수 호출 전에 `core.py` 루프에서 다음 가드가 먼저 적용된다.

1. **풀 캡/예산 중단 조건**
   - `len(proposals) >= pool_cap`면 중단.
   - `_budget_or_hard_cap_reached(...)`면 중단.

2. **BUY 전용 soft-stop(예산 근접 조기종료)**
   - validations/evaluations가 soft-stop 임계치 근처 + 결과 절반 이상 확보 시 조기 break.

3. **팀 제약 가드**
   - BUY: seller가 buyer와 동일 팀이면 skip.
   - BUY: seller 팀 `cooldown_active`면 skip.
   - SELL: buyer가 seller와 동일 팀이면 skip.
   - SELL: buyer 팀 `cooldown_active`면 skip.

이 가드를 통과한 경우에만 `build_offer_skeletons_*`로 들어간다.

---

## 3) 스켈레톤 모드 분기: overhaul on/off

`build_offer_skeletons_buy`와 `build_offer_skeletons_sell`은 둘 다 동일하게 다음 분기를 탄다.

- `config.skeleton_overhaul_enabled == True` (기본값 True)
  - **신규 registry 기반 v3 경로**로 진입.
- False
  - 함수 하단의 **레거시 archetype 하드코딩 경로**(picks_only / young+pick / p4p / consolidate 등)를 사용.

현재 기본 설정상 운영 로직은 v3 경로를 우선 사용한다.

---

## 4) v3 경로에서 실제 스켈레톤 템플릿 검토 절차

v3 경로를 타면 검토는 아래 순서로 이뤄진다.

### 4-1. Registry/Context 준비

1. `build_default_registry()`로 전체 스펙(`SkeletonSpec`) 목록 생성.
2. `BuildContext` 생성:
   - 공통: `mode`, `buyer_id`, `seller_id`, `tick_ctx`, `catalog`, `config`, `budget`, `rng`, 금지 집합들.
   - BUY만: `target` 포함.
   - SELL만: `sale_asset`, `match_tag` 포함.

이 `ctx`가 이후 `gate_fn`과 실제 builder 함수에 전달되는 단일 컨텍스트다.

### 4-2. Target tier 산출 시도

코드상 `classify_target_profile(...)`를 호출해 `target_profile`을 만든다.

- BUY: `classify_target_profile(target=target, config=config)`
- SELL: `classify_target_profile(sale_asset=sale_asset, match_tag=match_tag, config=config)`

다만 **현재 파일 상태에서는 `target_tier` 변수가 별도로 할당되지 않은 채 사용**된다.
즉, 아래 호출부에서 `target_tier`를 참조하지만 코드 내 정의가 보이지 않는다.

- `registry.get_specs_for_mode_and_tier("BUY", target_tier, ...)`
- `registry.get_specs_for_mode_and_tier("SELL", target_tier, ...)`

그리고 `target_profile`도 생성만 하고 직접 사용하지 않는다. 즉, 현재 텍스트 기준으로는 “분류 결과를 tier 라우팅으로 연결하는 중간 바인딩이 누락된 형태”다.

### 4-3. Registry 라우팅 필터(=템플릿 검토 핵심)

`SkeletonRegistry.get_specs_for_mode_and_tier(mode, tier, config, ctx)`에서 템플릿이 걸러진다.

필터 순서:

1. `mode_allow` 포함 여부
2. `target_tiers`에 현재 tier 포함 여부
3. config route table 포함 여부
   - tier별 route attribute 선택:
     - ROLE -> `skeleton_route_role`
     - STARTER -> `skeleton_route_starter`
     - HIGH_STARTER -> `skeleton_route_high_starter`
     - STAR -> `skeleton_route_high_starter` (STAR는 HIGH_STARTER 라우트 재사용)
     - PICK_ONLY -> `skeleton_route_pick_only`
     - 그 외 -> 기본 `skeleton_route_starter`
4. `gate_fn(ctx)`가 있으면 True인 spec만 통과

통과한 spec은 정렬되어 반환된다.

- 정렬키: `(priority 오름차순, route 명시 spec 우선, skeleton_id)`

즉 스켈레톤 템플릿 검토는 “등록 목록 전체를 mode/tier/route/gate 4단계로 순차 필터링 후 우선순위 정렬” 구조다.

### 4-4. 빌더 실행 + 메타 부착

필터 통과 spec마다:

1. `built = spec.build_fn(ctx)` 실행 (복수 후보 가능)
2. 각 후보에 `_attach_v3_meta(...)` 적용
   - `skeleton_id`, `skeleton_domain`, `compat_archetype`, `target_tier`, `archetype` 정리
   - 태그 추가:
     - `skeleton:<id>`
     - `domain:<domain>`
     - `target_tier:<tier>`
     - `arch_compat:<compat>`
     - `arch:<archetype>`

### 4-5. Modifier 확장

`apply_modifiers(...)`가 후보를 변형 확장한다(활성 시).

후보별 최대 변형 수 제한(`modifier_max_variants_per_candidate`) 내에서 순차 시도:

1. `protection_step_up_down`
   - 첫 라운드 픽이 있고 보호가 없으면 Top-10/14/20 보호 variants 추가
2. `swap_substitute_for_first`
   - buyer의 first를 제거하고 swap/second 패키지로 대체 variant 추가
3. `second_round_rebalance`
   - second 1개 추가 variant 생성

각 variant는 원본 skeleton metadata를 계승하고 `modifier_trace`, `tags`를 누적한다.

### 4-6. Shape 필터 + beam 자르기

modifier 이후 후보 전체에 대해 `_shape_ok(...)`로 형상 제약 검증.
통과분만 유지 후 다음 컷을 적용:

- `return trimmed_v3[: max(2, int(budget.beam_width))]`

즉 builder/modifier가 많이 만들어도 skeleton 함수 반환 시 1차 beam 컷이 걸린다.

---

## 5) 스켈레톤 단계 이후(후속 파이프라인)

스켈레톤 단계가 끝나도 제안으로 확정되지 않는다. `core.py`에서 후속 단계를 계속 탄다.

1. 통계 반영
   - `stats.skeletons_built += len(candidates)`
   - `_record_candidate_observability(...)`로 skeleton/domain/tier 카운트 적재

2. (BUY만) `expand_variants(...)` 추가 변형

3. 선택 컷
   - optional soft_guard(second apron)
   - `_beam_select_candidates(...)`로 후보 축소

4. 후보별 시도 루프
   - `dedupe_hash(cand.deal)`로 skeleton 중복 제거
   - `repair_until_valid(...)`로 유효화 수리
   - 수리 후 base 딜 hash(`h_valid`) 기준 중복/시도 여지 검사
   - 평가 캐시/스코어링

즉 스켈레톤은 “초기 거래 모양 생성” 단계이고, 실제 성립은 repair/eval/dedupe를 추가로 통과해야 한다.

---

## 6) 현재 registry에 등록된 템플릿 도메인

`build_default_registry()` 기준 도메인/계열은 다음과 같다.

- `compat.*` (기존 archetype 호환)
- `player_swap.*`
- `timeline.*`
- `salary_cleanup.*`
- `pick_engineering.*`

각 스펙은 `skeleton_id`, `domain`, `compat_archetype`, `mode_allow`, `target_tiers`, `priority`, `build_fn`(및 선택적 `gate_fn`)으로 정의된다.

---

## 7) “스켈레톤 템플릿 검토”를 한 문장으로 요약

현재 구현에서 스켈레톤 템플릿 검토는

> “호출 전 예산/쿨다운 가드 통과 → `build_offer_skeletons_*` 진입 → registry에서 mode/tier/route/gate 필터링 → spec별 build_fn 실행 → 메타 태깅 → modifier 확장 → shape/beam 컷”

의 고정 파이프라인으로 동작한다.

---

## 8) 코드상 주의점(현재 상태 그대로 관찰)

현재 `skeletons.py`의 overhaul 분기에는 다음 관찰 포인트가 있다.

1. `target_profile`은 계산하지만 해당 값으로 `target_tier`를 직접 뽑는 코드가 보이지 않는다.
2. 그 상태에서 `target_tier`를 registry 조회와 메타 부착에 사용하고 있어, 파일 텍스트 기준으로는 바인딩 누락 상태다.

즉 “스켈레톤 단계 진입/검토 구조” 자체는 명확하지만, **tier 바인딩 연결 한 줄이 빠진 형태**로 보인다.
