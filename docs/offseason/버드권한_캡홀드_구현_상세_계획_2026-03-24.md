# 버드 권한 + 캡 홀드 시스템 구현 상세 계획 (2026-03-24)

## 0) 문서 목적
이 문서는 현재 코드베이스에 **버드 권한(풀/얼리/논)** 및 **캡 홀드**를 안정적으로 도입하기 위한 실행 설계서다.
핵심 목표는 아래 두 가지다.

1. 재계약(RE_SIGN)에서 버드 권한 규칙을 정확히 강제한다.
2. 오프시즌 FA 대상자의 캡 홀드를 팀 샐러리에 반영하고, 권리 포기(renounce) 시 즉시 해제한다.

---

## 1) 목표(완료 조건)

### 1.1 기능 목표
- 버드 권한 종류
  - `FULL_BIRD`: 동일 팀 3년 이상
  - `EARLY_BIRD`: 동일 팀 2년 이상
  - `NON_BIRD`: 동일 팀 1년 이상
- 버드 권한은 **재계약(RE_SIGN) 시에만** 사용 가능
- 버드 권한 사용 시 캡 룸 초과 재계약 허용(권한별 상한 강제)

### 1.2 권한별 계약 상한
- `FULL_BIRD`
  - 첫해 연봉: **기존 exp-based max 정책(샐캡 비율) 재사용**
  - 최대 기간: 5년
  - 최대 인상률: 8%
- `EARLY_BIRD`
  - 첫해 연봉: `max(직전 연봉 * 1.75, 시즌별 리그 평균 연봉 * 1.05)`
  - 최대 기간: 4년
  - 최대 인상률: 8%
- `NON_BIRD`
  - 첫해 연봉: `직전 연봉 * 1.20` 까지만 허용
  - 최대 기간: 4년
  - 최대 인상률: **5%**

### 1.3 캡 홀드 규칙
- 오프시즌에 우리 팀에서 FA가 된 선수는 캡 홀더로 팀 샐러리에 포함
- 권리 포기(renounce) 시 캡 홀드 해제
- 권리 포기 후에는 외부 FA와 동일하게 `STANDARD_FA/MLE/MINIMUM`만 허용
- 캡 홀드 금액
  - `FULL_BIRD`: 직전 연봉 x 1.5
  - `EARLY_BIRD`: 직전 연봉 x 1.3
  - `NON_BIRD`: 직전 연봉 x 1.2

### 1.4 비기능 목표
- 오프시즌 재실행 시 결과가 일관되도록 처리 순서 고정
- 팀 샐러리 계산 SSOT 유지(중복 계산 최소화)
- 정책/서비스/API 계층 분리 유지

## 2) 완료 시 사용자 관점 동작 시나리오

### 시나리오 A: EARLY_BIRD 재계약
1. 오프시즌 시작 후 자팀 만료 계약자 A가 FA 상태가 된다.
2. 시스템이 A의 동일팀 근속을 계산해 `EARLY_BIRD` 권한과 cap hold를 생성한다.
3. 유저가 RE_SIGN 협상에서 캡룸을 초과하는 오퍼를 입력한다.
4. 오퍼가 EARLY_BIRD 첫해 상한/기간/인상률(8%)을 만족하면 커밋 성공한다.
5. 커밋 시 A의 cap hold가 자동 해제되고, 실제 계약 연봉만 팀 샐러리에 남는다.

### 시나리오 B: NON_BIRD 상한 위반
1. NON_BIRD 대상자에게 첫해 연봉을 직전 연봉 120% 초과로 입력한다.
2. 검증은 즉시 실패한다.
3. 동일하게 연 인상률이 5%를 초과해도 실패한다.

### 시나리오 C: 권리 포기 후 재영입
1. 유저가 선수 B에 대해 `renounce`를 실행한다.
2. B의 cap hold가 즉시 해제된다.
3. 이후 B 재영입 시 BIRD 채널은 선택 불가이며 `STANDARD_FA/MLE/MINIMUM`만 가능하다.

## 4) 규칙 계산 SSOT 설계

### 4.1 신규 정책 모듈

#### `contracts/policy/bird_rights_policy.py` (신규)
담당 함수:
- `classify_bird_type(tenure_years_same_team: int) -> str`
- `cap_hold_multiplier(bird_type: str) -> float`
- `max_years_for_bird_type(bird_type: str) -> int`
- `max_raise_pct_for_bird_type(bird_type: str, trade_rules: Mapping | None) -> float`
- `first_year_max_for_bird_type(...) -> int`
  - 입력: `bird_type`, `prev_salary`, `exp`, `season_year`, `trade_rules`
  - 출력: 첫해 최대 연봉

핵심 계산 원칙:
- `FULL_BIRD`: 기존 exp-based max 정책 함수 호출(샐캡 비율)
- `EARLY_BIRD`: `max(prev_salary*1.75, league_avg_salary(season_year)*1.05)`
- `NON_BIRD`: `prev_salary*1.20`

### 4.2 시즌별 리그 평균 연봉 산식
- 평균 연봉은 고정 상수가 아니라 **시즌별로 계산**한다.
- 계산 위치: 정책 호출 전에 서비스 계층에서 시즌 대상 선수 집합 기준 산출
- 기본 쿼리 제안:
  - 대상 시즌 활성 로스터(`roster.status='active'`)의 `salary_amount` 평균
  - 값이 0이거나 데이터가 비어 있으면 정책 fallback 사용

### 4.3 trade_rules 키
- `bird_early_multiplier_prev_salary` (default: 1.75)
- `bird_early_multiplier_league_avg` (default: 1.05)
- `bird_non_multiplier_prev_salary` (default: 1.20)
- `bird_raise_max_pct_by_type` (FULL/EARLY/NON)
- `bird_cap_hold_multiplier_by_type` (FULL/EARLY/NON)

권장 기본값:
- FULL raise: 0.08
- EARLY raise: 0.08
- NON raise: 0.05

## 5) 팀 샐러리 계산 원칙

### 5.1 계산식
- `team_salary_for_cap = active_roster_payroll + active_cap_holds`
- active cap hold 조건
  - `season_year = 대상 시즌`
  - `is_released = 0`

### 5.2 적용 범위
- STANDARD_FA cap room 판단
- MLE 채널 eligibility 판단
- RE_SIGN 시 버드 권한 검증 전후 금액 검증

### 5.3 중복 반영 방지
- RE_SIGN 커밋 성공 시 대상 선수 cap hold를 먼저 `release` 처리 후 계약 반영
