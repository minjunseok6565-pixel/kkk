# 오프시즌 로터리/컴바인 단계형 구현 브리핑

## 1) 요청사항 요약 (내가 이해한 목표)
- 현재 구현된 **로터리 확률 표시 화면** 다음 단계에서 `다음으로` 버튼 기반으로 아래 순서를 만들고 싶다.
  1. 로터리 당첨팀(in 4)만 먼저 공개
  2. 한 번 더 눌러 1~4픽 순서 공개
  3. 보호/스왑 반영 + 최종 턴 생성 후 1~30픽 공개
  4. 다시 `다음으로` 누르면 컴바인 화면으로 이동
  5. 컴바인 화면에서 항목별 Top10 카드 + 클릭 시 전체 결과(전문가 빅보드 UX 유사)

---

## 2) 백엔드와의 정합성 체크

### 2-1. 로터리/정산 분리는 이미 백엔드에서 지원됨
- `/api/offseason/draft/lottery`는 `run_lottery`를 호출해 **플랜(로터리 결과 포함)** 을 저장/반환한다.
- `/api/offseason/draft/settle`는 `run_settlement`으로 **보호/스왑 정산 + 최종 턴(turns) 생성**을 수행한다.
- 즉, 요청하신 “로터리 단계”와 “정산/최종 턴 단계”는 백엔드 API 레벨에서 이미 분리돼 있다.

### 2-2. “in 4 공개”와 “1~4픽 공개”는 프론트 단계 분리가 맞음
- Lottery 결과(`winners_top4`)는 한 번 계산되며, 1~4픽 팀도 같은 데이터에 이미 들어 있다.
- 그래서 백엔드에 `in4 전용 API`가 없더라도, 프론트에서
  - 1차 화면: 당첨팀 4개만 노출
  - 2차 화면: 같은 결과를 슬롯(1~4)으로 재배치해 노출
  방식으로 충분히 단계형 UX를 만들 수 있다.

### 2-3. 현재 프론트 오프시즌 플로우와의 차이
- 현재 `offseasonDevFlow.js`는 은퇴 이후 `runLotteryAndSettleStep()`에서 lottery/settle을 **한 번에 연속 호출**하고,
  화면도 `LOTTERY_SETTLED` 한 단계로 “확률표 + 소유권(1~14)”를 같이 보여준다.
- 따라서 지금 요청사항을 반영하려면 오프시즌 상태머신(step)만 더 잘게 쪼개면 된다.

---

## 3) 현실적 구현안 (현재 프로젝트 기준)

## 3-1. 권장 단계 상태 설계
기존 흐름(은퇴 이후)을 아래처럼 분해:
1. `LOTTERY_ODDS` (기존처럼 확률표)
2. `LOTTERY_IN4_REVEALED` (당첨팀 4개만)
3. `LOTTERY_TOP4_ORDER_REVEALED` (1~4픽 순서)
4. `DRAFT_SETTLED_ROUND1` (settle 후 1~30픽)
5. `COMBINE_OVERVIEW` (항목별 Top10 카드)
6. `COMBINE_DETAIL` (카드 클릭 시 전체 결과)

핵심은 `lottery` 호출은 1회만 하고, 결과를 여러 스텝에서 재사용하는 것.

## 3-2. 로터리/정산 UX 구현 포인트
- `RETIREMENT_PROCESSED -> LOTTERY_ODDS`에서 `/api/offseason/draft/lottery`만 호출.
- `LOTTERY_IN4_REVEALED`, `LOTTERY_TOP4_ORDER_REVEALED`는 API 추가 호출 없이 `draftLotteryResult.plan.lottery_result.winners_top4`를 단계적으로 렌더.
- `LOTTERY_TOP4_ORDER_REVEALED -> DRAFT_SETTLED_ROUND1`로 넘어갈 때 `/api/offseason/draft/settle` 호출.
- `DRAFT_SETTLED_ROUND1`에서는 `turns`에서 `round===1`, `slot 1..30`만 정렬해 표기.

## 3-3. 컴바인 화면 구현 포인트
현재 백엔드 사실상 선택지:
- `POST /api/offseason/draft/combine`: 컴바인 결과 생성/저장(실행)
- `GET /api/offseason/draft/bundle`: 공개 가능한 풀(선수 메타 포함) 조회

권장 구현:
1. `DRAFT_SETTLED_ROUND1 -> COMBINE_OVERVIEW` 전환 시 `draft/combine` 실행(멱등 고려).
2. `draft/bundle`로 풀을 가져와 prospect `meta.combine`에서 항목별 값 추출.
3. 예: `vertical`, `lane_agility`, `sprint_3_4`, `shuttle`, `standing_reach` 등 카테고리별 Top10 카드 렌더.
4. 카드 클릭 시 해당 항목 기준 전체 정렬 테이블(혹은 패널) 노출.

참고: “전문가 빅보드 유사 UX”는 이미 `college/bigboard.js`에 카드+디테일 패턴이 있으므로 재사용 가능성이 높다.

---

## 4) 백엔드와 안 맞을 수 있는 포인트 / 확인 필요사항

1. **컴바인 조회 전용 API 부재**
   - 현재는 combine 실행 API는 있으나, “항목별 랭킹 전용 응답” API는 없다.
   - 다만 `draft/bundle`의 pool 메타를 사용하면 프론트에서 정렬/집계 가능.

2. **컴바인 메타 키 스키마 확인 필요**
   - 프론트 카드 항목명을 확정하려면 `meta.combine` 내부 키(측정/드릴 필드명)를 실제 payload 기준으로 1회 확인해야 한다.

3. **현재 Dev Flow 단일 단계 구조**
   - `LOTTERY_SETTLED` 단일 스텝을 다단계로 나누는 리팩터링이 필요.
   - 기술 난이도는 높지 않지만, step 전이와 버튼 라벨/비활성 조건 정리가 중요.

---

## 5) 작업 범위/난이도/달성 가능성 판단
- **달성 가능성: 높음 (현 구조에서 현실적)**
- 이유:
  - 로터리/정산 API가 이미 분리되어 있음.
  - 로터리 결과를 프론트 단계형으로 쪼개 보여주는 것은 순수 UI 상태머신 문제.
  - 컴바인 Top10 카드 UX는 기존 bigboard UI 패턴 재활용 가능.

예상 난이도(상대):
- 로터리 단계 분리: 하
- 1~30픽 최종 정렬 렌더: 하
- 컴바인 카드+디테일(프론트 정렬/필터): 중
- 전체 step 전이 안정화(연타/오류 복구): 중

---

## 6) 권장 구현 순서 (실행 플랜)
1. `offseasonDevFlow` step 분해 (lottery odds -> in4 -> top4 -> settle -> round1)
2. 기존 `LOTTERY_SETTLED` 렌더를 3개 렌더 함수로 분할
3. settle 결과에서 1~30픽 전용 테이블 렌더 추가
4. combine 실행 + bundle 로드 유틸 추가
5. combine 카테고리 Top10 카드 컴포넌트 작성
6. 카드 클릭 시 전체 리스트 패널/모달 작성
7. 버튼 연타 방지/오류 재시도 UX 정리

---

## 7) 결론
- 요청하신 플로우는 **현재 백엔드와 큰 충돌 없이 구현 가능**하다.
- 특히 로터리 2단계 공개(in4 -> 1~4)는 백엔드 추가개발 없이 프론트 상태 전이로 해결 가능.
- 컴바인은 조회 전용 API가 없어도 `draft/bundle` 기반 프론트 집계로 시작할 수 있고,
  추후 필요하면 카테고리 랭킹 전용 API를 추가하는 확장 경로도 깔끔하다.
