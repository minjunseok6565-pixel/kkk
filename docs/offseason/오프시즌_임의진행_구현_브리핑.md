# 오프시즌 임의 진행(DEV) 구현 브리핑

## 결론 요약
- **지금 계획은 프론트 관점에서 충분히 구현 가능**합니다.
- 다만 현재 서버 동작 기준으로는, 화면 흐름 사이에 반드시 들어가야 하는 API 단계가 몇 개 있고(특히 `contracts/process`), 몇 화면은 **신규 API 보강 없이 완전 구현이 어렵습니다**(예: 만료 계약자 목록 조회 전용 API).
- 따라서 1차 구현은 **"개발용 임시 UI + 기존 API 조합"**으로 진행하고, 2차로 필요한 API를 보강하는 방식이 가장 안전합니다.

---

## 현재 코드 기준 핵심 사실(계획과 맞물리는 포인트)

### 1) 오프시즌 진입 전 챔피언 확정은 필수
- `/api/season/enter-offseason`는 `postseason.champion`이 없으면 실패합니다.
- 즉 "임의 진행" 버튼은 내부적으로 **챔피언을 먼저 확정**해줘야 합니다.
- 다행히 현재 코드에는 플레이오프를 전부 수동 경기로 돌리지 않고도 챔피언을 만들 수 있는 루트가 있습니다.
  - `/api/postseason/setup` (필드 생성)
  - `/api/postseason/playoffs/auto-advance-round` (라운드 자동 소화)
  - 이걸 반복해서 챔피언이 생기면 `enter-offseason` 호출 가능

> 의미: "한 시즌 통째로 임의 진행"을 **매치엔진 일자 진행 없이** 가는 전략과 코드 방향이 맞습니다.

### 2) `college/finalize` 다음에 바로 드래프트/은퇴로 가면 안 되고 `contracts/process`가 중간에 필요
- 현재 오프시즌 실제 파이프라인은 `enter-offseason` 응답의 `steps`에도 명시되어 있고,
  `college/finalize -> contracts/process -> retirement/process -> ...` 순서를 기대합니다.
- 특히 `draft/apply`는 내부 하드게이트로
  - `contracts_offseason_done_{to_year}`
  - `retirement_processed_{to_year}`
  둘 다 없으면 409로 막습니다.

> 의미: 사용자가 제안한 UI 단계에서 `contracts/process`를 명시적으로 넣어야 합니다.

### 3) TEAM 옵션과 PLAYER 옵션은 분리 UX는 가능하지만, 서버 처리 시점은 다름
- TEAM 옵션:
  - `/api/offseason/options/team/pending`
  - `/api/offseason/options/team/decide`
  - 유저 팀 미결정이면 `/api/offseason/contracts/process`가 409 차단
- PLAYER/ETO 옵션:
  - 별도 "유저 결정 API"가 아니라 `contracts/process`에서 자동 정책으로 처리됩니다.

> 의미: 기획한 "선수 옵션 화면"은 **정보 표시 화면**으로는 가능하지만,
> 실제 반영 트리거는 별도 버튼이 아니라 이미 `contracts/process` 완료 시점입니다.

### 4) `contracts/process`는 `user_team_id`가 필수
- 요청 바디에 `user_team_id`를 넣어야 합니다.
- 프론트에선 이미 `state.selectedTeamId`를 갖고 있으므로 연결은 쉽습니다.

### 5) "재계약 제의/방출" 단계는 일부 API 조합이 필요
- 협상 시작/오퍼/커밋 API는 이미 존재합니다.
  - `/api/contracts/negotiation/start`
  - `/api/contracts/negotiation/offer`
  - `/api/contracts/negotiation/commit`
  - `/api/contracts/re-sign-or-extend`
- 방출(FA 전환)도 `/api/contracts/release-to-fa`가 있습니다.
- 다만 **"지난 시즌 종료로 만료된 선수 목록" 전용 조회 API**는 바로 보이는 형태로 준비돼 있지 않습니다.
  - 현재는 이 목록을 직접 만들려면 DB/서비스 조회를 위한 별도 API를 추가하는 게 낫습니다.

### 6) 로터리 확률/1픽 확률 + 픽 실소유권 판정은 기존 draft 단계로 가능
- `/api/offseason/draft/lottery`에서 로터리 플랜(상위픽 결과) 생성
- `/api/offseason/draft/settle`에서 보호/스왑 정산 + 최종 턴 생성
- "어느 팀 픽이 어디로 넘어갔는지"는 settle 결과를 화면에 매핑하면 표현 가능합니다.

## 사용자가 제시한 단계별 계획과 실제 코드 간 차이

## A. "임의 진행 버튼" 직후 플레이오프 우승 화면
- **가능**합니다.
- 다만 현재 프론트에 해당 화면/라우트가 없으므로 신규 스크린을 만들어야 합니다.
- 추가로, 우승 화면 진입 전에 챔피언 보장이 필요하므로 아래 중 하나가 필요:
  1. 기존 postseason API를 연쇄 호출해서 챔피언 생성
  2. DEV 전용 백엔드 API 하나로 "챔피언 강제 확정 + 요약데이터 생성" 처리

권장: 1차는 (1)로 구현(서버 변경 최소화), 2차에 DEV 단축 API 추가.

## B. 오프시즌 진입 후 "다음으로" 단계
- 제안한 흐름 자체는 맞지만, 실제론 아래 순서로 재배열해야 안전합니다.

권장 실제 순서(현재 코드 적합):
1. 우승 화면 → `/api/season/enter-offseason`
2. 오프시즌 안내 화면 → `/api/offseason/college/finalize`
3. 대학 최종 순위/리더보드 표시
4. 드래프트 선언자 표시
5. TEAM 옵션 결정 화면
   - pending 조회/decide
6. **여기서 `/api/offseason/contracts/process` 호출(필수)**
7. PLAYER 옵션 결과 화면(이미 처리된 결과를 읽어 보여줌)
8. 만료 선수 재계약/방출(협상 API + release API)
9. `/api/offseason/retirement/process`
10. 은퇴 선수 목록 표시
11. 로터리 확률/1픽 확률 + 픽 소유권(정산 전/후) 표시
   - 필요 시 `/api/offseason/draft/lottery`, `/api/offseason/draft/settle`

## C. "player option에서 행사/미행사에 따라 로스터 잔류/FA 이동"
- 결과적으로는 맞는 설명입니다.
- 단, 이 반영이 일어나는 트리거는 **선수 옵션 화면 버튼이 아니라 `contracts/process`**입니다.

## D. "대학 최종 순위와 리더보드"
- `college/finalize`는 데이터를 만들지만,
- 이를 화면에서 보여주기 위해서는 `/api/college/teams`, `/api/college/players` 등의 조회를 함께 묶어야 합니다.
- 즉 finalize 호출만으로 자동 렌더링되진 않습니다.

## 프론트 구현 관점 제안(안전한 1차 범위)

### 1) DEV 임의 진행 버튼
- 메인 화면의 DEV 버튼 영역에 추가.
- 클릭 시:
  1. postseason setup/auto-advance로 챔피언 확보
  2. 임시 "플레이오프 우승" 화면으로 이동

### 2) 임시 오프시즌 화면 스택
- 신규 스크린 5~7개를 최소 UI 카드 형태로 추가:
  - 우승 화면
  - 오프시즌 안내
  - 대학 결과/선언자
  - TEAM 옵션
  - PLAYER 옵션 결과
  - 만료 선수 처리
  - 은퇴 결과
  - 로터리/픽 소유권

### 3) 단계 상태머신
- 프론트 state에 `offseasonDevFlowStep` 같은 상태를 두고,
- 각 단계의 "다음으로" 클릭 시 필요한 API를 실행 후 step 전환.
- API 실패 시 현재 단계 유지 + 에러 텍스트 출력.

### 4) "되돌리기 어려운" 단계는 확인 모달
- TEAM 옵션 확정
- release-to-fa
- retirement/process

## 지금 바로 필요한 보강(백엔드/프론트)

## 필수(또는 거의 필수)
1. 만료 계약자 목록 조회 API(권장 신규)
   - 예: `/api/offseason/contracts/expired?team_id=...&season_year=...`
2. PLAYER 옵션 처리 결과 조회 API(권장 신규)
   - 현재는 contracts 결과 payload나 DB 조회를 프론트가 직접 파싱해야 할 수 있음

## 있으면 좋은 것
3. DEV 단축 API
   - 예: `/api/dev/season/simulate-to-offseason`
   - 내부에서 챔피언 확정 + postseason summary 생성까지 처리하면 프론트 단순화
4. 오프시즌 플로우 요약 API
   - 현재 단계/완료 메타키/다음 권장 단계 반환

## 리스크 체크
- `contracts/process` 누락 시 이후 단계가 논리적으로 틀어지거나 `draft/apply`에서 막힙니다.
- TEAM 옵션 미결정 상태에서 `contracts/process` 호출 시 409가 정상 동작입니다(예외가 아니라 설계).
- "선수 옵션 화면에서 유저가 결정"은 현재 설계와 다르므로, UX 문구를 "결과 확인"으로 분리해야 혼선이 없습니다.

---

## 최종 판단
- 사용자의 방향은 **큰 틀에서 코드와 합치**합니다.
- 다만 실제 게임 동작과 맞추려면, 핵심 차이점은 다음 3개입니다.
  1. 챔피언 확정 선행 필수
  2. TEAM 옵션 뒤 `contracts/process` 필수 삽입
  3. PLAYER 옵션은 결정 화면이 아니라 결과 확인 화면
- 이 3개만 반영하면, 요청한 "임의 진행 기반 오프시즌 진입/초기 단계"는 현실적으로 구현 가능합니다.
