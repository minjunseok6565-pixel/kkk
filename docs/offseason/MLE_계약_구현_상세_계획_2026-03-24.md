# MLE 계약 시스템 구현 상세 계획 (2026-03-24)

## 1) 목표

현재 프로젝트의 일반 FA 계약 흐름을 유지하면서, **MLE(Non-Taxpayer / Taxpayer / Room) 계약을 별도 채널**로 추가한다.

핵심 방향:

- 일반 FA 계약(`STANDARD_FA`)은 기존과 동일하게 `salary_cap` 하드 체크를 받는다.
- MLE 계약(`NT_MLE`, `TP_MLE`, `ROOM_MLE`)은 **cap 초과 상황에서도** 체결 가능하되,
  - 팀의 apron/room 자격,
  - 타입별 금액/기간 제한,
  - 연봉 상승률 제한,
  - 시즌별 MLE 예산(채널별 첫해 한도)의 잔액 제한
  을 반드시 통과해야 한다.

---

## 2) 최종 구현이 완료됐을 때의 동작 형태 (To-Be)

### 2.1 협상 시작

사용자가 FA 선수와 협상을 시작하면, 서버는 다음을 계산해 반환한다.

- `available_contract_channels`: 사용 가능한 계약 채널 목록
  - 예: `["STANDARD_FA", "NT_MLE"]`
- 각 채널별 제약
  - `max_first_year_salary`
  - `max_years`
  - `max_raise_pct` (기본 0.05)
  - `remaining_first_year_budget_this_season`

UI는 이 목록을 바탕으로 계약 채널을 선택하게 한다.

### 2.2 오퍼 제출/카운터

오퍼 payload에 `contract_channel`이 포함된다.

- `STANDARD_FA`: 기존 협상 규칙 유지
- `*_MLE`: 금액/기간/상승률 제약을 실시간 검증
  - 조건 위반 시 협상 단계에서 즉시 REJECT/에러 응답

### 2.3 커밋(계약 확정)

커밋 시 서버는 협상 세션의 `contract_channel`을 기준으로 분기한다.

- `STANDARD_FA`
  - 기존 `sign_free_agent` 경로 사용
  - `payroll_after > salary_cap`이면 거절
- `NT_MLE` / `TP_MLE` / `ROOM_MLE`
  - MLE 전용 커밋 경로 사용
  - 일반 cap 하드체크는 우회
  - 채널별 **사용 횟수 제한은 두지 않음**
  - 대신 아래를 강제:
    - 채널 자격(팀 payroll vs apron, room history)
    - 채널별 첫해 금액 상한
    - 채널별 최대 기간
    - 연차 인상률(첫해 대비 시즌별 상승폭 <= 5%)
    - 시즌 내 채널별 잔여 첫해 예산

성공 시 transactions_log에 `tx_type="SIGN_FA_MLE"`(또는 channel 포함 payload)로 기록되고,
팀의 시즌 MLE 사용 상태가 반영된다.

### 2.4 시즌 전환

- `season_year` 변경 시, MLE 기준 금액은 전 시즌 대비 10% 상승한다.
- 팀별/채널별 MLE 첫해 예산 사용 누적은 새 시즌 기준으로 리셋된다.
- Room MLE 자격 플래그(“해당 시즌에 cap 아래로 내려간 적 있는지”)도 시즌 단위로 재평가/초기화된다.

### 2.5 예시 (25-26 기준)

- NT_MLE: 첫해 최대 14.0M, 최대 4년
- TP_MLE: 첫해 최대 5.6M, 최대 2년
- ROOM_MLE: 첫해 최대 8.8M, 최대 3년
- 연봉 인상률: year-to-year 최대 5%
- 다음 시즌(26-27): 각 MLE 첫해 최대 금액 10% 증가

## 3) 데이터/규칙 모델 설계

## 3.1 contract channel enum 추가

- `STANDARD_FA`
- `NT_MLE`
- `TP_MLE`
- `ROOM_MLE`

협상 세션과 최종 계약 이벤트 payload 모두에 channel을 남겨야 한다.

### 3.2 MLE 규칙 SSOT

`league.trade_rules` 하위에 MLE 규칙을 추가한다.

예시:

```json
{
  "mle_enabled": true,
  "mle_base_season_year": 2025,
  "mle_annual_growth_rate": 0.10,
  "mle_channels": {
    "NT_MLE": {"first_year_base": 14000000, "max_years": 4},
    "TP_MLE": {"first_year_base": 5600000, "max_years": 2},
    "ROOM_MLE": {"first_year_base": 8800000, "max_years": 3}
  }
}
```

> `max_raise_pct`는 MLE 전용 키로 두지 않고, 아래 3.4의 **공통 연봉 상승률 SSOT**에서 계약 채널별로 관리한다.

### 3.3 팀별 시즌 상태(예산 사용 누적/room 자격)

아래 중 하나로 관리:

1. `meta` key-value로 JSON 저장(빠른 도입)
2. `team_exception_budget_usage` 같은 전용 테이블 추가(중장기 안정성)

권장: 전용 테이블.

예시 컬럼:

- `season_year`
- `team_id`
- `channel`
- `first_year_spent_total`
- `updated_at`

Room 자격 이력은 별도 테이블 또는 meta 키로 관리:

- `team_room_mle_eligibility(season_year, team_id, became_below_cap_once)`

### 3.4 공통 연봉 상승률 SSOT 분리 (신규)

MLE뿐 아니라 아직 구현되지 않은 계약 형태까지 확장 가능하도록,
연봉 상승률 제한을 별도 SSOT 파일/모듈로 분리한다.

권장 신규 파일:

- `contracts/policy/raise_limits.py`

권장 책임:

1. 계약 채널/계약 타입별 최대 상승률 정의 (기본값 + 오버라이드)
2. 시즌/리그 룰에 따른 동적 override 로딩
3. 공통 검증 함수 제공
   - `max_raise_pct_for_contract_channel(channel, trade_rules, season_year)`
   - `validate_salary_raise_curve(salary_by_year, max_raise_pct)`

예시 구조:

```python
RAISE_LIMITS_SSOT = {
    "STANDARD_FA": 0.08,   # 예시 (추후 튜닝)
    "NT_MLE": 0.05,
    "TP_MLE": 0.05,
    "ROOM_MLE": 0.05,
    "RE_SIGN": 0.08,       # 예시
    "EXTEND": 0.08,        # 예시
}
```

핵심 원칙:

- MLE/비-MLE 모두 동일한 validator를 사용한다.
- 개별 서비스 로직에서 5% 하드코딩 금지.
- 신규 계약 형태 추가 시 SSOT 매핑만 확장하면 되게 설계한다.

## 4.2 MLE 계산/자격 도메인 모듈 신설

### 신규: `contracts/mle_policy.py`

구현 책임:

1. 시즌별 채널 상한 계산
   - `first_year_limit_for_channel(channel, season_year, trade_rules)`
2. 팀 자격 판정
   - `eligible_channels_for_team(team_id, season_year, cur, trade_rules)`
3. 오퍼 검증
   - `validate_mle_offer(channel, offer, season_year, trade_rules)`
   - 첫해 금액, 기간
   - 상승률은 `contracts/policy/raise_limits.py` 공통 validator 호출
4. 시즌 예산 잔액 검증/차감
   - `get_remaining_first_year_budget(...)`
   - `consume_first_year_budget(...)`

입력은 `trade_rules` + DB cursor를 받아 순수 함수/작은 서비스 형태로 유지.

## 4.4 LeagueService 계약 커밋 경로

### `league_service.py`

#### A. 신규 메서드 추가

- `sign_free_agent_with_channel(...)`
  - 인자: `contract_channel`
  - 내부 분기:
    - `STANDARD_FA` -> 기존 `sign_free_agent` 재사용
    - `*_MLE` -> MLE 검증 + 커밋

#### B. 기존 `sign_free_agent` 유지

- 하위 호환을 위해 변경 최소화
- 내부적으로 `STANDARD_FA` 전용 경로로 남김

#### C. MLE 커밋 시 검증

- team payroll / apron 기반 자격 재검증
- 오퍼 검증(금액/기간 + 공통 상승률 SSOT 검증)
- 시즌 예산 잔액 검증 후 첫해 연봉만큼 차감
- 거래 로그 기록:
  - `type="SIGN_FA_MLE"`
  - payload에 `contract_channel` 포함

#### D. Room 자격 플래그 업데이트 훅

- 팀 payroll 변동이 발생하는 지점(계약/트레이드 처리)에서
  `payroll < salary_cap` 순간을 감지해 `team_room_mle_flags` 갱신.

## 7) 완료 정의 (Definition of Done)

아래를 모두 만족하면 본 계획의 구현 완료로 본다.

1. 일반 FA 계약은 기존 cap 하드체크 동작 유지.
2. MLE 3종 채널이 협상/커밋에서 선택 가능.
3. 각 채널별 자격/금액/기간/인상률 제약이 강제됨.
4. 시즌별 MLE 금액 10% 증가가 자동 반영됨.
5. 시즌 내 채널별 첫해 예산 잔액 제한이 강제되고, 다음 시즌에 reset됨.
6. Room MLE 자격(시즌 중 cap below 이력)이 동작함.
7. API/프론트에서 실패 사유가 명확히 노출됨.
8. 신규 테스트(정책/서비스/API) 통과 + 기존 회귀 테스트 통과.
