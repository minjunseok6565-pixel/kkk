# 연장 계약(Extend) 시스템 정밀 룰 패치 설계안 (2026-03-27)

## 1) 목표 (Goals)

현재 프로젝트의 `EXTEND` 흐름을 단일 일반 연장 처리에서, 아래 3가지 연장 유형을 명시적으로 구분하고 CBA 유사 규칙을 강제하는 구조로 고도화한다.

- **Rookie Scale Extension (RSE)**
- **Veteran Extension (VE, 일반 베테랑 연장)**
- **Designated Veteran Extension (DVE, 슈퍼맥스)**

이번 패치의 범위는 **코드 레벨 룰 강제 및 API/협상 흐름 반영**이며, **DB 마이그레이션/하위 호환 처리 없이 현 개발 DB 기준으로 즉시 반영**한다.

---

## 2) 패치 완료 시 기대 동작 (Expected Behavior)

### A. 루키 스케일 연장 + 팀옵션

1. `EXTEND` 협상 시작 시, 해당 선수가 1라운드 루키스케일 계약 대상인지 판정하고, 아니면 RSE 모드 불가 응답을 준다.
2. RSE는 **3년차 종료 후~4년차 개막 전** 윈도우에서만 시작/커밋 가능하다.
3. RSE는 **4년차 팀옵션이 EXERCISED** 상태여야만 가능하다.
4. RSE 총 기간(서명 시점 기준, 옵션 포함)은 6시즌 한도를 넘지 않는다.
5. RSE 연장 1년차 연봉 상한은 기본적으로 `max(25% of cap, 직전연봉의 105%)`로 계산한다.
6. RSE 연봉 증감은 **매년 최대 (연장 1년차 금액 * 8%)의 고정폭**으로 제한된다.

### B. 일반 베테랑 연장

1. 1년/2년 원계약은 연장 불가.
2. 3~4년 원계약은 계약서명 2주년 이후, 5~6년 원계약은 3주년 이후만 연장 가능.
3. 일반 연장의 총 기간(옵션 포함)은 서명 시점 기준 최대 5시즌.
4. 일반 연장 1년차 상한은 `max(직전시즌 regular salary의 140%, EAPS의 140%)`와 max-tier 상한을 동시에 만족해야 한다.
5. max-tier는 연장 첫해 기준 경력 구간(<=6:25%, 7~9:30%, >=10:35%)과 `직전연봉의 105%` 비교를 반영한다.
6. 연봉 증감은 **연장 1년차 기준 고정 8%폭**.
7. 연장 첫해 시작이 1년보다 더 먼 시점이면 오프시즌에서만 허용한다.

### C. DVE (슈퍼맥스)

1. 계약 잔여 1~2년일 때만 가능.
2. 서명 시점 7~8년 경력 선수만 가능.
3. 팀 자격(원소속팀 또는 초기 4개 cap year 내 트레이드 이동 예외) 검증.
4. Higher Max Criteria 충족 필요.
5. DVE 총 기간은 서명 시점 기준 6시즌.
6. 연장 1년차는 cap 30~35% 범위, 연봉 증감은 1년차 기준 고정 8%폭.
7. DVE는 incentive compensation을 허용하지 않는다.

### D. 옵션 행사 윈도우

1. 팀옵션 행사 API에서 옵션 타입/시즌 뿐 아니라 **의사결정 날짜가 허용 윈도우에 있는지** 검증한다.
2. 허용 윈도우 밖이면 명시 에러코드로 거절한다.
3. 루키스케일 팀옵션은 아래 절대 구간을 사용한다.
   - 3년차 옵션: **1년차 시즌 종료 다음날 ~ 2년차 개막일 전날**
   - 4년차 옵션: **2년차 시즌 종료 다음날 ~ 3년차 개막일 전날**

## 3) 구현 원칙

- **DB 스키마 추가/마이그레이션 없음.**
- 기존 `contracts.options` JSON 필드와 `contract_json` 확장 필드를 적극 활용.
- 협상 단계(offer 검증) + 커밋 단계(최종 안전장치) **이중 검증** 적용.
- 금액/룰 계산은 정책 모듈로 분리하여 테스트 가능하게 유지.
- 날짜 윈도우는 상대 표현이 아닌 절대 날짜 경계(시작 포함/종료 제외)로 판정한다.
- 연봉 증감 검증은 **모든 계약 타입 공통 SSOT**로 관리한다(복리 금지, 1년차 anchor 고정폭).

### 4-4. 신규 파일 `contracts/policy/extension_rules.py` (추가)

#### 목적
연장 자격/기간/연봉 규칙 SSOT.

#### 구현 함수 (정확히 이 시그니처로 생성)

1) `infer_extension_type(player_ctx: Mapping[str, Any], contract_ctx: Mapping[str, Any], request_type: str | None) -> str`
- request_type 우선, 없으면 조건 기반 추론.
- 반환값: `ROOKIE | VETERAN | DVE`.

2) `validate_extension_eligibility(*, extension_type: str, player_ctx: Mapping[str, Any], contract_ctx: Mapping[str, Any], league_ctx: Mapping[str, Any], now_date_iso: str) -> None`
- 실패 시 `ContractNegotiationError` raise.
- 검증 항목:
  - RSE: 1라운드 루키 계약 여부, 4년차 옵션 행사 여부, 허용 윈도우
  - VE: 계약연차별 서명 경과기간 조건, 1~2년 계약 제외
  - DVE: 경력(7~8), 잔여(1~2), 소속팀/트레이드 예외, Higher Max Criteria

3) `max_total_seasons_at_sign(extension_type: str) -> int`
- RSE/DVE=6, VE=5

4) `calc_extension_first_year_limit(*, extension_type: str, salary_cap: float, prev_salary: float, eaps: float, exp: int, dve_min_pct: float = 0.30, dve_max_pct: float = 0.35) -> dict[str, float]`
- 반환 예시:
  - `{ "min_first_year": ..., "max_first_year": ..., "rule_meta": {...} }`
- VE에서 140% 룰 + max-tier + 105% 비교 반영.

5) `validate_fixed_raise_curve(*, salary_by_year: Mapping[int|str, float], first_year_salary: float, max_delta_pct: float = 0.08) -> list[dict[str, Any]]`
- 전년도 기준이 아닌 1년차 기준 고정폭 검증.
- 위반 목록 반환(빈 목록=OK).

6) `validate_extension_total_seasons(*, current_start_year: int, current_years: int, added_years: int, sign_year: int, extension_type: str) -> None`
- 옵션 포함 총기간 한도 검증.

7) `validate_option_decision_window(*, option_type: str, season_year: int, decision_date_iso: str, contract_meta: Mapping[str, Any]) -> None`
- 루키 3/4년차 팀옵션 윈도우 규칙 포함.

#### 주의
- 모든 계산은 pure 함수로 작성.
- DB 접근 금지.

### 4-5. `contracts/policy/raise_limits.py`

#### 변경
- 기존 `validate_salary_raise_curve`를 deprecate 처리하고, 신규 SSOT 함수로 통합:
  - `validate_salary_curve_with_anchor(salary_by_year, anchor_salary, max_delta_pct, *, allow_descend=True)`
- 동작(공통 SSOT):
  - 각 연도 i에 대해 `salary_i <= anchor + i * (anchor * max_delta_pct)`
  - 계약 타입과 무관하게 동일 규칙 적용(FA/RE_SIGN/EXTEND/MLE/MINIMUM 포함)
  - `allow_descend=False` 설정 시 하향도 고정폭 한도 검증 가능(기본은 상향 제한 중심)
- `max_raise_pct_for_contract_channel(...)`는 “채널별 상한 퍼센트”만 반환하고, 실제 커브 검증은 모두 anchor SSOT 경유.

#### 이유
- 복리식 배제와 검증 일관성을 위해, 계약 전 경로에서 하나의 커브 검증기로 통일해야 한다.
