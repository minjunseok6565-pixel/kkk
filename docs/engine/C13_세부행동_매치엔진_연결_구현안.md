# C13 세부행동을 매치엔진에 실제 적용하는 구현안

## 목표
- `docs/engine/C13_역할별_세부행동_선택규칙.md`에 정의한 세부행동을 **저장 → 로드 → 시뮬레이션 적용**까지 일관되게 연결한다.
- 역할 고정(선택 불가) 항목은 서버에서 최종 보정해, UI 우회 입력이 들어와도 엔진 일관성을 보장한다.

---

## 1. 현재 코드 기준 연결 포인트

### 1) 전술 저장/로드 경로
- 전술 API는 프론트 payload를 엔진용 형태로 정규화한다 (`app/api/routes/tactics.py`).
- 현재도 `context` dict를 그대로 엔진에 전달하므로, 세부행동 맵을 `context` 하위에 넣어 전달 가능하다.

### 2) 시뮬레이션 중 컨텍스트 사용 경로
- `TacticsConfig`는 `context` 필드를 기본 제공한다 (`matchengine_v3/tactics.py`).
- possession 단계에서 `ctx`를 통해 action 확률/결과 priors에 영향을 주는 구조가 이미 있다 (`matchengine_v3/sim_possession.py`, `matchengine_v3/builders.py`).
- 리바운드 참여는 이미 `ORB_*` 계열 컨텍스트를 읽을 수 있는 구조가 있다 (`matchengine_v3/participants_parts/participants_rebounds.py`).

### 3) 역할 정보 접근 경로
- pid별 공격 역할 맵은 `rotation_offense_role_by_pid` 또는 tactics context에서 읽는 유틸이 있다 (`matchengine_v3/sim_game.py`).
- 같은 패턴으로 pid별 세부행동 맵도 읽을 수 있게 확장하면 된다.

---

## 2. 데이터 계약(SSOT) 제안

전술 payload(엔진형) 내 `context`에 아래 키를 추가한다.

```json
{
  "context": {
    "PLAYER_BEHAVIOR_BY_PID": {
      "p123": {
        "ON_PULLUP": "HIGH",
        "ON_RIM_ATTACK": "LOW",
        "ON_PNR_BH": "HIGH",
        "ON_POST_TOUCH": "LOW",
        "ON_PASS_FIRST": "MID",
        "onball_isolation": "HIGH",
        "OFF_SPOTUP_SPACE": "MID",
        "OFF_CUT": "LOW",
        "OFF_OFFSCREEN": "MID",
        "OFF_SCREEN_ROLE": "ROLL",
        "offball_crash_off_reb": "HIGH"
      }
    }
  }
}
```

### 값 체계
- 강도형: `LOW | MID | HIGH`
- 유형형: `OFF_SCREEN_ROLE`은 `ROLL | POP | AUTO`

### 저장 위치 선택 이유
- 스키마를 크게 깨지 않고(현재 `tactics: Dict[str, Any]`) 바로 도입 가능.
- 팀 전술 단위로 저장되므로 로테이션/역할 설정과 함께 버전 관리 가능.

---

## 3. 역할 잠금(선택 불가) 서버 강제 로직

문서 규칙을 엔진/서버에서 강제하기 위해, 아래 순서로 보정한다.

1. pid의 역할 확인 (`rotation_offense_role_by_pid` 우선)
2. 역할별 잠금 규칙 조회
3. 유저 입력값이 잠금 규칙과 다르면 잠금값으로 덮어쓰기
4. 최종 보정 결과만 엔진에 전달

권장 모듈:
- `matchengine_v3/player_behavior.py` (신규)
  - `sanitize_player_behavior_by_pid(raw_map, role_by_pid) -> normalized_map`
  - `LOCK_RULES_BY_ROLE` (문서 SSOT 반영)

이렇게 하면 UI에서 disabled 처리 누락/우회가 발생해도 엔진은 항상 동일한 정책으로 동작한다.

---

## 4. 시뮬레이션 적용 설계

핵심은 "행동 옵션 → (액션 선택 확률, 결과 priors, 참여자 선택)"의 3축 반영이다.

## 4-1) 액션 선택 확률 반영 (`build_offense_action_probs` 전후)

`sim_possession.py`에서 공격팀 볼핸들러 pid를 얻은 뒤 해당 pid의 behavior를 읽어 `off_probs`에 multiplier 적용:

- `onball_isolation=HIGH`  → `ISO` 가중(+), `PnR`/`ExtraPass` 소폭(-)
- `ON_PNR_BH=HIGH`         → `PnR`, `PnP` 가중(+)
- `ON_RIM_ATTACK=HIGH`     → `Drive` 가중(+)
- `ON_POST_TOUCH=HIGH`     → `PostUp` 가중(+)
- `ON_PASS_FIRST=HIGH`     → `ExtraPass`, `Kickout` 가중(+), `ISO` 가중(-)

적용 방식:
- 기존 `off_probs` 계산 후 `apply_player_behavior_to_action_probs(off_probs, behavior, game_cfg)` 호출
- 멀티플라이어는 clamp(예: 0.82~1.22)로 제한

## 4-2) 결과 priors 반영 (`build_outcome_priors` 이후)

행동별로 outcome mult를 추가 적용:

- `ON_PULLUP=HIGH`:
  - `SHOT_3_OD`, `SHOT_MID_PU` +
  - `PASS_KICKOUT` -
- `ON_RIM_ATTACK=HIGH`:
  - `SHOT_RIM_*`, `FOUL_DRAW_RIM` +
  - `SHOT_3_OD`, `SHOT_MID_PU` -
- `ON_PASS_FIRST=HIGH`:
  - `PASS_KICKOUT`, `PASS_EXTRA`, `RESET_HUB` +
  - `SHOT_*` 일부 -

적용 위치:
- `sim_possession.py`에서 `outcome_priors = build_outcome_priors(...)` 직후
- `apply_player_behavior_to_outcome_priors(base_action, priors, behavior)` 호출

## 4-3) 참여자 선택 반영 (participants)

### A. 오프볼 컷/오프스크린/스팟업
- `participants_offense.py`의 슈터/림피니셔 후보 weight에 pid별 multiplier 주입:
  - `OFF_CUT`은 `choose_finisher_rim(..., base_action="Cut")` 후보 weight 조정
  - `OFF_OFFSCREEN`은 `choose_shooter_for_three/mid`에서 C&S 후보 weight 조정
  - `OFF_SPOTUP_SPACE`는 SpotUp 계열 outcome에서 슈터 선택 weight 조정

### B. 공격 리바운드 가담
- `participants_rebounds.py`의 `choose_orb_rebounder`에서 기존 score에 pid별 보정:
  - `offball_crash_off_reb=HIGH`: ORB score multiplier +
  - `LOW`: multiplier -
- 팀 단위 `ORB_MULT`와 충돌하지 않게, 개인 보정은 ±15% 내로 제한

### C. 롤/팝 성향
- `OFF_SCREEN_ROLE`을 `PnR/PnP` 계열 액션 선택과 outcome 분기에 반영:
  - `ROLL`이면 `PnR`, `SHOT_RIM_*`, `PASS_SHORTROLL` 쪽 가중
  - `POP`이면 `PnP`, `SHOT_3_CS`, `SHOT_MID_CS` 쪽 가중

## 5. 구체 구현 단계(권장 순서)

### Phase 1 — 저장/검증 파이프
1. `app/api/routes/tactics.py`
   - UI payload ↔ 엔진 payload 변환 시 `PLAYER_BEHAVIOR_BY_PID` 입출력 지원
2. `matchengine_v3/player_behavior.py` 신규
   - 허용 키/값 검증, 기본값 보정, 역할 잠금 강제
3. `matchengine_v3/validation.py`
   - context 내 behavior map shape 검증(비정상 값 drop + warn)

### Phase 2 — 액션/결과 반영
4. `matchengine_v3/sim_possession.py`
   - 현재 볼핸들러 기준 behavior 조회
   - action probs / outcome priors 보정 함수 호출
5. `matchengine_v3/builders.py`(선택)
   - 공용 보정 유틸 위치를 builders로 둘지 player_behavior로 둘지 정리

### Phase 3 — 참여자 선택/리바운드 반영
6. `matchengine_v3/participants_parts/participants_offense.py`
   - 컷/오프스크린/스팟업 관련 후보 weight에 behavior mult 반영
7. `matchengine_v3/participants_parts/participants_rebounds.py`
   - `offball_crash_off_reb` 개인 보정 반영

---

## 6. 튜닝 원칙 (과한 변동 방지)

- 개인 행동 보정은 처음엔 약하게 시작 (권장: `LOW=0.92`, `MID=1.00`, `HIGH=1.08`).
- 액션 확률/결과 priors/참여자 선택 세 축이 동시에 같은 방향으로 작동하므로, 축별 상한을 둔다.
- 역할 잠금 항목은 멀티플라이어를 유저값이 아니라 **고정값**으로 계산한다.

---

## 7. 검증 시나리오

### 단위 테스트
- 역할 잠금 강제 테스트:
  - 예: `Shot_Creator + onball_isolation=LOW 입력` → sanitize 결과 `HIGH`
- 값 검증 테스트:
  - 허용되지 않은 key/value 입력 시 drop 또는 default

### 시뮬레이션 회귀 테스트
- 동일 seed 1,000포제션 비교:
  - `onball_isolation HIGH` 그룹의 `ISO` 점유율 상승 확인
  - `offball_crash_off_reb HIGH` 그룹의 ORB% 상승 확인
  - 잠금 역할은 유저 입력과 무관하게 동일 분포 확인

### 운영 관측 지표
- `player_behavior_applied_count`
- `player_behavior_locked_override_count`
- 행동 옵션별 경기당 이벤트 점유율(ISO%, Cut%, ORB chance 등)

## 8. 리스크와 대응

- 리스크: 개인 행동 옵션이 전술 스킴보다 과도하게 우세해질 수 있음
  - 대응: 개인 보정 상한 clamp + 스킴 가중 우선순위 유지
- 리스크: 로테이션 중 role/pid 변경 시 behavior 참조 누락
  - 대응: possession 시작마다 현재 on-court pid 기준으로 behavior 조회
- 리스크: 문서 규칙과 코드 규칙 드리프트
  - 대응: `LOCK_RULES_BY_ROLE`를 코드 SSOT로 두고, 문서는 코드에서 생성하거나 테스트로 동기화 검증
