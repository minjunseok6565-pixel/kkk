# 수비 프리셋 스킴 계획 검토 보고서

## 1) 결론 요약

요청하신 최종 계획은 **현재 프로젝트 구조와 방향성에 대체로 정합**합니다. 특히 수비 측에서 공격 의사결정에 개입하는 기존 경로(`opp_action_weight_mult`, `opp_outcome_by_action_mult`, `opp_outcome_global_mult`)가 이미 있어, 액션 버짓/액션별 outcome shaping/전역 압박 슬라이더는 무리 없이 수용 가능합니다.

다만, 아래 2가지가 핵심 갭입니다.

1. **Preset_Defense 전용 UI/모달이 아직 없음** (현재는 Preset_Offense만 상세 설정 UI 존재)
2. **"outcome 성공률(weak/tight)"를 동적으로 바꾸는 런타임 경로가 없음** (현재 품질 라벨은 스킴 기반 정적 테이블)

따라서, "지금 바로 구현 가능" 범위와 "엔진 확장 필요" 범위를 분리하면 계획 리스크를 낮출 수 있습니다.

---

## 2) 현재 코드베이스 정합성 체크

## 2-1. 이미 준비된 기반 (정합)

- 수비 스킴 키 `Preset_Defense`는 이미 상수/정규화/검증 경로에 포함되어 있습니다.
- 수비가 상대 공격 분포에 개입하는 데이터 경로가 이미 존재합니다.
  - 액션 분포: `opp_action_weight_mult`
  - outcome 전역: `opp_outcome_global_mult`
  - outcome 액션별: `opp_outcome_by_action_mult`
- 멀티플라이어 검증 범위가 `0.70~1.40`이므로, 계획 수치(0.85~1.15)는 안전하게 수용됩니다.

=> 즉, 이번 설계의 핵심인 "수비 프리셋을 수치 멀티플라이어로 컴파일해 저장/반영" 자체는 현재 엔진 계약과 잘 맞습니다.

## 2-2. 즉시 보완이 필요한 부분 (갭)

- 프론트엔드 전술 화면에는 현재 `Preset_Offense` 전용 버튼/모달/serde/컴파일러만 연결되어 있고, `Preset_Defense` 전용 상세 설정 진입점이 없습니다.
- 전술 저장 시에도 프리셋 컴파일 경로가 공격 전용입니다. 수비 프리셋은 동일한 패턴의 컴파일/주입 경로가 추가되어야 합니다.
- `weak/tight`는 `quality.py`의 정적 스킴 라벨 테이블(`SCHEME_BASE_OUTCOME_LABELS`)을 통해 계산되며, 현재는 전술 context를 받아 동적으로 override하는 인터페이스가 없습니다.

=> 결론적으로, shaping(1.xx/0.xx)은 바로 얹기 좋고, success label 동적 변경은 엔진 확장이 필요합니다.

## 3) 요청안 항목별 실현성 평가

## 3-1. 액션 버짓 + 9개 액션 구성 선택

**실현성: 높음**

- 대상 액션(Cut, DHO, Drive, ISO, PnP, PnR, PostUp, SpotUp, TransitionEarly)은 현재 엔진의 공격 base_action 체계와 일치합니다.
- 컴파일 타깃은 `opp_action_weight_mult`가 적합합니다.
- 권장: 버짓 입력은 내부적으로 정규화(합=100) 후 baseline 대비 비율 멀티플라이어로 환산.

## 3-2. 액션별 2묶음/3선택지 + 억제/강억제 파생 규칙

**실현성: 높음**

- outcome별 shaping 값은 `opp_outcome_by_action_mult[action][outcome]`에 직접 매핑 가능.
- "한 묶음 선택 시 반대 묶음 자동 대응" 규칙은 컴파일러 레벨에서 결정적으로 계산 가능합니다.
- "기본" 상태 1.00/neutral 처리도 컴파일러 기본값으로 충족 가능.

## 3-3. outcome 성공률(weak/tight) 동적 조정

**실현성: 중간 (엔진 확장 필요)**

현재 품질 계산은:

- 스킴별 정적 라벨(`weak/tight/neutral`) 조회
- 수비 역할 스탯 기반 연속 보정

으로 구성되어 있어, 프리셋 드래프트 선택에 따라 특정 outcome 라벨을 동적으로 바꾸려면 아래 중 1개가 필요합니다.

- **권장안 A**: `quality.py`에 `context` 기반 라벨 override 맵 주입 경로 추가
- **대안 B**: weak/tight를 라벨이 아닌 "추가 멀티플라이어"로 근사(엔진 변경 최소, 하지만 의미론적으로는 완전 동일하지 않음)

## 3-4. Global Pressure 슬라이더

**실현성: 매우 높음**

- 요청하신 대상 outcome(TO_HANDLE_LOSS, TO_CHARGE, FOUL_DRAW_*)은 모두 outcome 멀티플라이어 경로에 자연스럽게 매핑됩니다.
- 5단계 샘플(-2~-+2)은 고정 맵으로 바로 사용 가능하고, 연속 슬라이더가 필요하면 선형 보간으로 확장 가능합니다.

## 4) 미정의 액션/미정의 outcome 처리안 (권장 정책)

요청 의도(미정의는 neutral/미지정 처리)와 운영 안정성을 같이 만족하도록 다음 정책을 권장합니다.

## 4-1. 미정의 액션 (`Kickout`, `ExtraPass`)

1. **기본 정책**: 2분할 묶음 정의가 없는 동안은 프리셋 UI에서 "설정 불가(비활성/준비중)"로 표시
2. **컴파일 정책**: 해당 액션 키는 `opp_outcome_by_action_mult`를 생성하지 않음
3. **엔진 결과**: 미지정이므로 기존 스킴/기본 priors만 적용 (사실상 neutral)
4. **추적성**: context에 `UNDEFINED_ACTION_POLICY: "omit"` 같은 메타 스냅샷만 남겨 디버깅 가능하게 유지

## 4-2. 미정의 outcome (언급되지 않았거나 묶음 밖)

1. **기본 정책**: 해당 액션 내 명시 대상 outcome만 멀티플라이어를 기록
2. **나머지 outcome**: 미기록(=1.00으로 동작) 또는 명시 1.00 중 한 방식으로 통일
3. **권장 선택**: 저장 payload는 간결성을 위해 **미기록 방식** 권장
4. **예외 정책**: QA 가시성이 필요하면 디버그 모드에서만 1.00 명시 출력

## 4-3. 상충 규칙 방지

- 하나의 outcome이 "액션별 shaping"과 "전역 압박" 양쪽에서 동시에 조정되면 최종값은 곱연산이 됩니다.
- 따라서 정책 문서에 **우선순위가 아니라 합성 규칙(곱합성)** 을 명확히 선언해야 합니다.

---

## 5) 구현 전 설계 확정 시 체크리스트 (사전 합의용)

1. 액션 버짓 범위/단위(정수 %, step)와 자동 정규화 UX
2. 3선택지 + 비기본 세분화(억제/강억제)의 정확한 상태머신 정의
3. weak/tight 동적 반영 방식(A: quality override / B: multiplier 근사)
4. 미정의 액션 UI 표기(숨김 vs 비활성 + 안내문)
5. 저장 포맷 스냅샷 키 네이밍 (예: `USER_PRESET_DEFENSE_DRAFT_V1`)
6. QA 기준: 동일 입력 -> 동일 payload(결정론), 미정의 항목 무변형 보장

---

## 6) 최종 판단

- **정합성 관점**: 현재 프로젝트 아키텍처와 충돌 없이 수용 가능
- **실무 리스크 포인트**: 성공률 라벨(weak/tight) 동적 반영만 별도 설계 결정 필요
- **권장 진행 방식**:  
  1) 1차: shaping + pressure + 스냅샷/모달 완성  
  2) 2차: weak/tight 동적 override 엔진 확장(또는 근사안 채택)

즉, 요청하신 최종 계획은 "구조적으로 실행 가능"하며, 핵심은 성공률 라벨 반영 방식을 조기에 확정하는 것입니다.
