# 프리셋(백지 전술) 스킴 구현: 사전 작업 vs 유저 선택 분리

목표: 공격/수비에서 `Preset_Offense`, `Preset_Defense` 같은 "빈 캔버스" 스킴을 추가하되,
- 엔진/검증/UI가 깨지지 않도록 **개발자가 미리 고정해야 할 항목**과
- 전술 성향을 실제로 바꾸는 **유저 입력 항목**을 명확히 분리한다.

---

## 0) 핵심 원칙 (중요)

1. **스킴 키(식별자)와 입력 슬롯은 사전 고정**한다.
   - 예: `Preset_Offense`, `Preset_Defense` 키 자체는 코드/테이블에 미리 등록.

2. **실제 성향값(가중치/멀티플라이어/quality label)은 유저 선택값으로 채운다.**
   - 예: 어떤 `base_action`을 높일지, 어떤 `outcome`을 깎을지, 어떤 outcome을 `tight/weak`로 둘지.

3. **유저가 아무것도 선택하지 않아도 최소 동작 가능한 안전 기본값**을 반드시 둔다.
   - 저장/시뮬레이션/훈련 API가 실패하면 안 됨.

---

## 1) 유저 선택 없이 미리 해둘 작업 (개발 고정 영역)

아래는 "프리셋 스킴 기능 자체를 열어두기" 위한 사전 작업이다.

### 1-1. 스킴 키 등록/노출 (UI + 검증 경로 오픈)

1. 프론트 스킴 목록에 프리셋 키 추가
   - `static/js/core/constants/tactics.js`
   - `TACTICS_OFFENSE_SCHEMES`에 `Preset_Offense`
   - `TACTICS_DEFENSE_SCHEMES`에 `Preset_Defense`

2. 수비 프리셋 canonical/alias 등록
   - `matchengine_v3/tactics.py`
   - `_CANON_DEFENSE_SCHEMES`에 `Preset_Defense`
   - `_DEFENSE_SCHEME_ALIAS_NORM`에 `presetdefense`, `프리셋수비` 등 alias 추가

3. 샷다이어트 지원 공격 스킴 등록
   - `matchengine_v3/validation.py`
   - `SHOT_DIET_SUPPORTED_OFFENSE_SCHEMES`에 `Preset_Offense` 추가

### 1-2. "빈 선택"에서도 엔진이 도는 최소 안전값 골격

4. 공격 액션 분포용 기본 뼈대 추가
   - `matchengine_v3/profiles_data/off_scheme_action_weights.py`
   - `OFF_SCHEME_ACTION_WEIGHTS["Preset_Offense"]` 추가
   - 완전 0 대신, 합리적인 균등/완만 분포로 초기값 제공(유저가 덮어쓸 값)

5. 공격 결과 보정 기본 뼈대 추가
   - `matchengine_v3/profiles_data/offense_scheme_mult.py`
   - `OFFENSE_SCHEME_MULT["Preset_Offense"]` 추가
   - 기본값은 대부분 `1.00` 근처(중립)

6. 수비 결과 보정 기본 뼈대 추가
   - `matchengine_v3/profiles_data/defense_scheme_mult.py`
   - `DEFENSE_SCHEME_MULT["Preset_Defense"]` 추가
   - 기본값은 대부분 `1.00` 근처(중립)

7. 수비 도움수비 baseline 추가
   - `matchengine_v3/possession/tactics_ctx.py`
   - `_SCHEME_HELP_BASELINE["Preset_Defense"]` 추가
   - 완전 중립값(기존 스킴 중간 수준)으로 시작

8. 수비 메타 규칙 빈-안전 테이블 추가
   - `matchengine_v3/era.py`
   - `DEFENSE_META_PARAMS["defense_meta_action_mult_tables"]["Preset_Defense"]`
   - `DEFENSE_META_PARAMS["defense_meta_priors_rules"]["Preset_Defense"]`
   - 초깃값은 "보정 최소"로 구성

9. QUALITY 테이블에 프리셋 슬롯 추가
   - `matchengine_v3/quality_data/`
   - `SCHEME_BASE_OUTCOME_LABELS`, `ROLE_STAT_PROFILES`, `GROUP_SCHEME_ROLE_WEIGHTS`, `SCHEME_ALIASES`
   - 라벨 기본값은 `neutral` 중심(편향 최소)

10. 수비 프리셋 롤 구조(5슬롯) 고정
    - `static/js/core/constants/tactics.js`
    - `TACTICS_DEFENSE_ROLE_BY_SCHEME["Preset_Defense"]` 정의
    - 내부적으로는 기존 role key 재사용 권장(새 role 키를 늘리지 않기)

### 1-3. 유저 커스텀 값을 태울 "입력 컨테이너" 계약 확정

11. 저장 페이로드 스키마/직렬화 계약 고정
    - `app/api/routes/tactics.py` 중심으로
    - `offenseScheme/defenseScheme`와 함께
      - 유저 커스텀 action/outcome/quality 선택값을 담을 필드(예: `context.preset_config`)를 스펙화

12. 유효성/범위 제한 규칙 고정
    - multiplier 허용 범위(예: 0.7~1.4), 합계 정규화 정책, 잘못된 key 무시 정책
    - 기존 sanitize 흐름(`matchengine_v3/validation.py`)에 맞춰 확정

13. "초기화/리셋" 동작 정의
    - 유저가 프리셋 스킴 선택 시 기본값 주입
    - 유저가 값을 지우면 안전 기본값으로 복원

### 1-4. QA/운영 안전장치

14. 최소 테스트 선반영
    - `tests/test_tactics_api.py`: 프리셋 스킴 round-trip + 기본값 주입 확인
    - 시뮬레이션 smoke: 프리셋 스킴 + 빈 커스텀 입력에서도 possession 정상 진행

15. 관측 가능성(디버그) 추가
    - 로그/이벤트에 "프리셋 유저 커스텀값 적용 여부"를 남겨 추적 가능하게 함

## 2) 유저 선택으로 남길 작업 (커스텀 전술 영역)

아래는 실제 전술 성향이며, 프리셋 기능의 본체다.

### 2-1. 공격 프리셋에서 유저가 고르는 것

1. `base_action` 빈도(우선순위)
   - 예: `PnR`/`Drive`/`SpotUp` 비중 상향, `PostUp` 비중 하향
   - 반영 위치: 최종적으로 `OFF_SCHEME_ACTION_WEIGHTS` 또는 동등한 runtime overlay

2. action별 `outcome` 성향
   - 예: `PnR`에서 `PASS_SHORTROLL`↑, `SHOT_MID_PU`↓
   - 반영 위치: `OFFENSE_SCHEME_MULT` 또는 동등 overlay

3. 샷다이어트 전술 성향
   - action feature weight, outcome delta, alpha, screener priority
   - 반영 위치: `shot_diet_data` 계열과 대응되는 사용자 값 overlay

4. (선택) 팀 스타일 보정
   - `three_bias`, `rim_bias`, `tempo_mult` 방향

### 2-2. 수비 프리셋에서 유저가 고르는 것

5. 상대 outcome 억제/유도
   - 예: `SHOT_RIM_LAYUP` 억제, 대신 `SHOT_MID_PU` 허용
   - 반영 위치: `DEFENSE_SCHEME_MULT` 또는 동등 overlay

6. 도움수비/로테이션 강도 성향
   - 반영 위치: `_SCHEME_HELP_BASELINE`에 대응되는 사용자 보정값(절대값 or delta)

7. 수비 메타 우선순위
   - action 억제 강도, priors rule(특정 outcome 최소/배율)

8. 품질 라벨 전략
   - 어떤 outcome을 `tight`, `tough`, `weak`, `neutral`로 둘지
   - 반영 위치: `SCHEME_BASE_OUTCOME_LABELS` 대응 사용자 설정

9. 수비 role 강조도
   - 그룹별(role involvement) 가중치 조정
   - 반영 위치: `GROUP_SCHEME_ROLE_WEIGHTS` 대응 사용자 설정

## 3) 경계선 항목: "고정 틀 + 유저 값"으로 나눠야 하는 것

아래는 한쪽으로 몰면 위험한 항목들이다.

1. **품질 시스템 라벨 집합 자체**
   - 고정: 허용 라벨 집합(`weak/tough/tight/neutral`)
   - 유저 선택: 라벨을 outcome에 어떻게 배치할지

2. **role 키 체계**
   - 고정: 사용 가능한 role 키 목록
   - 유저 선택: 각 role의 강조 강도(가중치)

3. **수치 범위/클램프**
   - 고정: 안전 범위
   - 유저 선택: 범위 안의 실제 값

4. **fallback 규칙**
   - 고정: 누락 키 처리(무시/중립값/경고)
   - 유저 선택: 없음

## 4) 구현 권장안 (실무)

프리셋을 "새 고정 테이블"로만 만들면 유저 커스텀이 어려워진다.
권장 구조는 아래와 같다.

1. **고정 Canonical 스킴 키 + 사용자 Overlay 분리**
   - 스킴 키: `Preset_Offense`, `Preset_Defense`
   - 사용자값: `tactics.context.preset_config` 같은 별도 구조체

2. **계산 시점 병합 순서 고정**
   - 기본 테이블(중립) → 유저 overlay → clamp/normalize → 최종 확률

3. **테이블 파일을 실시간 수정하지 않기**
   - `profiles_data/*.py`, `quality_data/*.py`를 유저 저장 시 직접 바꾸지 말고
   - 런타임 병합으로 처리(멀티유저/세이브 간 격리)

## 5) 착수 순서(추천)

1. 스킴 키 등록(UI/검증/canonical)
2. 안전 기본 테이블(중립값) 전부 추가
3. 유저 커스텀 payload 스키마 확정
4. 런타임 병합 로직(overlay) 추가
5. 프론트 편집 UI 연결
6. 테스트/밸런스 튜닝

## 7) 요약

- **미리 해둘 것**: 스킴 키 등록, 안전 기본값, 검증/범위/직렬화 계약, 테스트 안전망.
- **유저에게 맡길 것**: 액션 빈도, outcome 성향, 수비 억제 방향, quality 라벨 전략, role 강조도.
- 핵심은 "코드에 고정된 틀"과 "유저가 바꾸는 값"을 분리해, 백지 전술의 자유도와 시스템 안정성을 동시에 확보하는 것.
