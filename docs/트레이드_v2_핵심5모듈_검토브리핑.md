## 0) 결론 요약

현재 생성된 5개 모듈은 **v2 골격을 빠르게 세운 1차 구현**으로서는 의미가 있다.  
다만, 상세설계에서 강제한 원칙(특히 SSOT 의미론 참조, fallback 규약, dual-read 품질, 단계적 전환) 기준으로 보면 **즉시 운영 전환하기에는 중요한 정합성 갭이 존재**한다.

- **총평:** `부분 적합 (구조 OK / 의미론·전환·검증 미흡)`
- **가장 큰 리스크 축:**
  1) `pick_distribution`의 보호/스왑 의미론 단순화,
  2) `role_texture`의 fallback 규약 위반,
  3) `context_v2` dual-read diff 품질 미달.

---

## 1) 전역 수정 원칙 준수 점검

### 1-1. 무가정(없는 값 생성 금지)

**대체로 준수**하려는 흔적은 있으나, 일부는 설계 의도와 충돌한다.

- 장점
  - 누락 입력 시 coverage flag와 `MISSING_INPUT_*` note를 남기는 패턴이 모듈 전반에 존재.
- 우려
  - `role_texture`는 role_fit이 없을 때도 supply 벡터로 점수를 채워 넣어, 설계서의 “없으면 0.0 + coverage false” 고정 규약과 충돌.
  - `context_v2` dual-read에서 v1 metric을 실제로 읽지 않고 빈 dict를 넣어 delta를 계산해, 의미 없는 수치를 telemetry로 만들 가능성.

### 1-2. 중복 금지(단일 계산 책임)

**중간 수준 준수.**

- 장점
  - `contract_texture`는 `contracts/terms.py`를 직접 사용해 remaining years/salary를 계산(SSOT 준수).
  - `pick_distribution`는 `normalize_protection()`을 사용.
- 우려
  - `pick_distribution`의 스왑 반영은 `pick_settlement.py` 의미론을 참조하지 않고 자체 단순 조합으로 구현되어, “의미론 재정의 금지” 원칙과 충돌 소지.
  - `context_v2`는 시장/패키지 adapter 연동(특히 pick v2 호환 주입·기존 보정 비활성화)까지는 아직 미구현.

### 1-3. 안전 전환(dual-read + feature flag)

**미흡.**

- `dual_read` 스위치는 존재하나, v1 실측 대비 diff가 아니라 사실상 v2 raw aggregate 기반의 의사-delta다.
- 단계적 활성화(pick → role/contract → cap)를 위한 세분 플래그가 없다.
- feature flag off 시 기존 동작 100% 동일성 검증 루프가 코드상으로 아직 보이지 않는다.

## 2) 모듈별 상세 점검

## 2.1 `context_v2.py`

### 잘된 점
- 설계서의 핵심 타입(`ValuationContextV2`, diagnostics, diff report) 구조를 구현.
- Provider 기반 snapshot 수집 구조를 채택했고, repo 직접 접근 코드는 없음.
- 하위 4모듈 조립 오케스트레이션 형태는 설계 방향과 일치.

### 문제/리스크
1. **dual-read diff 품질 문제(중요)**
   - `collect_v1_v2_diff()`는 key 이름이 이미 delta 성격(`pick_ev_delta`)인데, 다시 `v2-v1` 연산을 수행.
   - 호출부에서는 `v1_metrics={}`를 넣어 항상 `MISSING_INPUT_*`를 만들며, 결과 delta가 실질적으로 v2 aggregate가 되어버림.
   - 결과적으로 “v1 대비 변화량” telemetry로 사용하기 어렵다.

2. **단계적 전환 제어 미흡**
   - 단일 `dual_read` bool만 있고, 상세설계의 단계별 활성화 스위치 구조가 없음.

3. **연계 파라미터 미사용**
   - 시그니처로 받은 `current_date_iso`, `team_utility_config`, `package_effects_config`, `decision_policy_config`가 내부에서 사용되지 않음.
   - 초기 골격 단계로는 가능하지만, 향후 “조립 오케스트레이터” 책임 완성도 측면에서는 미완성.

4. **cap_ledger 입력 결합 정확도 리스크**
   - payroll이 player snapshot 합산 기반이라, 실제 팀 payroll 정의(예: dead cap/예외항목 포함 여부)와 괴리 가능성.

## 2.2 `pick_distribution.py`

### 잘된 점
- PMF/CDF/EV/분산/quantile/tail 리스크까지 번들화되어, 단일 기대순번 모델 대비 표현력이 크게 개선.
- PMF 합 검증 및 coverage/note 기록 패턴이 들어가 있음.
- `normalize_protection()` 경유로 보호 입력 정규화를 재사용.

### 문제/리스크
1. **보호 의미론 단순화 과도(중요)**
   - `TOP_N` 보호 시 conveyed outcome만 재정규화하여 “당해년도 전달되는 경우의 조건부 분포”처럼 계산됨.
   - non-convey/롤오버/대체 보상의 상태 전이는 모델링되지 않음(최소한 명시적 제한/비활성 경로 필요).

2. **스왑 의미론 SSOT 참조 부족(중요)**
   - 설계서가 요구한 `pick_settlement.py semantics` 참조 대신, owner 비교 + min/max 조합의 단순 모델.
   - 실제 스왑 행사 조건/보호와 결합될 때 오차 가능성이 큼.

3. **복수 스왑/상호의존 시나리오 안정성 검증 부재**
   - 각 swap을 순차적으로 pmf에 곱하는 방식이라, 상호배타/상호조건 관계를 보장하지 못할 가능성.

4. **v2 이중 보정 방지 연동 미확인**
   - 모듈 자체엔 계산이 있지만, 설계서의 “market_pricing 기존 보정 단계 비활성화”와 연결된 코드는 현재 확인되지 않음.

## 2.3 `role_texture.py`

### 잘된 점
- role_fit 우선순위(`meta.role_fit` → `attrs.role_fit`)는 설계 요구와 일치.
- 필수 4축 + connector index 구조를 구현.
- `fit_engine` 벡터 주입 경로(`injected_supply`)를 제공해 adapter 가능성을 열어둠.

### 문제/리스크
1. **fallback 규약 위반(중요)**
   - 설계서: role_fit이 없으면 축별 0.0 + coverage false.
   - 구현: role_fit이 없어도 supply 벡터가 있으면 점수를 채워 0이 아닐 수 있음.
   - 이는 “없는 값 보수적 처리” 원칙과 충돌.

2. **시그니처 드리프트(경미)**
   - 설계서에 없는 `defense_role_groups` 파라미터가 추가됨.
   - 유용한 확장이지만, 계약 문서와 구현 간 차이를 문서화해야 함.

---

## 2.4 `contract_texture.py`

### 잘된 점
- `contracts/terms.py` 기반으로 remaining years/salary 계산(핵심 SSOT 준수).
- 옵션 방향성(`TEAM/PLAYER/ETO`) 반영 로직 존재.
- salary cap 미입력 시 coverage false + note 처리로 안전장치 구현.

### 문제/리스크
1. **coverage 정의의 의미 혼선 가능성**
   - `rem_years <= 0`를 `MISSING_INPUT_CONTRACT_TERMS`로 기록하는데, 실제로는 입력 누락이 아니라 “계약 만료 상태”일 수 있음.
   - reason code가 운영 telemetry에서 오탐을 만들 수 있다.

2. **질감 지표 정규화 근거 문서화 부족**
   - `matching_utility`, `toxic_risk`, `reversibility` 계수(0.15, 0.25 등)는 초기 휴리스틱으로 보이는데, 설계/튜닝 근거 메모가 없어 후속 튜닝 추적성이 약함.

> 본 모듈은 5개 중 설계 적합도가 상대적으로 높은 편.

---

## 2.5 `cap_ledger.py`

### 잘된 점
- posture enum 검증 + fallback(`STAND_PAT`) 처리 구현.
- cap/apron 수치를 `cap_model`에서 가져와 자체 재구현을 피함.
- `score_cap_flex_delta(before, after, posture)` 인터페이스 제공.

### 문제/리스크
1. **cap_model 입력 coverage 부족**
   - `cap_model.numbers_for_season()` 실패/누락에 대한 방어 및 `MISSING_INPUT_CAP_MODEL` reason 처리가 없음.

2. **posture 가중 적용 방식 해석 리스크**
   - `build_cap_ledgers`에서 이미 posture weight를 곱하고, `score_cap_flex_delta`에서도 posture weight를 다시 곱함.
   - 사용 경로에 따라 posture 효과가 이중 반영될 가능성이 있어 계약(한 번만 반영 vs 단계별 반영) 명확화 필요.

3. **payroll 정의 일관성 리스크**
   - 외부 입력 `team_payroll_map` 품질에 강하게 의존하므로, context 조립단에서 payroll 산식 표준(SSOT)을 명시할 필요.

## 3) 구현구상서(이슈 1/2/3/7/8) 기준 진척도 평가

- **이슈 1 (픽 분포):**
  - PMF/CDF/분산 도입으로 “첫 단추”는 잘 끼워짐.
  - 다만 보호/스왑 상태기계의 의미론 정밀도는 아직 초기 단계.

- **이슈 2 (역할/계약 질감):**
  - 형태는 구현됨(긍정).
  - 역할 fallback 규약 위반과 계수 근거 문서화 부족은 보완 필요.

- **이슈 3 (fit-swap 고도화):**
  - 이번 5모듈 범위 밖이지만, role texture가 향후 lineup/ecosystem fit의 기반이 될 데이터 구조를 제공.
  - 다만 현재 role texture 신뢰도 규약이 흔들리면 이후 모듈 품질에 연쇄 영향.

- **이슈 7 (CAP_FLEX 분리):**
  - 별도 ledger 축 분리는 성공.
  - posture 적용 계약(언제/몇 번 반영)과 cap_model fail-safe는 정리 필요.

- **이슈 8 (패키지 중복 다차원화):**
  - 직접 구현은 아직 아님.
  - role/contract texture가 이후 package_effects 교체의 입력 기반으로는 충분히 유망.

## 4) 우선순위 권고 (수정 전 브리핑용)

### P0 (즉시 정합성 이슈)
1. `context_v2` dual-read를 실제 v1/v2 동일 지표 비교로 교정.
2. `role_texture`에서 role_fit 부재 fallback을 설계 규약대로 고정(0.0 + coverage false).
3. `pick_distribution`의 swap/protection 의미론을 SSOT(`pick_settlement`)와 정렬하거나, 지원 범위를 명시적으로 축소/격리.

### P1 (운영 안정화)
4. 단계별 feature flag 설계(픽→역할/계약→cap) 추가.
5. cap_model/payroll 입력 누락 reason/coverage 강화.
6. posture 가중의 단일 적용 규약 확정(ledger build vs delta score 중 한 축).

### P2 (추적성/튜닝성)
7. contract/cap 계수 근거 문서화(튜닝 로그와 연결).
8. 신규 모듈 단위 테스트(특히 경계 케이스) 보강.

## 5) 최종 평가

- **구조적 출발점:** 좋음 (핵심 5모듈의 틀은 계획과 일치)
- **설계 충실도:** 중간 (핵심 규약 일부 위반/미완)
- **운영 전환 준비도:** 낮음~중간 (dual-read/flag/의미론 정합 보강 필요)

즉, “첫 단추는 끼웠지만, 아직 잠금이 덜 된 상태”로 판단된다.  
바로 대규모 활성화하기보다, 위 P0/P1 항목을 먼저 정리한 뒤 dual-read 데이터를 수집하는 것이 안전하다.
