## 근거 및 변경 후 동작

### 1) 스켈레톤 가짓수 확대 (2-3)

기존 4개 호환 아키타입(compat) 중심 구조 위에, registry 기반으로 player_swap / timeline / salary_cleanup / pick_engineering 도메인이 추가되어 기본 스켈레톤 카탈로그가 크게 확장되었습니다.

- 기본 registry spec 수: **27개**
- STAR/HIGH_STARTER 라우트에서 role·pick-only 대비 고가치 딜용 스켈레톤(예: `three_for_one_upgrade`, `star_lateral_plus_delta`, `bluechip_plus_first_plus_swap`)이 활성화됨.
- BUY/SSELL 모두 tier 기반 라우팅을 타며, 같은 아키타입 내에서도 서로 다른 거래 형태를 별도 스켈레톤으로 생성함.

**변경 후 동작 요약**
- 고가 자산 타깃에서도 `p4p_salary`/`young_plus_pick` 1~2종에 수렴하지 않고, 다자산 패키지/타임라인 교환/픽 엔지니어링 조합까지 초기 생성 단계에서 후보로 올라옵니다.
- 따라서 “대형딜 모양을 만들 수 있는 시작점” 자체가 증가해, 평가 단계까지 도달할 수 있는 유효 후보의 다양성이 개선됩니다.

### 2) 딜 shape 상한 완화 (2-5)

shape 검증(`_shape_ok`)은 여전히 하드 컷이지만, 컷 기준 자체가 대형딜 허용 방향으로 완화되었습니다.

- `max_assets_per_side = 9`
- `max_players_per_side = 4`
- `max_players_moved_total = 7`
- `max_picks_per_side = 4`

또한 player_swap 빌더에 `three_for_one_upgrade`가 포함되어, 실제 3-for-1 패키지를 생성하도록 구현되어 있습니다.

**변경 후 동작 요약**
- 과거처럼 2-for-1/2-for-2 범위에서 조기 탈락하지 않고, 3-for-1·3-for-2 성격의 패키지가 skeleton 단계에서 통과 가능합니다.
- 즉, “고가치 자산 거래에 필요한 선수 수/자산 수”가 shape 검증에서 즉시 잘리는 비율이 감소합니다.

## 검토 결론

현재 브랜치 기준으로 질문하신 두 이슈(스켈레톤 가짓수, 딜 상한)는 **모두 해결된 상태로 판단**됩니다.

다만 체감 오퍼량(실제 accepted deal 증가)은 스켈레톤 이후의 valuation/decision/counter 정책에도 영향을 받으므로, 다음 사이클에서 `unique_skeleton_count`, `target_tier_counts`, 고가 타깃의 valid->accepted 전환율을 함께 모니터링하는 것을 권장합니다.
