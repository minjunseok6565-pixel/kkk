# 팀 훈련 화면 UI 실행 기획안 (상업용 게임 품질 목표)

## 0) 문서 목적
- 목적: 현재 `팀 훈련` 화면을 **출시 가능한 상업용 게임 UI 품질**로 끌어올리기 위한 실제 구현 지시서.
- 범위: **`static/NBA.html`, `static/NBA.css`, `static/NBA.js`만 수정**.
- 화면 범위: `#training-screen` 한정. 다른 화면/기능은 비침범.
- 데이터 제약: **새 API 생성 금지**. 기존 조회 API 응답 + 프론트 파생 계산만 사용.

---

## 1) 목표 UX 정의

### 1-1. UX 목표
1. 캘린더 기반 일정 편집을 1~2클릭으로 끝낼 것.
2. “오늘 어떤 훈련을 넣어야 하는지”를 화면이 먼저 알려줄 것.
3. 프로 게임 UI처럼 보이는 밀도/리듬/반응성을 제공할 것.

### 1-2. 품질 목표 (Done 기준)
- 정보 위계가 명확하고, 빈 여백이 아닌 “의미 있는 정보”로 채워져 있음.
- 캘린더 셀 상태(경기/과거/선택/자동/수동)가 시각적으로 즉시 구분됨.
- 훈련 타입 적용 후 피드백(성공/실패/변경 반영)이 즉시 인지됨.
- 기존 API 호출만으로 주간 요약/위험 배지/추천 문구가 동작함.

---

## 2) 현재 데이터 계약(변경 없음)

## 2-1. 현재 이미 사용하는 API
1. `GET /api/team-schedule/{team_id}`
   - 경기일/상대팀 매핑.
2. `GET /api/practice/team/{team_id}/sessions?date_from=&date_to=`
   - 기간별 저장된 팀 훈련 세션.
3. `GET /api/practice/team/{team_id}/session?date_iso=`
   - 단일 날짜 자동 추천/기본 세션.
4. `GET /api/team-detail/{team_id}`
   - 로스터(청백전 참여자 기본값/선수 표시용).
5. `GET /api/readiness/team/{team_id}/familiarity?scheme_type=offense|defense`
   - 전술 숙련도.
6. `POST /api/practice/team/{team_id}/preview`
   - 선택 세션 효과 프리뷰.
7. `POST /api/practice/team/session/set`
   - 훈련 세션 저장.

## 2-2. API 추가 없이 파생 가능한 정보
- 이번 7일 훈련/경기/휴식 카운트.
- 다음 경기까지 D-day.
- 연전 여부(전후 날짜 경기 존재로 계산).
- 훈련 편중도(OFF/DEF/FILM/SCRIMMAGE/RECOVERY/REST 비율).
- 선택 날짜 묶음 공통 위험 경고(예: 경기 전날 고강도 훈련).

## 3) 화면 IA(정보 구조) 재설계

현재 단일 컬럼 구조를 아래 3영역으로 재구성한다.

### 3-1. A영역: 상단 요약 스트립 (신규)
- 위치: 훈련 화면 타이틀 하단, 캘린더 상단.
- 구성:
  - `기간`: 예) `10/13 - 11/09`
  - `이번 주`: 훈련 N / 경기 N / 휴식 N
  - `다음 경기`: `vs MIL · D-2`
  - `주의 배지`: `연전`, `휴식 부족`, `공격 편중`
- 목적: 사용자가 달력 읽기 전에 상황을 먼저 파악.

### 3-2. B영역: 메인 캘린더 (기존 강화)
- 4주 그리드 유지(7x4).
- 날짜 카드에 다음 요소를 고정 배치:
  1) 좌상단: 날짜 `M/D`
  2) 우상단: 상태 아이콘(경기/훈련/휴식)
  3) 본문: `vs XXX` 또는 `AUTO · 수비`
  4) 하단: 위험 점(연전/경기 전날/경기 다음날)
- 선택 가능 규칙 유지:
  - 과거일/경기일 비활성.

### 3-3. C영역: 우측 상세 패널 (신규/상시 표시)
- 기본: “날짜 선택 안내” 빈 상태.
- 날짜 선택 시:
  - 선택 날짜 목록(최대 N개, +more 처리)
  - 현재 세션 타입/자동·수동 여부
  - 전후 일정 컨텍스트(전날/다음날 경기 여부)
  - 추천 문구(룰 기반)
- 훈련 타입 버튼 클릭 시:
  - 기존 세부 설정 + 프리뷰를 이 패널에서 일관 노출.

## 4) 컴포넌트 상세 스펙

## 4-1. Calendar Day Card 상태 체계
- `is-past`: 비활성 톤, 포인터 없음.
- `is-game`: 경기 강조 톤 + 상대팀 텍스트 가독성 강화.
- `is-selected`: 외곽 글로우/보더 + 체크 마크.
- `is-auto`: 보조 배지 `AUTO`.
- `is-user-set`: 배지 `수동`.
- `is-risk-high`: 하단 위험 점 빨강.

> 상태 클래스는 중첩 가능(예: `is-selected is-auto`).

## 4-2. 상단 요약 카드(신규)
- 컴포넌트 4개:
  - `주간 훈련량`
  - `주간 경기 수`
  - `휴식 확보일`
  - `다음 경기 D-day`
- 숫자는 대형, 설명은 소형 캡션.

## 4-3. 훈련 타입 액션 바(기존 개선)
- 현재 버튼 나열 유지하되 다음 개선 적용:
  - 아이콘 + 단축 라벨(공격/수비/필름/청백전/회복/없음)
  - hover/active/disabled 상태 분리
  - 선택 날짜가 없으면 전체 disabled + 툴팁 문구

## 4-4. 상세 패널 컨텐츠 블록
1. `선택 날짜`
2. `추천 훈련`(룰 기반 텍스트)
3. `세션 설정`(공/수 스킴, 청백전 pid 입력)
4. `프리뷰`(익숙도 gain, 샤프니스 평균 변화)
5. `적용 버튼`

## 5) 룩앤필 가이드 (프로덕션 퀄리티)

## 5-1. 톤
- 배경: 차가운 다크-네이비 그라디언트(너무 검지 않게).
- 카드: 저채도 유리질감(미세 보더 + 미세 그림자).
- 포인트: 팀 컬러 1개 + 경고색 1개만.

## 5-2. 타이포 계층
- H1: 섹션 타이틀(굵게)
- H2: 블록 타이틀
- Body: 기본 설명
- Caption: 날짜/보조정보
- 숫자: tabular-nums 적용(정렬 품질 상승)

## 5-3. 공간 시스템
- 4/8pt 그리드 고정.
- 패널 간격/버튼 높이/라운드 값 토큰화.
- 모든 컴포넌트의 border-radius, border-opacity 통일.

## 6) 인터랙션/모션 스펙

## 6-1. 필수 반응
- 날짜 hover: 배경 밝기 + 보더 강조 (100~140ms).
- 날짜 선택: 선택 링 등장 (120~160ms).
- 타입 적용 성공: 토스트 + 해당 날짜 카드 상태 즉시 갱신.

## 6-2. 모션 원칙
- 과장 금지. 짧고 명확한 easing 사용.
- 캘린더 전체 재렌더 시 깜빡임 최소화(부분 업데이트 우선).

## 7) 프론트 구현 설계 (파일별)

## 7-1. `static/NBA.html` 변경 계획
- `#training-screen` 내부만 수정.
- 추가 구조:
  - 상단 요약 스트립 컨테이너(`training-summary-strip`)
  - 메인 2열 레이아웃 래퍼(`training-layout`)
    - 좌측: 기존 캘린더/훈련 버튼
    - 우측: 상세/추천 패널
- 기존 ID 유지 원칙:
  - `training-calendar-grid`, `training-type-buttons`, `training-detail-panel`는 유지.
  - JS 바인딩 깨지지 않게 신규 래퍼만 추가.

## 7-2. `static/NBA.css` 변경 계획
- `#training-screen` 범위 스코프로만 스타일 작성.
- 신규 클래스:
  - `training-summary-strip`, `training-kpi-card`, `training-risk-badge`
  - `training-layout`, `training-main-col`, `training-side-col`
  - `training-day-cell` 상태 클래스 확장
- 반응형:
  - 1366px 이상: 2열
  - 1365px 이하: 단일열(상세 패널 하단 이동)

## 7-3. `static/NBA.js` 변경 계획
- 기존 함수 확장 중심, 함수명 최대 유지.
- 추가 유틸:
  - `buildTrainingDerivedMetrics()`
  - `buildTrainingRiskFlags(iso)`
  - `buildTrainingRecommendation(selectedDates, type)`
- 렌더 함수:
  - `renderTrainingSummaryStrip()` 신규
  - `renderTrainingCalendar()` 상태 클래스/아이콘/배지 강화
  - `renderTrainingDetail()`에 컨텍스트/추천 블록 추가

## 8) 추천 문구 룰셋 (서버 없이 프론트 계산)

## 8-1. 룰 우선순위
1. 내일 경기 + 오늘 고강도(청백전/전술) 선택 시: `회복/필름 우선 권장`
2. 3일 내 2경기 이상 + 회복 0회: `회복 세션 1회 확보 권장`
3. 최근 7일 공격 편중(OFF_TACTICS >= 60%): `수비/필름 균형 권장`
4. 선택 날짜가 경기 다음날: `RECOVERY 권장`

## 8-2. 출력 포맷
- `추천`: 한 줄 핵심 문장
- `근거`: 작은 글씨로 규칙 설명

예시)
- 추천: `내일 경기 예정: 오늘은 필름/회복 중심이 효율적입니다.`
- 근거: `경기 전날 고강도 훈련 시 피로 리스크 증가(일정 기반).`

## 9) 접근성/완성도 체크리스트
- 키보드 탐색: 날짜 셀/훈련 버튼/적용 버튼 포커스 링 명확.
- 명암비: 텍스트 4.5:1 이상.
- 색상만으로 상태 전달 금지(아이콘/텍스트 병행).
- 로딩 상태: 스켈레톤 또는 로딩 문구 유지.
- 빈 상태 문구: “무엇을 해야 하는지” 행동 유도형.

## 10) 구현 단계(실행 순서)

## Phase 1 — 구조/디자인 토대
1. HTML 레이아웃 래핑 및 요약 스트립 슬롯 추가
2. CSS 토큰/상태 스타일/레이아웃 구축
3. 기존 동작 회귀 확인

## Phase 2 — 데이터 파생/정보 강화
1. 주간 집계/다음 경기 D-day 계산
2. 캘린더 위험 배지(연전/전후 경기) 계산
3. 상세 패널 컨텍스트 정보 렌더

## Phase 3 — 추천/피드백 폴리싱
1. 추천 룰셋 적용
2. 적용 성공 토스트/상태 갱신 UX 개선
3. 모션/hover/focus 디테일 마감

## 11) 수용 기준 (QA Acceptance)
1. 훈련 화면 외 다른 스크린 DOM/CSS/JS 동작 변화가 없어야 한다.
2. 기존 API 외 호출이 발생하지 않아야 한다.
3. 날짜 선택 → 훈련 타입 선택 → 적용 플로우가 기존 대비 단계 수 증가 없이 동작해야 한다.
4. 상단 요약/추천 문구/위험 배지가 일정 데이터와 모순 없이 표시되어야 한다.
5. 1920x1080 기준 정보가 과밀/과소 없이 균형 있게 보여야 한다.
