작성일: 2026-03-10  
기준 문서: `docs/트레이드_잔여_단순화_확정_리포트.md`

## 0) 목적과 제약

본 문서는 아래 5개 확정 이슈를 **신규 시스템 관점**으로 재설계하기 위한 구현 구상이다.

- (1) 픽 기대값
- (2) 패키지 효과의 역할/계약 분류
- (3) fit-swap 적합도
- (7) CAP_FLEX
- (8) 패키지 중복

핵심 제약:

1. 기존 시스템을 완전히 개편하되, **기존 데이터 소스가 내려주지 않는 값은 가정하지 않는다**.
2. 신규 API/모듈 추가는 허용하되, **기존 로직/API와 중복 구현 금지**(SSOT 유지).
3. 레거시는 호환 안전장치(dual-read/feature flag) 이후 단계적으로 삭제한다.
4. 구현 단위는 “모델 개선”이 아니라 “운영 가능한 파이프라인(계산/검증/관측/전환)”까지 포함한다.

## 1) 공통 아키텍처: `Trade Context v2`로 단일화

### 1-1. 왜 먼저 공통 레이어가 필요한가

이슈 1/2/3/7/8은 서로 강하게 결합된다.
- 픽 분포(1)는 package 평가(2,7)와 결합되어야 하며,
- 역할 중복(8)은 fit-swap(3)의 라인업 맥락과 동일 데이터에서 계산되어야 한다.

따라서 기능별로 따로 패치하지 않고, `Trade Context v2`를 먼저 만든다.

### 1-2. 신규 모듈(안)

- `trades/valuation/context_v2.py`
  - 기존 `data_context.py`를 대체/흡수하는 오케스트레이션 레이어.
- `trades/valuation/pick_distribution.py`
  - standings 기반 단일 기대값 대신 분포형 픽 자산 생성.
- `trades/valuation/contract_texture.py`
  - 옵션/보장/통제권/유동성 중심의 계약 질감 계산.
- `trades/valuation/role_texture.py`
  - 단일 archetype이 아닌 다차원 역할 벡터 계산.
- `trades/valuation/lineup_ecosystem.py`
  - 코어/주전/벤치 맥락 적합도 및 대체·충돌 점수 계산.
- `trades/valuation/cap_ledger.py`
  - CAP_FLEX를 재무 운영축으로 분리 평가.

> 중복 방지 원칙: 기존 `package_effects.py` 내부 helper를 재구현하지 않고, 기존 입력 파싱/기초 계산 함수는 재사용하고 점진 이관한다.

### 1-3. 전환 전략

- Step A: `Trade Context v2` 생성 + 결과를 telemetry로만 기록 (결정 미반영)
- Step B: 읽기 경로 dual-read (`v1`, `v2`) + 차이 리포트
- Step C: 전략 플래그별 점진 활성화 (픽→역할/계약→fit/cap)
- Step D: 안정화 후 레거시 경로 삭제

예상 작업량: **5~7 MD**

## 2) 이슈 1 구현안: 픽 기대값 → 분포/시나리오 자산

### 2-1. 목표 모델

단일 `expected_pick_number`를 폐기하고 아래 형태로 표준화:

- `PickOutcomeDistribution` (PMF/CDF)
  - 예: `{1:0.14, 2:0.134, ...}` 형태
- `PickScenarioEnvelope`
  - best/base/worst percentile (p10/p50/p90)
- `ProtectedPickResolver`
  - 보호 발동, 미발동, 롤오버, 보상픽, 소멸 조건을 상태 기계로 계산
- `SwapRightEvaluator`
  - 스왑 행사 시점의 상대 분포 비교 기반 기대 이득 계산

### 2-2. 실제 데이터 제약 반영

- standings/record 기반 순위 정보가 이미 있으면 이를 seed로 사용
- 로터리 odds 표는 시즌 룰 테이블로 내장(상수 모듈)
- 보호/스왑 문구는 **이미 시스템이 보유한 pick rule 파싱 결과만 사용**
- 없는 값(예: 외부 예측모델 승률)은 추가 가정 금지

### 2-3. 기존 코드 연결

- `build_pick_expectations_from_standings()` 호출 지점을 `build_pick_distributions_from_standings()`로 교체
- `market_pricing.py`는 단일 기대순번 입력 대신
  - `ev`(expected value)
  - `variance`
  - `tail_upside`
  - `tail_downside`
  를 함께 반영

### 2-4. 삭제 대상 레거시

- standings→1..30 선형 매핑 helper
- 픽 가치 계산에서 단일 기대순번 강제 분기

### 2-5. 예상 작업량

- 모델/룰 테이블 구축: 3~4 MD
- 보호/스왑 상태기계: 2~3 MD
- 기존 valuation 연결/회귀 검증: 2~3 MD

합계: **7~10 MD**

## 3) 이슈 2 구현안: 단일 archetype·salary×years → 역할 질감 + 계약 질감

### 3-1. 역할 질감(`Role Texture`)

선수 1인당 단일 태그 대신 다차원 벡터를 사용:

- 온볼 생성
- 오프볼 슈팅 중력
- 림 압박/마무리
- 스크린/허브
- POA 수비
- 헬프/림보호
- 멀티포지션 대응
- 세컨드 유닛 자립도

출처는 기존 role_fit/supply 계열 신호를 재매핑하고, 없는 축은 0 또는 보수적 기본치로 둔다.

### 3-2. 계약 질감(`Contract Texture`)

`salary * years` 단일값 대신 아래 분리 지표를 표준화:

- `guaranteed_commitment` (보장금)
- `control_direction` (team/player option 방향)
- `reversibility` (되팔기/만기 활용 용이성)
- `trigger_risk` (옵션 행사/부분보장 데드라인 이벤트)
- `matching_utility` (트레이드 매칭 실무 유용성)

### 3-3. `package_effects` 적용 방식

- 기존 total score에 혼합하던 구조에서
  - `basketball_component`
  - `contract_component`
  를 분리 계산 후 합성
- 결과 노출도 분리(왜 싫어하는지 설명 가능)

### 3-4. 삭제 대상 레거시

- `_primary_archetype_tag()` 의사결정 경로
- `_commitment_metric()` 단독 의존 경로

### 3-5. 예상 작업량

- role 벡터화 + 정규화: 3~4 MD
- contract texture 모델링: 3~4 MD
- package_effects 리팩터 + 설명 필드: 2~3 MD

합계: **8~11 MD**

## 4) 이슈 3 구현안: need tag overlap → 라인업 생태계 fit

### 4-1. 핵심 구조

`lineup_ecosystem.py`에서 팀 맥락 기반 점수를 계산:

- `complementarity_gain`: 기존 코어를 강화하는가
- `redundancy_conflict`: 이미 과밀한 기능과 충돌하는가
- `star_synergy`: 핵심 스타의 효율을 높이는가
- `touches_friction`: 점유/분배 충돌 리스크
- `coverage_resilience`: 부상/벤치까지 포함한 운영 안정성

### 4-2. 계산 단위

- 단일 선수 vs 팀이 아니라,
  - 예상 주전 5인
  - 클로징 5인
  - 세컨드 유닛 핵심 조합
  에 대해 시뮬레이션 점수를 내고 가중 평균

### 4-3. fit-swap 연결

- `_need_fit_score()`를 폐기하고 `compute_ecosystem_fit_score()`로 대체
- counter 제안 시 “플러스/마이너스 이유”를 구조화된 reason code로 반환

### 4-4. 예상 작업량

- 라인업 샘플러/조합 생성: 2~3 MD
- ecosystem scoring 함수: 3~4 MD
- fit-swap/counter 연동 + 튜닝: 2~3 MD

합계: **7~10 MD**

## 5) 이슈 7 구현안: CAP_FLEX 선형 근사 → 밸런스시트·옵셔널리티 평가

### 5-1. 축 분리 원칙

CAP_FLEX는 talent 보조항이 아니라 별도 운영축으로 분리:

- `current_operating_room` (당해 운영 여지)
- `future_locked_money` (미래 보장금 락인)
- `optionality_profile` (누가 선택권 보유)
- `liquidity_profile` (만기/부분보장/비보장 기반 유동성)
- `apron_pressure` (apron 근접 시 거래 제약 리스크)
- `extension_pressure` (향후 연장 의사결정 압박)

### 5-2. 적용 방식

- 거래 점수 최종 합성 시 `CAP_FLEX`를 별도 차원으로 보유
- 팀 상태(컨텐더/리빌더/중간)에 따라 같은 계약의 CAP_FLEX 해석 가중치만 다르게 적용
- 즉 동일 계약도 팀 문맥별로 “매칭용/독성/픽 받고 흡수 가능” 차등 반영

### 5-3. 레거시 제거

- `_cap_flex_adjustment()`의 `_commitment_metric()` 직접 합산 경로 제거

### 5-4. 예상 작업량

- cap_ledger 지표 계산기: 3~4 MD
- 팀 포스처별 가중 적용 로직: 2~3 MD
- 회귀/튜닝 및 설명 필드: 1~2 MD

합계: **6~9 MD**

## 6) 이슈 8 구현안: 단일 버킷 중복 → 다차원 중복 + 로테이션 맥락

### 6-1. 다차원 중복 모델

중복/보완/연결재를 분리:

- `functional_overlap` (기능 중복)
- `rotation_overlap` (동시 기용 충돌)
- `playoff_portability` (플옵 로테이션 적합)
- `injury_insurance_value` (부상 보험 가치)
- `connector_bonus` (하이브리드 연결재 보너스)

### 6-2. 팀 단계별 정책

- 컨텐더: rotation 충돌/플옵 휴대성 가중치 상향
- 리빌더: 중복 페널티 완화, 재능 축적 허용

### 6-3. package_effects 반영

- 기존 diminishing returns를 단일 archetype 버킷에서 제거
- 다차원 score matrix로 교체
- 2-for-1/3-for-1에서 “평평한 결과”를 줄이기 위해 조합 시너지 항 추가

### 6-4. 예상 작업량

- 중복 matrix 정의/계산: 3~4 MD
- 로테이션 맥락 결합: 2~3 MD
- 기존 diminishing 로직 제거/이관: 1~2 MD

합계: **6~9 MD**

## 8) 검증/운영 계획

### 8-1. 정합성 검증

- 동일 입력 상태에서 `v1`/`v2` 산출 diff를 저장
- 승인 기준:
  - top-N 제안 다양성 증가
  - 보호픽/스왑 시나리오 반영 일관성
  - 계약 옵션 구조에 따른 반응 차이 가시화

### 8-2. 회귀 안정성

- 기존 API 응답 스키마는 유지하되, 신규 설명 필드는 optional로 추가
- feature flag off 시 기존 동작 100% 동일 보장

### 8-3. 삭제 기준

아래 조건 만족 시 레거시 삭제:

1. 최소 1 릴리즈 dual-read 무중단
2. 주요 리그 시나리오(탱킹/컨텐딩/데드라인) 리플레이 통과
3. 운영 telemetry에서 v2 오류율/이상치 임계치 이하

## 10) 최종 결론

이번 개편은 “정확한 숫자 하나”를 맞추는 작업이 아니라,
**불확실성과 맥락을 자산 구조로 모델링하는 시스템 전환**이다.

따라서 구현 우선순위는
- (1) 공통 컨텍스트 단일화,
- (2) 분포형 픽,
- (3) 역할/계약/중복 다차원화,
- (4) 라인업 생태계 fit,
- (5) CAP 밸런스시트 분리
순으로 고정하는 것이 가장 안전하다.
