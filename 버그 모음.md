1) MINIMUM 서명 타입 불일치
MINIMUM 경로는 트랜잭션 type/action_type을 SIGN_FA_MINIMUM으로 기록합니다.

그런데 룰 메타 수집의 계약 타입 allowlist에는 SIGN_FA_MINIMUM이 없습니다. (SIGN_FREE_AGENT, RE_SIGN, SIGN_TWO_WAY, RELEASE_TO_FA 등만 포함)

최근 서명 밴 계산은 last_contract_action_type in {SIGN_FREE_AGENT, RE_SIGN, SIGN_TWO_WAY} 또는 signed_via_free_agency=True일 때만 적용됩니다.

따라서 MINIMUM 서명이 메타에 안 잡히면 밴이 약해질 수 있다는 지적은 맞습니다.

2) cap hold 삽입 시점 vs room flag 계산 시점
시즌 전환 중 선수 이동(_move_player_team_in_cur) 때 room-flag를 즉시 계산합니다.

이 계산은 _compute_team_payroll_for_season_in_cur(순수 payroll) 기반입니다.

cap hold는 expire_contracts_for_season_transition 이후 process_offseason에서 나중에 upsert됩니다.

ROOM_MLE 자격은 team_room_mle_flags를 직접 참조합니다.

즉 “홀드 반영 전 아래로 내려갔다고 찍혀서 ROOM_MLE가 잘못 열릴 수 있다”는 우려가 타당합니다.

3) cap hold 해제가 re_sign에만 있음
re_sign에서는 release_cap_hold(...)를 호출합니다.

반면 sign_free_agent/_sign_free_agent_minimum에서는 hold 해제 호출이 없습니다.

cap 체크는 hold를 합산하는 함수(_compute_team_cap_salary_with_holds_in_cur)를 사용합니다.

그래서 기존 팀 hold가 남아 이후 cap 계산을 부풀릴 수 있다는 지적은 맞습니다.

4) _channel_years_cap의 snapshot 참조 위치
_channel_years_cap는 session["trade_rules_snapshot"](top-level)만 읽습니다.

start_contract_negotiation은 snapshot을 constraints에도 넣고, top-level에도 넣습니다.

세션 저장은 dict 전체 deepcopy로 이뤄져 top-level 필드를 버리지 않습니다.

따라서 “top-level이 저장되지 않아 항상 무시된다”는 표현은 현재 코드 기준으로 부정확합니다. 다만, engine이 constraints fallback 없이 top-level만 보는 구조는 취약점이 맞습니다.

5) raise validator snapshot 참조 위치
raise validator도 top-level session["trade_rules_snapshot"]만 읽습니다.

반면 서비스 계층의 채널 정책 검증은 constraints.trade_rules_snapshot을 읽습니다.

즉 참조 일관성이 깨져 있어 흐름에 따라 override 가시성이 달라질 수 있다는 문제 제기는 타당합니다.

다만 4번과 동일하게 “저장이 안 된다” 자체는 현재 저장 코드와 맞지 않습니다.

6) 제안된 dead-cap unique index
현재 실제 스키마 unique는 (team_id, player_id, origin_contract_id, source_type, applied_season_year)입니다.

저장/upsert도 동일 키를 사용합니다.

그래서 질문의 새 인덱스(team_id, player_id, season_year, mode)는 현재 코드엔 아직 적용되어 있지 않음이 먼저입니다.

다만 그 제안이 도입된다면 재계약 후 재방출 같은 정상 시나리오에서 과도 충돌을 낼 수 있다는 우려는 설계적으로 타당합니다.

7) release 타입 변경 시 downstream 누락
move-event 수집은 release 타입을 RELEASE_TO_FA/release_to_free_agency 중심으로 필터링합니다.

rule-player-meta의 계약 타입도 RELEASE_TO_FA/legacy lowercase 위주입니다.

따라서 RELEASE_WAIVE/RELEASE_STRETCH만 새로 쓰면 누락될 수 있다는 지적은 맞습니다.

8) RE_SIGN_OR_EXTEND 누락
현재 CONTRACT_TYPES에 RE_SIGN_OR_EXTEND(대문자)는 없고 lowercase re_sign_or_extend만 있습니다.

get_transactions_for_players는 SQL에서 tx_type IN (...) exact 매칭을 씁니다.

그래서 과거에 대문자 타입이 저장되어 있다면 누락될 수 있다는 지적은 유효합니다.

9) raw action/type 저장 + 정규화 미흡
메타 빌더는 action_type/type 값을 그대로 넣고(정규화 없음),

signed_via_free_agency도 last_action_type == "SIGN_FREE_AGENT" 정확 일치만 봅니다.

최근 서명 밴도 canonical uppercase 집합만 인정합니다.

따라서 legacy lowercase가 safety net 역할을 못 할 수 있다는 지적이 맞습니다.

10) 선택 직후 save 버튼 상태
선택 변경 시 playerTrainingSelectedPlayerId만 바꾸고 로딩 문구를 띄우지만, 즉시 refreshPlayerTrainingSaveUi()를 호출하지 않습니다.

save 버튼 활성 판단은 현재 선택 pid + dirty 상태 기반입니다.

즉 느린 응답 중 구 draft 상태가 남아 잘못 저장되는 레이스가 생길 수 있다는 지적은 타당합니다.

11) 응답 순서 역전(race) 보호 없음
await fetchPlayerTrainingPlan(pid) 후 바로 공용 draft/UI를 갱신합니다.

이 사이에 선택이 바뀌었는지 검증(selectedPlayerId === pid)이 없습니다.

그래서 A→B 빠른 클릭에서 A 응답이 늦게 와 B 화면을 덮어쓸 수 있다는 지적이 맞습니다.

12) 테스트 patch 대상 불일치 (실패 재현)
테스트는 core.repair_until_valid를 patch합니다.

그런데 core는 repair_until_valid_many만 import해 사용합니다.

실제로 이 테스트 파일 실행 시 AttributeError로 3개 테스트 모두 실패했습니다 (아래 명령어 결과).
→ 지적 완전 유효.

13) tier 분류 경로 성능 회귀
_classify_tier_by_league_rank는 _get_league_rank_snapshot()를 호출하고,

snapshot 생성 시 active 팀 전체를 순회하며 성능/로스터 시그널을 계산합니다.

캐시는 evaluator 인스턴스 로컬입니다.

non-tick_ctx 경로는 호출마다 TeamSituationEvaluator(...)를 새로 만듭니다.

그래서 요청당 리그 전체 계산 반복 가능성이 있어 성능 이슈 지적이 타당합니다.

14) soft boundary exp overflow 위험
소프트 경계 계산에서 math.exp(d / max(tau, eps))를 그대로 사용합니다.

config 검증은 rank_boundary_tau > 0만 강제합니다(매우 작은 양수 허용).

따라서 tau=1e-6 같은 값이면 overflow 가능성이 있고, 런타임 예외 위험 지적이 맞습니다.
