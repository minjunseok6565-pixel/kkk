# 버그 모음

## A. 계약/캡/룰 엔진 핵심 이슈

### 1) MINIMUM 서명 타입 불일치
- MINIMUM 경로는 트랜잭션 `type/action_type`을 `SIGN_FA_MINIMUM`으로 기록함.
- 룰 메타 수집의 계약 타입 allowlist에는 `SIGN_FA_MINIMUM`이 없음 (`SIGN_FREE_AGENT`, `RE_SIGN`, `SIGN_TWO_WAY`, `RELEASE_TO_FA` 등만 포함).
- 최근 서명 밴 계산은 `last_contract_action_type in {SIGN_FREE_AGENT, RE_SIGN, SIGN_TWO_WAY}` 또는 `signed_via_free_agency=True`일 때만 적용됨.
- 결론: **MINIMUM 서명이 메타에 안 잡히면 밴이 약해질 수 있다는 지적이 타당함.**
- 해결 완

### 2) cap hold 삽입 시점 vs room flag 계산 시점
- 시즌 전환 중 선수 이동(`_move_player_team_in_cur`)에서 room-flag를 즉시 계산함.
- 계산 기준은 `_compute_team_payroll_for_season_in_cur`(순수 payroll).
- cap hold는 `expire_contracts_for_season_transition` 이후 `process_offseason`에서 나중에 upsert됨.
- `ROOM_MLE` 자격은 `team_room_mle_flags`를 직접 참조함.
- 결론: **hold 반영 전 상태로 ROOM_MLE가 잘못 열릴 수 있다는 우려가 타당함.**

### 3) cap hold 해제가 re_sign에만 존재
- `re_sign`에서는 `release_cap_hold(...)`를 호출함.
- 반면 `sign_free_agent` / `_sign_free_agent_minimum`에는 hold 해제 호출이 없음.
- cap 체크는 hold 합산 함수 `_compute_team_cap_salary_with_holds_in_cur`를 사용함.
- 결론: **기존 팀 hold 잔존으로 이후 cap 계산이 부풀 수 있다는 지적이 맞음.**
- 해결 : renounce api 호출시 캡 홀드를 해제하는 로직이 존재


### 4) `_channel_years_cap`의 snapshot 참조 위치
- `_channel_years_cap`는 `session["trade_rules_snapshot"]`(top-level)만 읽음.
- `start_contract_negotiation`은 snapshot을 `constraints`와 top-level 모두에 넣음.
- 세션 저장은 dict 전체 deepcopy라 top-level 필드가 버려지지 않음.
- 결론:
  - “top-level이 저장되지 않아 항상 무시됨”은 **현재 코드 기준 부정확**.
  - 다만 engine이 constraints fallback 없이 top-level만 보는 구조는 **취약점이 맞음**.

### 5) raise validator snapshot 참조 위치
- raise validator도 top-level `session["trade_rules_snapshot"]`만 읽음.
- 서비스 계층의 채널 정책 검증은 `constraints.trade_rules_snapshot`을 읽음.
- 결론:
  - 참조 일관성 깨짐으로 흐름별 override 가시성 차이 우려는 **타당**.
  - 다만 4번과 동일하게 “저장이 안 됨” 자체는 **현재 저장 코드와 불일치**.

### 6) 제안된 dead-cap unique index
- 현재 실제 스키마 unique: `(team_id, player_id, origin_contract_id, source_type, applied_season_year)`.
- 저장/upsert도 동일 키 사용.
- 질문의 새 인덱스 `(team_id, player_id, season_year, mode)`는 현재 코드에 **미적용 상태**.
- 결론: 제안 인덱스를 도입할 경우 재계약 후 재방출 같은 정상 시나리오에서 과도 충돌 우려는 **설계적으로 타당**.

### 7) release 타입 변경 시 downstream 누락
- move-event 수집은 `RELEASE_TO_FA/release_to_free_agency` 중심 필터.
- rule-player-meta 계약 타입도 `RELEASE_TO_FA/legacy lowercase` 위주.
- 결론: **`RELEASE_WAIVE/RELEASE_STRETCH`만 신규 사용 시 누락 가능 지적이 맞음.**
- 해결 완료

### 8) RE_SIGN_OR_EXTEND 누락
- 현재 `CONTRACT_TYPES`에 대문자 `RE_SIGN_OR_EXTEND`는 없고 lowercase `re_sign_or_extend`만 있음.
- `get_transactions_for_players`는 SQL `tx_type IN (...)` exact 매칭 사용.
- 결론: **과거 대문자 타입 저장 데이터가 있으면 누락 가능.**

### 9) raw action/type 저장 + 정규화 미흡
- 메타 빌더는 `action_type/type` 원문 그대로 저장(정규화 없음).
- `signed_via_free_agency`도 `last_action_type == "SIGN_FREE_AGENT"` exact match만 사용.
- 최근 서명 밴도 canonical uppercase 집합만 인정.
- 결론: **legacy lowercase가 safety net 역할을 못할 수 있다는 지적이 맞음.**

### 10) 선택 직후 save 버튼 상태
- 선택 변경 시 `playerTrainingSelectedPlayerId`만 바꾸고 로딩 문구만 표시.
- 즉시 `refreshPlayerTrainingSaveUi()` 호출 없음.
- save 버튼 활성 조건은 현재 선택 pid + dirty 상태 기반.
- 결론: **느린 응답 중 구 draft 상태가 남아 오저장 레이스 가능.**

### 11) 응답 순서 역전(race) 보호 없음
- `await fetchPlayerTrainingPlan(pid)` 후 공용 draft/UI 즉시 갱신.
- 중간에 선택 변경 검증(`selectedPlayerId === pid`) 없음.
- 결론: **A→B 빠른 클릭에서 A 늦은 응답이 B 화면을 덮어쓸 수 있음.**

### 12) 테스트 patch 대상 불일치 (실패 재현)
- 테스트는 `core.repair_until_valid`를 patch.
- 실제 `core`는 `repair_until_valid_many`만 import해 사용.
- 결과: 테스트 실행 시 `AttributeError`로 3개 테스트 실패(명령어 결과 기준).
- 결론: **지적 완전 유효.**

### 13) tier 분류 경로 성능 회귀
- `_classify_tier_by_league_rank` → `_get_league_rank_snapshot()` 호출.
- snapshot 생성 시 active 팀 전체 순회하며 성능/로스터 시그널 계산.
- 캐시는 evaluator 인스턴스 로컬.
- non-tick_ctx 경로는 호출마다 `TeamSituationEvaluator(...)`를 새로 생성.
- 결론: **요청당 리그 전체 계산 반복 가능성으로 성능 이슈 타당.**

### 14) soft boundary exp overflow 위험
- 소프트 경계 계산에서 `math.exp(d / max(tau, eps))`를 그대로 사용.
- config 검증은 `rank_boundary_tau > 0`만 강제(매우 작은 양수 허용).
- 결론: **`tau=1e-6` 같은 값에서 overflow 및 런타임 예외 위험 지적이 맞음.**

## B. 협상 / 트레이드 / 마켓

1. **협상 시작 시 기본 공개설정이 실제 커밋에 반영되지 않음**
   - `default_offer_privacy`를 시작 시 저장하지만, 커밋 시에는 항상 `req.offer_privacy`와 `PRIVATE` 기본값만 사용.
   - 결과적으로 세션 기본값이 사실상 무력화됨.

2. **직렬화된 딜의 `legs` 구조 불일치로 grievance 로직 비활성화**
   - helper는 `legs`가 list일 때만 처리.
   - 실제 `serialize_deal` 결과는 team ID keyed dict라 항상 빈 결과.

3. **공개화(publicized) idempotency 키가 너무 넓음**
   - session key 또는 deal key 중 하나만 publicized여도 이미 처리된 것으로 간주.
   - 같은 세션 이후 leaked deal 부작용(리스팅/이벤트/grievance) 스킵 가능.

4. **같은 상대 팀 쌍에서 낮은 점수 제안이 좋은 제안을 덮어쓸 수 있음**
   - duplicate-opponent guard 제거로 동일 tick 내 같은 pair 다중 proposal 처리.
   - 나중 proposal이 `set_last_offer`로 top offer를 overwrite.

5. **`ai_proactive_listing_player_cooldown_days=0`이 실질 반영되지 않음**
   - `or 7` 처리 때문에 0이 다시 7로 치환되어 튜닝 무효.

6. **listing-interest scoring 비활성 시 PUBLIC 리스팅도 발견되지 않음**
   - `select_targets_buy`에서 `is_public_listing`이 `listing_meta_by_player`에만 의존.
   - 해당 플래그를 끄면 PUBLIC 리스팅 예외도 동시 소실.

7. **음수 `min_surplus_required` 허용 시 confidence 과대평가 가능**
   - reject 경계는 `accept_threshold`인데 `_compute_confidence()`는 `overpay_floor` 거리 기준 사용.
   - reject confidence 왜곡 가능.

## K. 트레이드 / 협상 / 오퍼 프로모션

1. **ACTIVE 협상 세션 수 제한이 사실상 무력화**
- 현상: active session 집계/캐시는 하지만 생성 경로에서 hard cap 강제 없음.
- 영향: 특정 tick에서 ACTIVE 세션 과증가 → lock/state 및 commit 경로 부담 증가.
- 쉽게 설명하면: 제한 장치는 있는데 실제 제동이 걸리지 않아 동시 협상이 폭증할 수 있음.

2. **REJECT 오퍼가 과도하게 LOWBALL로 승격**
- 현상: `exceed_overpay > probe_max` 조건에서 REJECT가 광범위하게 LOWBALL로 이동.
- 영향: 원래 상한 억제/완충 경로 우회 가능.
- 쉽게 설명하면: 거절로 끝낼 제안이 공격적 재협상 대상으로 잘못 분류됨.

3. **`overpay_allowed` vs `probe_max` 스케일 불일치로 톤 오분류**
- 현상: `DecisionPolicy.decide`는 multi-axis 허용치, reject tone은 `max(outgoing_total, 6.0)` 기반 `probe_max` 사용.
- 영향: 픽/스왑 비중 딜에서 경계값 주변 오분류.
- 쉽게 설명하면: 같은 제안을 서로 다른 자로 재서 판정이 엇갈리는 문제.

4. **세션 재사용 시 `created_at` 미갱신으로 즉시 만료 가능**
- 현상: promotion에서 기존 세션 재사용 시 offer payload만 변경하고 `created_at` 유지.
- 영향: 다음 open 시 age hard-cap 즉시 도달 가능.
- 쉽게 설명하면: 새 제안인데 나이만 옛날로 남아 바로 만료되는 상황.

## L. 트레이드 생성 로직 / Skeleton / Template / Sweetener

1. **`PackageEffectsConfig`가 legacy constructor 필드를 받지 않아 런타임 호환성 실패**

- 쉽게 설명하면: 점수 차이 수준이 아니라 객체 생성부터 실패.

2. **아웃고잉 후보 빌더가 실제 불가 선수까지 후보 포함**
- 현상: 후보 수집이 `out_cat.players` 전체 순회에 치우쳐 lock/recent-signing 등 금지 조건 필터링 불충분.
- 영향: 실패가 예정된 skeleton 대량 생성.
- 쉽게 설명하면: 처음부터 거래 불가 선수를 넣어 후단에서 계속 반려되는 비효율.

3. **greedy fill이 고가치 자산을 먼저 소모해 저·중티어 딜 효율 저하**
- 현상: 고점수순 선집중 채움.
- 영향: 더 저렴한 조합 가능 케이스에서도 상위 자산이 먼저 소모됨.
- 쉽게 설명하면: 소액 결제에 고액권부터 깨서 전체 효율이 떨어짐.

4. **`skeleton_overhaul_enabled` 플래그가 실질 무시됨**
- 현상: 호출 경로가 v3 registry 기본 경로를 타며, False 설정으로 legacy 강제를 의도해도 반영 누락 구간 존재.
- 쉽게 설명하면: 스위치를 꺼도 내부 배선이 새 회로에 고정된 상태.

5. **template_only에서 빈 route가 ‘비활성’이 아니라 ‘전체 허용’으로 해석될 수 있음**
- 현상: `skeleton_route_template_*`가 빈 튜플일 때 downstream이 제약 없음으로 간주.
- 영향: 특정 tier 비활성 의도와 반대로 전체 template 오픈.
- 쉽게 설명하면: “실행 금지”가 “제한 없음”으로 뒤집히는 의미 역전.

6. **template wrapper의 `expected_tag` 하드코딩으로 유효 후보 전량 드롭 가능**
- 현상: `tpl_<tier>_placeholder_<n>` 강제.
- 영향: planner tag naming이 조금만 달라도 정상 후보가 필터에서 소실.
- 쉽게 설명하면: 내용이 맞아도 라벨 규칙 불일치로 전부 버려짐.

7. **pick protection 변형 생성이 sweetener ON/OFF에 종속됨**
- 현상: `maybe_apply_pick_protection_variants`가 sweetener 흐름 내부에 결합.
- 영향: sweetener 비활성/추가치 0이면 protection도 함께 비활성화.
- 쉽게 설명하면: 별개 스위치여야 할 기능이 함께 꺼지는 결합 문제.

8. **보호 변형 평가에서 `opponent_repeat_count=0` 하드코딩**
- 현상: 기본 proposal 평가는 repeat-partner 패널티 적용.
- 영향: protection 재평가에서는 0 고정으로 반복 거래 억제 신호 상실.
- 쉽게 설명하면: 본심사 감점 규칙이 재심사에서 빠져 점수 왜곡.

9. **`repair_until_valid_many` 이후 repaired variant 루프의 예산/하드캡 재확인 누락**
- 현상: 한 base에서 복수 repaired 후보가 생겨도 loop 내부 즉시 `_budget_or_hard_cap_reached` 재체크 누락 가능.
- 영향: 제한 도달 후에도 불필요한 평가/검증 추가 실행.
- 쉽게 설명하면: 예산 종료 경고등이 켜져도 다음 공정으로 진행하는 구조.

## N. 티어링 / 평가 / 성능

1. **bubble bonus가 경쟁권 밖 팀에도 적용될 수 있음**
- 현상: `gb_to_6th`, `gb_to_10th`를 거리값으로 쓰며 상한 조건만 검사.
- 영향: 안전권 상위팀에도 보너스 적용 케이스 발생.
- 쉽게 설명하면: 막차 경쟁팀 전용 가산점이 여유 팀에도 잘못 붙는 문제.

2. **tier 분류가 league-rank 전체 스냅샷 경로를 자주 타서 hot path 성능 저하**
- 현상: `_classify_tier_by_league_rank` 호출 단위로 리그 전체 신호 계산 동반.
- 영향: 단일 팀 평가 요청도 리그 전체 계산으로 확장되어 지연 증가.
- 쉽게 설명하면: 한 팀만 물어봐도 매번 30팀 재채점.

3. **soft-boundary 확률 계산이 작은 tau에서 overflow 위험**
- 현상: `math.exp(d / max(tau, eps))` 형태에 clamp 부족.
- 영향: tau가 매우 작을 때 `OverflowError`로 tier 계산 크래시 가능.
- 쉽게 설명하면: 경계 보정식이 과도하게 가팔라지면 수치 폭발로 계산 중단.
